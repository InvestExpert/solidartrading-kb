## 3. Главная польза для базы

Сохранить стоит не обзор интерфейса как таковой, а рабочую методику:

- Codex работает не как чат, а как агент с доступом к папке проекта.
- Для проекта нужно заводить правила в `agents.md`, чтобы Codex понимал стек, команды, ограничения и самопроверку.
- Для сложных изменений полезен `plans.md` — шаблон качественного плана перед реализацией.
- Для безопасности нужно ограничивать доступ к файлам через sandbox/workspace write и осторожно использовать полный доступ.
- Для технической базы полезны skills, plugins, MCP, subagents, worktrees, hooks и automations.
- Для TiTan Bots особенно полезны: код-ревью, поиск багов, проверка UI, обновление `agents.md`, автоматический аудит проекта, отдельные агенты под безопасность/качество/адаптивность.

## 4. Краткое резюме

Видео показывает Codex как агентский инструмент для разработки: он читает и меняет файлы проекта, запускает команды, проверяет результат, работает в терминале, десктопе, VS Code и Cursor. Автор проходит путь от установки и базового теста до более сложных инструментов: reasoning, sandbox, режим плана, slash-команды, файл `agents.md`, шаблон `plans.md`, MCP-серверы Context7 и Playwright, skills, plugins, subagents, worktrees, Git/GitHub, Vercel, automations и hooks. В конце даётся субъективное сравнение Codex и Claude Code: Codex проще купить и, по мнению автора, выгоднее по лимитам, но уступает Claude Code по зрелости хуков и взаимодействию субагентов.

## 11. Ошибки и решения

### Ошибка 1 — Codex меняет не ту область/папку
- Симптом: агент может работать вне нужного проекта при полном доступе или неверной папке.
- Причина: широкие разрешения, выбран не тот workspace.
- Решение: использовать sandbox/workspace write, отдельные проектные папки, проверять путь перед задачей.
- Где проявляется: 20:29–23:03.
- Применимость: критично для репозиториев торговых ботов.

### Ошибка 2 — Неправильная настройка websearch/config
- Симптом: Codex записал настройку не так, как ожидалось; в транскрипте видны искажения параметра.
- Причина: неточный промпт, OCR-искажение, возможная ошибка Codex.
- Решение: проверять `config.toml` вручную, не доверять автоматической правке без просмотра diff.
- Где проявляется: 23:56–24:59.
- Применимость: правила конфигурации Codex нужно хранить отдельно и проверять.

### Ошибка 3 — UI вышел узким/кривым
- Симптом: лендинг/магазин выглядит странно, элементы расположены неудачно.
- Причина: короткий и неконкретный промпт без референса.
- Решение: дать скриншот, конкретный layout, целевую ширину, адаптивные требования.
- Где проявляется: 35:18–36:16.
- Применимость: для лендингов SolidarTrading и кабинетов ботов.

### Ошибка 4 — Слишком большой `agents.md`
- Симптом: агент забывает стек, правила или архитектуру.
- Причина: файл загружается в начале сессии и перегружает контекст.
- Решение: держать `agents.md` коротким; длинные инструкции вынести в skills/docs.
- Где проявляется: 36:35–36:58.

## 12. Правила

1. Перед работой с Codex всегда проверять папку проекта.
2. Не использовать полный доступ без чёткой задачи и понимания последствий.
3. Для крупных правок сначала включать режим плана.
4. После крупной задачи запускать code review.
5. Для конкретного файла указывать файл через `@` или IDE context.
6. Для UI-задач прикладывать скриншоты и текстовое описание, что именно повторить.
7. `agents.md` держать коротким: стек, команды, правила, самопроверка, критичные запреты.
8. Ошибки, которые Codex повторяет, фиксировать в `agents.md` или отдельном skill.
9. Для сложных планов использовать `plans.md` и проверять путь к нему.
10. MCP подключать только при понятной пользе; после подключения проверять `/mcp`.
11. Skills использовать для повторяемых специализированных задач.
12. Subagents использовать только для серьёзных задач из-за расхода токенов.
13. Эксперименты делать в Git-ветках/worktrees, а не в основной рабочей версии.
14. Перед push на GitHub проверять `.gitignore`.
15. Не хранить ключи, токены, state-файлы и приватные данные в Git.
16. Automations должны вносить минимальные изменения и требовать ревью.
17. Hooks использовать для защиты и проверки, но держать команды простыми и безопасными.
18. Все утверждения о тарифах, моделях, лимитах и датах проверять перед публикацией.

## 16. Метод и объяснение продукта простым языком

Полезные объяснения для обычного человека:

- Codex — это не просто чат, а «разработчик внутри компьютера», который работает с файлами проекта.
- Папка проекта — это рабочее место агента. Что дали в доступ, с тем он и работает.
- Sandbox — это ограничение: агент не должен трогать лишние папки.
- Режим плана — сначала «чертёж», потом работа. Это снижает риск сломать проект.
- `agents.md` — инструкция для помощника: что за проект, какие правила, что нельзя ломать.
- `plans.md` — шаблон хорошего плана: шаги, проверки, результат.
- MCP — способ дать агенту внешний инструмент, например браузер или актуальную документацию.
- Skill — отдельная инструкция для конкретной роли: ревьюер кода, тестировщик, обработчик видео.
- Subagents — несколько специалистов вместо одного: один смотрит безопасность, другой качество, третий вёрстку.
- Worktree — безопасная копия проекта для экспериментов.
- Hook — автоматическая проверка или сигнал в нужный момент.

Применение к TiTan Bots:

- «Мы не просто просим ИИ что-то написать. Мы даём ему правила проекта, ограничиваем доступ, заставляем планировать, проверять изменения и фиксировать результат».
- «Для торговых ботов нельзя менять код вслепую. Сначала план, потом минимальная правка, потом тесты и сравнение логов».

## 20. Вопросы и ответы для Telegram-бота

### Вопрос 1
Вопрос: Что такое Codex простыми словами?  
Ответ: Codex — это ИИ-агент для работы с проектом. Он может читать файлы, предлагать правки, запускать команды и помогать с проверкой кода. Результат всё равно нужно проверять.

### Вопрос 2
Вопрос: Можно ли использовать Codex для торговых ботов?  
Ответ: Да, но осторожно: через план, отдельную ветку, минимальные правки, code review и тесты. Нельзя давать агенту менять торговую логику без проверки.

### Вопрос 3
Вопрос: Что такое `agents.md`?  
Ответ: Это короткая инструкция проекта для Codex: стек, команды, правила кода, запреты и самопроверка. Она помогает агенту не забывать базовые правила.

### Вопрос 4
Вопрос: Почему `agents.md` нельзя делать огромным?  
Ответ: Большой файл перегружает контекст. Агент может хуже помнить важные правила. Лучше держать базу короткой, а длинные инструкции вынести в отдельные skills.

### Вопрос 5
Вопрос: Что такое режим плана?  
Ответ: Это режим, где Codex сначала предлагает план работы и не меняет файлы, пока план не подтверждён.

### Вопрос 6
Вопрос: Что такое MCP?  
Ответ: MCP — способ подключить к Codex внешние инструменты: документацию, браузер, базу данных, деплой и другие сервисы.

### Вопрос 7
Вопрос: Что такое skills?  
Ответ: Skills — отдельные инструкции для повторяемых задач. Например, ревью кода, обработка видео, проверка логики бота или подготовка SEO-описания.

### Вопрос 8
Вопрос: Что такое subagents?  
Ответ: Это несколько отдельных агентов под разные роли. Например, один проверяет безопасность, второй качество кода, третий тесты. Это полезно, но расходует больше токенов.

### Вопрос 9
Вопрос: Что такое worktree?  
Ответ: Worktree — отдельная копия проекта или ветка для экспериментов, чтобы не ломать основную версию.

### Вопрос 10
Вопрос: Можно ли полностью доверять Codex?  
Ответ: Нет. Codex ускоряет работу, но не заменяет ревью, тесты и контроль результата.

- Риск Git: можно случайно закоммитить `node_modules`, ключи, локальные state-файлы.
- Риск автоматизаций: scheduled-задачи могут менять файлы без достаточного контроля.
- Риск hooks: hook — это исполняемая команда; ошибочная команда может навредить.
- Риск токенов: high/extra high reasoning, fast и subagents могут быстро расходовать лимиты.
- Риск актуальности: тарифы, модели, лимиты и функции Codex быстро меняются.
- Юридический риск: сторонняя покупка подписок и обход доступа не должны попадать в рекомендации проекта.
- Маркетинговый риск: нельзя обещать, что ИИ-разработка гарантирует отсутствие багов.

Безопасная формулировка:

> Codex можно использовать как ускоритель разработки и аудита, но все изменения в торговых ботах должны проходить ручную проверку, тесты и контроль логов.

Не переносить:

- рекламный блок про VPN/ускоритель, кроме риска серой инфраструктуры доступа;
- эмоциональные реплики автора;
- бытовые вставки;
- личные действия автора с оплатой/аккаунтом;

- команды из транскрипта без проверки, так как часть распознана с ошибками;
- субъективные утверждения «лучше/хуже» без пометки как мнение.

## 31. Краткий итог

1. Codex полезен для проекта как агент разработки, но требует ограничений и контроля.
2. Основной безопасный процесс: план → минимальная правка → review → тесты → commit.
3. `agents.md` нужен как короткая инструкция проекта; длинные правила лучше выносить в skills.
4. `plans.md` полезен для стандартизации сложных задач.
5. MCP расширяет Codex внешними инструментами, но повышает риски доступа.
6. Skills и plugins можно использовать для повторяемых задач: ревью, обработка видео, RAG, маркетинг.
7. Subagents подходят для комплексного аудита, но расходуют больше токенов и не общаются напрямую.
8. Worktrees подходят для экспериментов с версиями ботов без порчи основной ветки.
9. Automations и hooks полезны, но должны быть безопасными и проверяемыми.
10. Все утверждения о тарифах, моделях, лимитах и датах нужно проверять перед публикацией.
