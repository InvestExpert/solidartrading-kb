## Краткое резюме

В чате разбиралась ошибка логики классификации событий после срабатывания защитного режима. После защитного завершения сессии система продолжала учитывать последующий тейк как V6, хотя фактически цикл уже был завершён защитой.

Проблема приводила к искажению статистики:
- завышению доли V6;
- ложной оценке эффективности reserveOrders;
- ложной оценке тейков;
- неправильной оптимизации параметров стратегии.

Главный вывод:
terminal event должен быть единственным и неизменяемым после завершения сессии.

## Идеи

- Ввести atomic final event.
- После guard/stop запрещать любые дальнейшие event transitions.
- Ввести отдельный session state lock.
- Разделить:
  - технический TP;
  - terminal TP;
  - TP после защиты.

## Гипотезы

- Ошибка могла искусственно улучшать результаты reserveOrders.
- Ложные V6 могли создавать иллюзию устойчивого recovery.
- Часть прибыльных сценариев являлась статистическим артефактом.
- После исправления возможна деградация некоторых “лучших” конфигураций.

## Выводы

- TP после защитного режима не должен считаться V6.
- После terminal event система должна становиться immutable.
- Защитное завершение должно иметь приоритет над любыми поздними тейками.
- Система статистики должна фиксировать только первое terminal событие.
- Поздние торговые события допустимы только как технические, но не статистические.

## Ошибки и решения

### Ошибка

После:
- STOP;
- GUARD;
- FORCE EXIT;
- PROTECT;

система могла позже записать TAKE/V6.

Это нарушало целостность статистики сессии.

### Причина

Не было:
- terminal lock;
- atomic session finalization;
- блокировки повторной записи final event.

### Решение

Рекомендовано:
- ввести session_locked=true;
- фиксировать final_event только один раз;
- игнорировать все поздние TP после guard;
- отменять pending take logic после защитного завершения.

### Дополнительная рекомендация

Добавить отдельный guard_context:
- guard_reason;
- guard_time;
- guard_level;
- terminal_source.

## Правила

- Terminal event записывается один раз.
- После STOP/GUARD дальнейшие TP не влияют на статистику.
- Статистика не должна зависеть от поздних технических событий.
- Event priority обязателен:
  - V5/VSTOP > V6.
- Все terminal события должны быть атомарными.

## Архитектура

### Рекомендуемая схема

Session lifecycle:

1. ACTIVE
2. GUARD/TRAIL/FORCE
3. TERMINAL EVENT
4. LOCK SESSION
5. IGNORE LATE EVENTS
