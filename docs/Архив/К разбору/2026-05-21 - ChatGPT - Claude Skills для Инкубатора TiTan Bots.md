## Краткое резюме

Видео посвящено набору Claude Skills / agent skills, которые автор считает наиболее полезными после тестирования более 200 вариантов.

Главная идея для TiTan Bots: использовать не отдельные промты, а постоянные skills/rules, которые задают агенту дисциплину работы:
- сначала думать и уточнять;
- потом строить план;
- потом писать код;
- потом тестировать;
- потом проверять результат;
- использовать проектные правила, UI-компоненты, базу данных и архитектурные ограничения.

Для TiTan Bots это относится не к основному техническому слою, а к **Инкубатору**, потому что идея требует проверки и адаптации под проект.

## Идеи

### Идея 1. Создать собственные skills для TiTan Bots

Суть: не копировать все skills из видео, а сделать собственный набор правил под проект.

Возможные skills:
- `TiTan Bots Code Auditor`
- `TiTan Bots Strategy Logic Reviewer`
- `TiTan Bots Wiki Extractor`
- `TiTan Bots Marketing Content Skill`
- `TiTan Bots Telegram/RAG Assistant Skill`
- `TiTan Bots UI/Report Designer`

Статус: сырая идея.

### Идея 2. Skill для строгого code review торговых ботов

Суть: отдельный skill должен заставлять агента сначала проверять код, а не сразу писать патч.

Правила skill:
- сначала аудит;
- потом список рисков;
- потом минимальная правка;
- не удалять существующие комментарии;
- проверять входы, выходы, вызовы, глобальные флаги, recovery;
- проверять совместимость с соседними функциями;
- не менять архитектуру без необходимости;
- не предлагать костыли до проверки существующих механизмов.

Статус: приоритетная идея.

### Идея 3. Skill для извлечения материалов в Obsidian wiki

Суть: использовать отдельный skill, который превращает чаты, видео, заметки и тесты в структурированные Markdown-файлы для Obsidian.

Задачи:
- отделять Технику, Метод и Инкубатор;
- сохранять точные параметры, выводы, гипотезы и ошибки;
- не переносить личное и нерелевантное;
- готовить материал для последующего импорта Codex-агентом.

Статус: уже частично используется через правила текущего чата.

### Идея 4. Skill для маркетинговой упаковки TiTan Bots

Суть: отдельный skill для подготовки постов, обложек, сценариев, презентаций и отчетов.

Что должен учитывать:
- нельзя обещать гарантированную прибыль;
- сначала боль/желание/интрига;
- потом выгода;
- потом простое объяснение;
- потом доказательства;
- потом ограничения и риски.

Статус: тема для Инкубатора.

### Идея 5. Skill для RAG / Telegram-бота по базе TiTan Bots

Суть: агент должен отвечать по базе проекта без ручного меню.

Применение:
- база знаний по роботам;
- инструкции по использованию;
- партнерские материалы;
- ответы на возражения;
- ссылки на презентации и видео;
- поддержка простых пользовательских вопросов.

Статус: тема на разбор.

## Гипотезы

### Гипотеза 1. Skills могут снизить количество ошибок агента при правках кода

Суть: если агент работает по постоянному skill с жестким порядком “аудит → план → минимальный diff → проверка”, он будет меньше ломать соседнюю логику.

К чему относится: TiTan Bots / разработка ботов / Codex / Cursor / Claude.  
Как проверить:
- взять один и тот же файл;
- дать задачу без skill;
- дать задачу со skill;
- сравнить количество лишних изменений, ошибок и пропусков.  
Статус: не проверена.

### Гипотеза 2. Собственный Wiki Extractor skill ускорит перенос материалов в Obsidian

Суть: если закрепить формат извлечения, можно быстрее превращать чаты и транскрипты в пригодные wiki-заметки.

К чему относится: TiTan Bots wiki / Obsidian / Codex.  
Как проверить:
- обработать 5 старых чатов вручную;
- обработать 5 чатов через skill;
- сравнить полноту, точность и время.  
Статус: частично проверяется текущим процессом.

### Гипотеза 3. UI/design skills помогут быстро делать интерфейсы и отчеты без отдельного дизайнера

Суть: готовые frontend/design skills могут ускорить создание кабинетов, отчетов, лендингов и презентационных страниц.

К чему относится: Метод и продвижение / веб-интерфейсы / отчеты.  
Как проверить:
- сделать одну страницу отчета без skill;
- сделать такую же страницу со skill;
- сравнить качество, скорость и количество правок.  
Статус: не проверена.

### Гипотеза 4. MCP Builder будет полезен позже, но сейчас преждевременен

Суть: MCP может связать AI-агента с внешними системами, но без стабильной архитектуры это добавит сложность.

К чему относится: AI-инфраструктура TiTan Bots.  
Как проверить:
- сначала стабилизировать RAG/Telegram-бота;
- потом выбрать один простой API-сценарий;
- сделать прототип MCP;
- проверить, дает ли он реальную пользу.  
Статус: не проверена.

## Ошибки и решения

### Ошибка 1. Слепо внедрять все skills из видео

Симптом: появляется много правил, но они не связаны с реальными задачами TiTan Bots.  
Причина: видео ориентировано на общий вайб-кодинг и веб-разработку, а TiTan Bots имеет специфическую торговую логику.  
Решение: брать только идеи, а не весь набор. Сделать 3–4 собственных skills под проект.  
Где проявляется: Codex / Claude / Cursor / разработка ботов.

### Ошибка 2. Использовать MCP слишком рано

Симптом: усложнение архитектуры до появления стабильного RAG/Telegram ядра.  
Причина: MCP требует понятных API, сценариев и прав доступа.  
Решение: отложить MCP до момента, когда будет понятно, что именно нужно связать.  
Где проявляется: AI-инфраструктура TiTan Bots.

### Ошибка 3. Доверять транскрипту как точному источнику названий

Симптом: названия skills могут быть искажены.  
Причина: автоматическая транскрибация.  
Решение: проверить ссылки и названия в описании видео вручную.  
Где проявляется: импорт в wiki.

## Правила

### Правило 1. Для TiTan Bots готовый skill нельзя внедрять без адаптации

Любой внешний skill должен быть проверен на совместимость с:
- торговой логикой;
- правилами минимальных правок;
- требованиями к тестам;
- правилами wiki;
- ограничениями по приватности;
- маркетинговыми дисклеймерами.

### Правило 2. Агент по коду не должен начинать с патча

Порядок:
1. аудит;
2. риски;
3. проверка цепочки вызовов;
4. минимальный план;
5. минимальная правка;
6. проверка совместимости;
7. список того, что проверить тестом.

### Правило 3. Wiki Extractor должен сохранять точные данные

Нельзя заменять точные параметры, версии, формулы, названия и выводы общими фразами.

Если точных данных нет, нужно писать:
“Недостаточно данных для надежного вывода”.

### Правило 4. Marketing Skill не должен обещать гарантированную прибыль

Разрешено:
- интрига;
- сильная подача;
- демонстрация результатов;
- объяснение метода;
- сравнение подходов.

Запрещено:
- гарантировать доход;
- скрывать риски;
- выдавать тесты за будущий результат.

## Формулы и параметры

Формул и торговых параметров в видео нет.

Есть процессная схема:

`Brainstorm → Plan → TDD → Refactor`

Расшифровка:
- Brainstorm: агент задает уточняющие вопросы и вытаскивает требования.
- Plan: агент дробит работу на короткие шаги с файлами и тестами.
- TDD: сначала красный тест, потом зеленый тест.
- Refactor: улучшение после прохождения теста.

Ограничение: схема полезна не всегда. Если задача простая и точная, избыточный TDD может замедлить работу.

## Возражения и ответы

### Возражение 1. “ИИ будет сам торговать?”

Короткий ответ: нет.  
Подробный ответ: ИИ используется для базы знаний, поддержки, документации, анализа и контроля логики. Торговая логика задается ботами и стратегиями, а не свободными ответами ИИ.  
Что нельзя обещать: нельзя обещать, что ИИ увеличит прибыль.

### Возражение 2. “Зачем skills, если можно просто написать промт?”

Короткий ответ: промт разовый, skill — постоянное правило.  
Подробный ответ: для сложного проекта важно, чтобы агент каждый раз работал по одной дисциплине: проверял логику, не ломал соседние функции, сохранял формат wiki и не терял параметры.  
Что нельзя обещать: нельзя обещать, что skill полностью уберет ошибки.

### Возражение 3. “MCP нужно внедрять сразу?”

Короткий ответ: нет.  
Подробный ответ: MCP имеет смысл, когда уже есть стабильная база, API и понятный сценарий. До этого он может только усложнить систему.  
Что нельзя обещать: нельзя обещать быструю интеграцию без разработки.
