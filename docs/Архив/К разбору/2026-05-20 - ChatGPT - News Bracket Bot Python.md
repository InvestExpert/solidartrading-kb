## Краткое резюме

В чате проектировался простой Python-бот для новостной стратегии "скобка". Бот не анализирует новости, а запускается вручную пользователем. После запуска он берет текущую цену, ставит несколько BUY Stop-ордеров выше цены и несколько SELL Stop-ордеров ниже цены. У каждой стороны есть стоп и несколько тейков. Когда срабатывает одна сторона, бот должен отменить противоположную сторону. В ходе тестов были найдены ошибки с использованием Limit-ордеров, округлением цен, хранением ордеров и отменой противоположной ветки.

## Справочники и соответствия

### Параметр -> смысл

- `step_pct` -> размер шага в процентах от текущей цены.
- `lot_size` -> фиксированный объем одной заявки.
- `order_count` -> количество ордеров в каждой стороне, используется для масштабирования позиции и нескольких тейков.
- `weekend_pause` -> блокировка торговли в субботу и воскресенье.
- `buy_entries` -> список только BUY ENTRY-ордеров.
- `sell_entries` -> список только SELL ENTRY-ордеров.
- `initialized` / `grid_done` -> флаг, что сетка уже поставлена.

### Версия / итерация -> что изменилось

- Начальная идея:
  - использовать скобку: BUY выше цены и SELL ниже цены.
- Первая реализация:
  - использовались `Limit`-входы.
  - проблема: Limit внутри OHLC-бара исполнялся сразу, что искажало бэктест.
- Следующая реализация:
  - ENTRY переведены на `bt.Order.Stop`.
  - проблема немедленного исполнения Limit-entry устранена.
- Упрощение хранения:
  - сначала предлагались плоские массивы всех ордеров (`buy_all`, `sell_all`);
  - затем принято решение хранить только ENTRY-ордера, потому что bracket должен сам отменять дочерние TP/SL.
- Исправление флага:
  - возникла ошибка `object has no attribute 'initialized'`;
  - решение: использовать один согласованный флаг или добавить `self.initialized = False` в `__init__`.

### Ошибка -> причина -> решение

- Warning: `Order price ... is not a multiple of the tick size 0.01`
  - Причина: цена после вычислений имела хвосты float или не была округлена до tick size.
  - Решение: округлять цены до двух знаков: `round(..., 2)`.
- Limit-entry исполняется сразу внутри 1-минутного бара.
  - Причина: Backtrader на OHLC-баре исполняет лимит, если цена уровня попала в диапазон бара.
  - Решение: использовать `bt.Order.Stop` для входа по пробою.
- После срабатывания BUY остались активные SELL-ордера.
  - Причина: отменялась не вся противоположная сторона или неверно хранились ссылки.
  - Решение: хранить все противоположные ENTRY в списке и отменять их при срабатывании одной стороны.
- Ошибка `object has no attribute 'initialized'`.
  - Причина: в `nextstart()` проверялся атрибут `self.initialized`, но в `__init__` он не был создан.
  - Решение: добавить `self.initialized = False` или заменить все обращения на единый флаг `grid_done`.

### Режим -> условие включения -> условие выключения

- Торговая сетка:
  - включение: первый бар / ручной запуск стратегии;
  - выключение повторной постановки: флаг `initialized` или `grid_done`.
- BUY-ветка:
  - включение: цена дошла до `buy_entry`;
  - после включения: отменяются все SELL-entry.
- SELL-ветка:
  - включение: цена дошла до `sell_entry`;
  - после включения: отменяются все BUY-entry.
- Ночная пауза:
  - включение: `hhmm >= 22*60 + 50 or hhmm <= 10`;
  - выключение: вне диапазона `22:50–00:10` Berlin.
- Weekend pause:
  - включение: `weekend_pause and weekday >= 5`;
  - выключение: будние дни.

## Идеи

- Создать новостного бота, который не анализирует новости, а только быстро выставляет заранее заданную скобку по ручной команде.
- Использовать несколько одинаковых ENTRY-ордеров на одной цене, но с разными take-profit уровнями.
- Масштабировать общий размер позиции через `order_count`, а не через один большой ордер.
- После срабатывания одной стороны снимать противоположную сторону, чтобы не получить реверс при возврате цены.
- Для простоты хранить только ENTRY-ордера, а TP/SL доверить bracket-механизму Backtrader.
- Оставить фильтр выходных и ночной паузы из исходного примера.

## Гипотезы

### Гипотеза 1: множественные ENTRY на одном уровне с разными TP удобнее одного ENTRY с частичными тейками

- Суть: можно поставить 3 BUY Stop на одной цене и каждому назначить свой TP: +1, +2, +3 шага.
- Бот/стратегия: News Bracket Bot.
- Чем проверить:
  - прогнать бэктест на нескольких новостных периодах;
  - сравнить с вариантом одного ENTRY объемом `order_count * lot_size` и тремя частичными TP.
- Статус: частично проверена на одном тесте XAUUSD.PRO 2025-07-15; требует дальнейшей проверки.

### Гипотеза 2: для новостного пробоя ENTRY должен быть Stop, а не Limit

- Суть: если вход выше текущей цены для BUY и ниже текущей цены для SELL, это логика пробоя, значит входной ордер должен быть Stop.
- Бот/стратегия: News Bracket Bot.
- Чем проверить:
  - сравнить логи Limit-entry и Stop-entry на 1m данных;
  - проверить отсутствие мгновенных ложных исполнений.
- Статус: подтверждена на текущих логах: Limit-entry давал искажение, Stop-entry стал исполняться только при достижении уровня.

### Гипотеза 3: достаточно отменять только противоположные ENTRY, а не TP/SL

- Суть: если используется `buy_bracket()` / `sell_bracket()`, отмена parent/entry должна отменить связанные дочерние ордера.
- Бот/стратегия: News Bracket Bot / Backtrader bracket.
- Чем проверить:
  - по логам убедиться, что после отмены entry дочерние TP/SL получают статус canceled и не исполняются позже.
- Статус: частично проверена; требует отдельного теста, потому что в логах были асинхронные эффекты и возможные гонки внутри одного бара.

## Выводы

- Для стратегии "новостная скобка" входы должны быть `bt.Order.Stop`, а не `bt.Order.Limit`.
- `Limit`-вход выше/ниже рынка в Backtrader на 1m OHLC может исполниться сразу, если уровень попал внутрь диапазона бара. Это дает нереалистичные сделки.
- Несколько BUY Stop на одной цене исполняются все, если цена достигает уровня. Это не ошибка, а ожидаемое поведение при `order_count = 3`.
- Не нужно отменять оставшиеся BUY Stop после первого BUY, если они специально поставлены ради разных тейков.
- При первом BUY fill нужно отменять SELL-ветку, но не BUY-ветку.
- При первом SELL fill нужно отменять BUY-ветку, но не SELL-ветку.
- Хранить все TP/SL в отдельных массивах избыточно, если используется нормальная bracket-связка.
- Флаг состояния должен быть единым и созданным в `__init__`; иначе Backtrader падает до запуска теста.
- Логи должны явно писать:
  - какой ENTRY исполнился;
  - по какой цене;
  - какой ref;
  - какие противоположные ENTRY отменяются.

## Ошибки и решения

### Ошибка 1: `Limit` вместо `Stop` для входа

- Симптом:
  - в логах сразу исполнялись ордера внутри первого 1-минутного бара;
  - BUY Limit и SELL Limit могли исполняться по одной цене в одной минуте.
- Причина:
  - для пробоя был выбран `bt.Order.Limit`, хотя логика требует stop-entry.
- Решение:
  - заменить `exectype=bt.Order.Limit` на `exectype=bt.Order.Stop` для entry-ордеров.
- Где проявляется:
  - Backtrader, 1m OHLC-тесты, News Bracket Bot.

### Ошибка 2: лишнее предположение, что после первого BUY нужно отменять остальные BUY

- Симптом:
  - было предложено отменять "дубликаты" BUY Stop после первого исполнения.
- Причина:
  - неверно понята цель `order_count`; это не дубликаты, а несколько частей позиции с разными тейками.
- Решение:
  - не отменять свои однонаправленные entry;
  - отменять только противоположную сторону.
- Где проявляется:
  - архитектура News Bracket Bot.

### Ошибка 3: противоположные SELL продолжали срабатывать после BUY

- Симптом:
  - после BUY fill позже исполнялись SELL Stop или их дочерние ордера.
- Причина:
  - не все противоположные entry корректно отменялись или были сохранены не те ссылки.
- Решение:
  - хранить список всех противоположных ENTRY:
    - `self.buy_entries`
    - `self.sell_entries`
  - при исполнении BUY cancel всех `sell_entries`;
  - при исполнении SELL cancel всех `buy_entries`.
- Где проявляется:
  - Backtrader bracket-цепочки.

### Ошибка 4: избыточные массивы всех ордеров

- Симптом:
  - появились предложения хранить `buy_all` и `sell_all`, включая entry, tp, sl.
- Причина:
  - попытка вручную контролировать всю цепочку.
- Решение:
  - упростить: хранить только ENTRY, так как TP/SL связаны bracket-механизмом.
- Где проявляется:
  - архитектура хранения ордеров.

### Ошибка 5: `AttributeError: ... object has no attribute 'initialized'`

- Симптом:
  - бэктест упал:
    - `Process failed: ... object has no attribute 'initialized'`
- Причина:
  - в `nextstart()` проверялся `self.initialized`, но в `__init__` он отсутствовал.
- Решение:
  - добавить `self.initialized = False` в `__init__`;
  - или заменить все проверки на один флаг, например `self.grid_done`.
- Где проявляется:
  - Backtrader strategy lifecycle.

### Ошибка 6: предупреждения tick size

- Симптом:
  - `Order price ... is not a multiple of the tick size 0.01`
- Причина:
  - цена не была округлена до `0.01`.
- Решение:
  - простое округление всех уровней:
    - `round(..., 2)`
- Где проявляется:
  - XAUUSD.PRO, tick size 0.01.

## Правила

- Для входа по пробою в Backtrader использовать `bt.Order.Stop`.
- Для входа откатом или покупки ниже текущей цены можно использовать Limit, но это не текущая логика новостной скобки.
- Все цены ордеров должны быть округлены до tick size инструмента.
- Для XAUUSD.PRO в данном тесте использован tick size `0.01`, значит округление до двух знаков.
- Если `order_count > 1`, однонаправленные ENTRY с разными TP не считаются ошибочными дубликатами.
- При исполнении первой стороны:
  - cancel противоположных ENTRY;
  - свои ENTRY не cancel.
- Не плодить массивы всех ордеров без необходимости.
- Если используется `buy_bracket()` / `sell_bracket()`, минимально достаточно хранить parent/entry-ордера.
- В `__init__` обязательно создавать все поля, которые будут использоваться в `nextstart()`, `next()` или `notify_order()`.
- Логирование должно показывать не только факт исполнения, но и управленческое действие: какие ордера отменяются.
- Временной фильтр должен работать в торговой таймзоне, а не слепо по UTC.

## Архитектура

### Компоненты стратегии

- `params`
  - хранит параметры шага, лота, количества ордеров и выходных.
- `__init__`
  - создает состояние стратегии:
    - `self.step`
    - `self.buy_entries`
    - `self.sell_entries`
    - `self.initialized` или `self.grid_done`
- `nextstart`
  - один раз рассчитывает уровни;
  - выставляет BUY bracket и SELL bracket;
  - сохраняет только ENTRY-ордера.
- `notify_order`
  - реагирует на `bt.Order.Completed`;
  - если исполнен BUY-entry, отменяет все SELL-entry;
  - если исполнен SELL-entry, отменяет все BUY-entry.
- `_cancel_entries`
  - отменяет список ENTRY-ордеров и пишет лог.
- `next`
  - применяет фильтр времени: выходные и ночная пауза.
- `bar_to_berlin`
  - переводит UTC-время бара в Berlin с учетом DST.
- `_last_sunday`
  - вспомогательная функция для расчета DST-переходов.

### Выбранный вариант хранения ордеров

Выбранный вариант:

- хранить только:
  - `self.buy_entries`
  - `self.sell_entries`

Отвергнутый вариант:

- хранить:
  - `self.buy_all`
  - `self.sell_all`
  - все entry/tp/sl в одном плоском списке.

Причина отказа:

- избыточно;
- bracket-механизм Backtrader должен сам связывать parent entry с дочерними TP/SL;
- стратегия должна отменять противоположные входы, а не вручную управлять всеми дочерними ордерами.

### Финальная логика cancel

```python
if order in self.buy_entries:
    self.log(f"BUY EXECUTED @{order.executed.price:.2f} (ref={order.ref})")
    self._cancel_entries(self.sell_entries, label="SELL")
    self.sell_entries.clear()

elif order in self.sell_entries:
    self.log(f"SELL EXECUTED @{order.executed.price:.2f} (ref={order.ref})")
    self._cancel_entries(self.buy_entries, label="BUY")
    self.buy_entries.clear()
```

### Формула шага

```python
step = round(price * step_pct / 100.0, 2)
```

Переменные:

- `price` — текущая цена.
- `step_pct` — шаг в процентах от цены.
- `step` — абсолютный шаг цены.

Пример из теста:

- `step = 6.69`

### BUY-уровни

```python
buy_entry = round(price + 2 * step, 2)
buy_tp_i = round(buy_entry + (i + 1) * step, 2)
buy_sl = round(buy_entry - step, 2)
```

Смысл:

- BUY entry ставится на 2 шага выше текущей цены.
- TP для каждого ордера:
  - первый: +1 шаг от entry;
  - второй: +2 шага от entry;
  - третий: +3 шага от entry.
- SL ставится на 1 шаг ниже entry.

Пример из теста:

- `buy_entry = 3358.46`
- `buy_sl = 3351.77`
- `buy_tp_1 = 3365.15`
- `buy_tp_2 = 3371.84`
- `buy_tp_3 = 3378.53`

### SELL-уровни

```python
sell_entry = round(price - 2 * step, 2)
sell_tp_i = round(sell_entry - (i + 1) * step, 2)
sell_sl = round(sell_entry + step, 2)
```

Смысл:

- SELL entry ставится на 2 шага ниже текущей цены.
- TP для каждого ордера:
  - первый: -1 шаг от entry;
  - второй: -2 шага от entry;
  - третий: -3 шага от entry.
- SL ставится на 1 шаг выше entry.

Пример из теста:

- `sell_entry = 3331.70`
- `sell_sl = 3338.39`
- `sell_tp_1 = 3325.01`
- `sell_tp_2 = 3318.32`
- `sell_tp_3 = 3311.63`

def notify_order(self, order):
    if order.status != bt.Order.Completed:
        return

    if order in self.buy_entries:
        self.log(f"BUY EXECUTED @{order.executed.price:.2f} (ref={order.ref})")
        self._cancel_entries(self.sell_entries, label="SELL")
        self.sell_entries.clear()
