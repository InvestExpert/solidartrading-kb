## Краткое резюме

В чате разбиралась задача поиска формулы для адаптации шага сетки `gridStepPerc_smart` по абсолютной волатильности `AbsVol`.

Цель: найти такую формулу, которая одновременно:

- сдвигает рабочее плато шага между месяцами;
- сохраняет или расширяет прибыльное плато внутри месяца;
- не требует новой логики, новых измерений и отдельной адаптации буфера/трейлинга;
- использует существующий `AbsVol`.

Ключевой конфликт:

- простая степенная формула хорошо двигает шаг между месяцами;
- pivot-формула лучше стабилизирует плато внутри апреля;
- но одна и та же форма не дала одновременно сильный сдвиг и широкую стабильную зону.

## Идеи

### Использовать AbsVol как внешний регулятор шага

`AbsVol` рассматривается как месячный / сессионный индекс абсолютной волатильности.

В коде он считается как:

```mql
AbsVol = ATR(short_period, D1) / ATR(long_period, D1)
```

В текущей реализации:

```mql
short_period = min(30, available_bars / 2)
long_period  = min(365, available_bars - 10)
```

Идея: при росте абсолютной волатильности увеличивать рабочий шаг сетки, при снижении уменьшать.

### Не трогать буфер и трейлинг в этом эксперименте

Для чистоты теста буфер и трейлинг временно не адаптируются:

```mql
g_snap.forcedBufferPerc_smart = forcedBufferPerc;
g_snap.trailingBufferPerc_smart = trailingBufferPerc;
```

Это важно: иначе результат формулы шага смешивается с влиянием защитной логики.

### Квантование шага

В коде использовалось округление итогового шага до `0.01`:

```mql
double stepQuant = 0.01;
g_snap.gridStepPerc_smart = MathFloor(g_snap.gridStepPerc_smart / stepQuant + 0.5) * stepQuant;
```

Идея: привести расчётный шаг к оптимизируемой сетке значений, чтобы тесты были сопоставимы.

## Гипотезы

### Гипотеза 1. Простая степень двигает плато, но может рвать прибыльный ковёр

Формула:

```mql
g_snap.gridStepPerc_smart = gridStepPerc * MathPow(g_snap.absVol, g_snap.absVolCoef);
```

Ожидание:

- при высокой волатильности шаг растёт;
- при низкой волатильности шаг уменьшается;
- можно привести разные месяцы к более близкой рабочей зоне.

Наблюдение:

- формула действительно сдвигает шаг;
- но может ломать широкое плато прибыльности в сильных месяцах;
- реакция слишком грубая, особенно при большом коэффициенте.

Статус: полезная базовая идея, но не финальное решение.

### Гипотеза 2. Pivot-степень стабилизирует апрель, но слабо двигает шаг

Формула:

```mql
g_snap.gridStepPerc_smart = gridStepPerc * MathPow(g_snap.absVol / SmartPivotValue, g_snap.absVolCoef);
```

Пример pivot:

```mql
SmartPivotValue = 1.50;
```

Наблюдение:

- при `pivot=1.50` формула дала интересное расширение прибыльного плато в апреле;
- при коэффициенте около `0.27–0.30` формула мягко меняет шаг;
- но смещение между месяцами оказалось недостаточным;
- в январе/феврале не удалось получить то же качество, что в апреле.

Статус: хорошая стабилизирующая формула, но слабая как универсальный перенос плато.

### Гипотеза 3. Лог-демпфер слишком сильно гасит сдвиг

Формула:

```mql
g_snap.gridStepPerc_smart =
    gridStepPerc * MathExp(
        (g_snap.absVolCoef / (1.0 + MathAbs(MathLog(g_snap.absVol / SmartPivotValue))))
        * MathLog(g_snap.absVol / SmartPivotValue)
    );
```

Идея:

- сохранить поведение степенной формулы;
- ослабить крайние отклонения через делитель `1 + |ln(r)|`;
- не рвать плато.

Где:

```mql
r = g_snap.absVol / SmartPivotValue;
```

Наблюдение:

- формула стала мягче;
- плато меньше рвётся;
- но сдвиг рабочего шага слишком слабый;
- при сильном коэффициенте плато снова начинает ломаться.

Статус: кандидат отклонён как недостаточно эффективный для главной задачи.

### Гипотеза 4. tanh-формула может дать лучший компромисс

Формула-кандидат:

```mql
g_snap.gridStepPerc_smart =
    gridStepPerc * MathExp(
        g_snap.absVolCoef * MathTanh(MathLog(g_snap.absVol / SmartPivotValue))
    );
```

Идея:

- рядом с pivot поведение похоже на обычную степень;
- далеко от pivot реакция насыщается;
- это может сохранить плато и при этом дать больше управляемого смещения.

Рабочие ориентиры из обсуждения:

```mql
SmartPivotValue ≈ 1.50
gridStepPerc ≈ 0.33
SmartAbsVolCoef ≈ 0.30–0.40
```

Статус: основной кандидат для дальнейших тестов.

### Гипотеза 5. Magnet-формула как альтернатива

Формула:

```mql
g_snap.gridStepPerc_smart =
    gridStepPerc
    + g_snap.absVolCoef
      * (MathLog(g_snap.absVol / SmartPivotValue)
      / (1.0 + MathAbs(MathLog(g_snap.absVol / SmartPivotValue))));
```

Идея:

- не умножать базовый шаг, а добавлять ограниченную дельту к `gridStepPerc`;
- `gridStepPerc` становится центром / магнитом;
- коэффициент задаёт максимальную добавку или вычитание;
- насыщение `x / (1 + |x|)` гасит крайние отклонения.

Важное отличие:

- в степенных формулах коэффициент — сила изменения;
- в Magnet-формуле коэффициент фактически становится максимальной дельтой шага.

Статус: экспериментальный кандидат. Требует отдельной проверки, потому что смысл коэффициента меняется.

## Выводы

### Универсального решения на одной оси пока нет

Одна только абсолютная волатильность `AbsVol` не объясняет все месяцы.

Причина: месяцы отличаются не только средним уровнем волатильности, но и структурой движения:

- трендовость;
- частота разворотов;
- длительность направленных фаз;
- распределение движений внутри месяца;
- качество флета.

Особенно подозрительные месяцы: март и июль. Для них может понадобиться второй фактор, например [ADX](ADX.md) или другой индекс трендовости.

### Разные формулы решают разные половины задачи

Степенная формула:

- лучше двигает шаг;
- хуже сохраняет плато.

Pivot-формула:

- лучше стабилизирует плато;
- хуже переносит рабочий шаг между месяцами.

Лог-демпфер:

- даёт мягкость;
- но слишком слаб по смещению.

`tanh`:

- потенциально лучший компромисс;
- требует теста на всех месяцах.

Magnet:

- интересен как формула притяжения к базовому шагу;
- может быть полезен, если нужно контролировать максимальную дельту.

### Коэффициенты нельзя сравнивать напрямую между формулами

Например, `0.27` в pivot-степенной формуле и `0.35` в `tanh` или Magnet-формуле — не одно и то же.

У каждой формулы разный смысл коэффициента:

- степень: чувствительность множителя;
- лог-демпфер: чувствительность после ослабления;
- tanh: чувствительность насыщенной лог-реакции;
- Magnet: максимальная дельта к базовому шагу.

Сравнивать надо не коэффициенты, а итоговую карту:

```text
AbsVol → gridStepPerc_smart → результат теста
```

## Ошибки и решения

### Ошибка: пытаться исправить формулу только подбором коэффициента

Если лучшая пара `gridStepPerc + coef` для января не совпадает с лучшей парой для апреля, это сигнал, что форма зависимости может быть недостаточной.

Решение:

- не только подбирать коэффициент;
- сравнивать форму кривой;
- смотреть, двигается ли плато и не рвётся ли оно.

### Ошибка: смешивать адаптацию шага с буфером и трейлингом

В одном из вариантов обсуждалась адаптация буфера и трейлинга обратно шагу, но это было отвергнуто как лишняя логика для текущего этапа.

Решение:

- в этом эксперименте менять только `gridStepPerc_smart`;
- буфер и трейлинг фиксировать;
- иначе невозможно понять вклад формулы шага.

### Ошибка: ожидать, что мягкий коэффициент даст сильный сдвиг

При `absVolCoef ≈ 0.30` и `pivot=1.50` мягкие формулы расширяют плато, но не обязаны сильно смещать центр.

Решение:

- считать фактический множитель `q`;
- строить график `gridStepPerc → gridStepPerc_smart`;
- проверять, достаточно ли отношения между месяцами.

### Ошибка: сохранять сырые логи целиком

Сырые `STAT_ROW` полезны для анализа, но в wiki их лучше не хранить полностью.

Решение:

- сохранить формат и назначение колонок;
- крупные логи держать отдельно как источник;
- в wiki переносить выводы и короткие фрагменты.

## Правила

### Правило 1. Сначала оценивать форму кривой, потом прибыль

Перед массовым тестом нужно понять, что формула делает с шагом:

```text
AbsVol → множитель/дельта → итоговый шаг
```

### Правило 2. Для сравнения формул строить карту шагов

Минимальная таблица для каждой формулы:

```text
месяц
AbsVol
base gridStepPerc
gridStepPerc_smart
PnL
eventV breakdown
```

### Правило 3. Апрель/май использовать как проверку плато

Апрель и май важны как месяцы, где уже найдено прибыльное широкое плато.

Формула не должна ломать этот участок ради улучшения слабых месяцев.

### Правило 4. Январь/февраль использовать как проверку переноса шага

Если формула не может увести шаг вниз для более спокойных месяцев, она не решает задачу универсализации.

### Правило 5. Март/июль считать подозрением на вторую ось

Если после подбора `AbsVol`-формулы март и июль остаются плохими, вероятная причина — не уровень волатильности, а трендовость или структура движения.

Кандидат на вторую ось:

- [ADX](ADX.md)
- [VI](VI.md)
- отдельный тренд-фильтр

## Архитектура

### Точка внедрения

Формула применяется в функции:

```mql
bool Regime_BeginSession()
```

Ключевые поля:

```mql
g_snap.absVol
g_snap.absVolCoef
g_snap.gridStepPerc_smart

```

### Расчёт AbsVol

Функция:

```mql
double CalculateAbsVol()
```

Логика:

```mql
AbsVol = ATR(short_period, PERIOD_D1) / ATR(long_period, PERIOD_D1)
```

### Логирование

Формат строк:

```text
STAT_HEADER | ts_start,VI,TI,AbsVol,AbsVolCoef,step,step_coef,forcedBufferPerc_smart,trailingBufferPerc_smart,forcedBufferPerc_coef,trailingBufferPerc_coef,eventV,pnl,success
STAT_ROW | ...
```

Смысл важных полей:

- `ts_start` — старт сессии;
- `VI` — локальный индекс волатильности;
- `TI` / `ADX` — индекс трендовости;
- `AbsVol` — абсолютная волатильность;
- `AbsVolCoef` — коэффициент формулы;
- `step` — итоговый `gridStepPerc_smart`;
- `eventV` — тип завершения;
- `pnl` — результат сессии;
- `success` — классификация результата.
