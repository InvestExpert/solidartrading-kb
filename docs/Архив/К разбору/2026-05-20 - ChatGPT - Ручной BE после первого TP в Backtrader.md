## Краткое резюме

В чате разобрана доработка Python/Backtrader-стратегии `MeanReversionStrategy`: после достижения первым ордером первого тейк-профита нужно вручную переносить стопы оставшихся ордеров в безубыток с процентным буфером.

Платформенный trailing stop оказался недоступен или неподдержан, поэтому выбрана ручная схема: сохранить ссылки на stop-ордера, в `next()` проверить касание первого TP через `high/low`, отменить старые стопы и выставить новые обычные Stop-ордера на уровне entry ± buffer.

Финальная рабочая логика подтверждена логом бэктеста на `XAUUSD.PRO`: после TP одного BUY-ордера два оставшихся стопа были перенесены на `3359.13`, затем закрылись по этому уровню, итоговый `PROFIT: 8.03000`.

## Справочники и соответствия

### Параметр -> смысл

- `step_pct` -> процентный шаг сетки.
- `step` -> абсолютный шаг цены, рассчитанный из `step_pct`.
- `order_count` -> число bracket-ордеров на сторону.
- `be_buffer` -> процентный буфер к цене входа для переноса стопа в реальный безубыток.
- `start_price` -> цена центра сетки, от которой вычисляются entry, TP и BE.
- `buy_entries` -> список BUY-entry ордеров.
- `sell_entries` -> список SELL-entry ордеров.
- `buy_stops` -> список stop-ордеров, относящихся к BUY-позициям.
- `sell_stops` -> список stop-ордеров, относящихся к SELL-позициям.
- `stops_shifted` -> защёлка, чтобы перенос стопов в BE выполнился один раз.

### Уровень -> формула

- BUY entry первого ордера -> `start_price + 2 * step`
- SELL entry первого ордера -> `start_price - 2 * step`
- BUY первый TP -> `start_price + 3 * step`
- SELL первый TP -> `start_price - 3 * step`
- BUY BE с буфером -> `(start_price + 2 * step) * (1 + be_buffer)`
- SELL BE с буфером -> `(start_price - 2 * step) * (1 - be_buffer)`

### Значение `be_buffer` -> процент

- `0` -> 0%, строго по цене входа.
- `0.01` -> 1%.
- `0.001` -> 0.1%.
- `0.0002` -> 0.02%.

### Ошибка -> причина -> решение

- `buy_bracket(trailpercent=...)` не дал ожидаемый trailing stop  
  Причина: `buy_bracket()` создал обычные Stop/Limit pending-ордера без нужных цен, а не полноценный поддержанный платформой trailing stop.  
  Решение: отказаться от платформенного trailing stop и реализовать ручной перенос стопов через отмену старого Stop и постановку нового Stop.

- Stop/Limit order rejected: `'Pending order types Limit,Stop require a price to be specified.'`  
  Причина: pending Stop/Limit ордера были созданы без обязательной цены.  
  Решение: не использовать этот вариант `buy_bracket()` для trailing; ручная постановка обычных Stop-ордеров с явным `price`.

- BE сначала считался от `start_price`  
  Причина: неверная логика, BE-уровень был взят из центра сетки, а не из точки открытия позиции.  
  Решение: считать BE от entry: BUY `start_price + 2 * step`, SELL `start_price - 2 * step`.

- При буфере 1% BUY stop был выставлен на `3392.04` и закрылся сразу  
  Причина: уровень stop оказался выше текущей рыночной цены после TP, поэтому Sell Stop стал фактически уже сработавшим/рыночным.  
  Решение: использовать небольшой буфер, например `0.0002` для 0.02%, чтобы stop был выше entry, но ниже текущей цены после TP.

- Warning: `Order price ... is not a multiple of the tick size 0.01`  
  Причина: формула дала дробную цену, не кратную минимальному шагу цены.  
  Решение: брокер/обвязка округлила цену до tick size `0.01`; можно дополнительно округлять цену в коде, если нужно убрать предупреждения.

## Идеи

- Реализовать не полноценный trailing stop, а ручной перенос стопов в BE после первого TP.
- Использовать факт касания первого TP через `data.high[0]` для BUY и `data.low[0]` для SELL.
- Хранить ссылки на stop-ордера, потому что их нужно отменять и заменять.
- Не хранить лишние уровни, если они детерминированы через `start_price` и `step`.
- Использовать `stops_shifted` как одноразовую защёлку.
- Для покрытия комиссии и спреда ставить BE не ровно в entry, а entry ± процентный буфер.

## Выводы

- Платформенный trailing stop в текущем окружении использовать нельзя: платформа/API не поддерживает нужный тип ордера.
- Backtrader `buy_bracket(trailpercent=...)` не является безопасной заменой ручной логики trailing/BE в этом окружении.
- Для задачи "первый TP сработал -> оставшиеся стопы в BE" лучше использовать ручной перенос stop-ордеров.
- Хранить ссылки на стопы нужно, потому что Backtrader не редактирует stop-ордера напрямую: старый stop надо отменить и поставить новый.
- Хранить `entry_price` отдельно не нужно, если entry всегда вычисляется как `start_price ± 2 * step`.
- Хранить `first_tp_price` отдельно не нужно, если первый TP всегда вычисляется как `start_price ± 3 * step`.
- Для BUY реальный BE с буфером: `(start_price + 2 * step) * (1 + be_buffer)`.
- Для SELL реальный BE с буфером: `(start_price - 2 * step) * (1 - be_buffer)`.
- Буфер должен быть достаточно большим, чтобы покрыть спред/комиссию, но достаточно маленьким, чтобы новый stop не оказался уже пересечённым рынком.
- В тесте `be_buffer = 0.0002` дал корректный результат: stop оказался выше entry и не сработал сразу.
- В тесте с чрезмерным буфером stop `3392.04` был выше текущей цены и исполнился немедленно. Это не рабочий BE, а аварийное закрытие из-за неправильного уровня.
- Предупреждения по tick size не ломают логику, но могут засорять лог; цена округляется к `0.01`.

## Правила

- Перед изменением ордерной логики сначала определить, какие уровни уже выводятся из существующих переменных, чтобы не добавлять лишнее состояние.
- Не переименовывать существующие переменные без необходимости.
- Если entry всегда детерминирован через `start_price` и `step`, не хранить отдельный `entry_price`.
- Если первый TP всегда детерминирован через `start_price` и `step`, не хранить отдельный `first_tp_price`.
- Для ручного BE обязательно хранить ссылки на stop-ордера, потому что их нужно отменять.
- Для переноса стопов использовать одноразовый флаг `stops_shifted`.
- Переносить стопы только после факта касания первого TP:
  - BUY: `high >= start_price + 3 * step`
  - SELL: `low <= start_price - 3 * step`
- Для BUY стоп в BE с покрытием комиссии должен быть выше цены входа.
- Для SELL стоп в BE с покрытием комиссии должен быть ниже цены входа.
- Буфер `be_buffer` хранить как долю, а не как "проценты текстом":
  - 1% -> `0.01`
  - 0.1% -> `0.001`
  - 0.02% -> `0.0002`
- При выставлении нового stop проверять здравый смысл уровня:
  - BUY/Sell Stop после TP должен быть выше entry, но ниже текущей цены;
  - SELL/Buy Stop после TP должен быть ниже entry, но выше текущей цены.
- Если брокер округляет цену до tick size, это допустимо, но лучше учитывать tick size в коде, если нужны чистые логи.

## Архитектура

### Выбранный поток ручного BE

1. При создании сетки:
   - вычислить и сохранить `start_price`;
   - вычислить `step`;
   - выставить BUY и SELL bracket-ордера;
   - сохранить ссылки на stop-ордера в `buy_stops` и `sell_stops`.

2. После исполнения первой стороны:
   - противоположные entry-ордера отменяются;
   - связанные противоположные stop/take также отменяются брокером/OCO;
   - списки противоположной стороны нужно чистить, чтобы не трогать мёртвые ссылки.

3. В `next()`:
   - если `stops_shifted == True`, ничего не делать;
   - для BUY проверить: `data.high[0] >= start_price + 3 * step`;
   - для SELL проверить: `data.low[0] <= start_price - 3 * step`;
   - если условие выполнено, вызвать перенос стопов.

4. Перенос стопов:
   - пройти по живым stop-ордерам активной стороны;
   - отменить старый stop;
   - выставить новый обычный Stop по BE-цене;
   - заменить ссылку в списке;
   - выставить `stops_shifted = True`.

### Отвергнутые альтернативы

- Полностью заменить списки на `buy_brackets` / `sell_brackets`.
  - Причина отказа: слишком большая перестройка ради простой задачи.

- Хранить `entry_price` для каждого ордера.
  - Причина отказа: entry вычисляется из `start_price ± 2 * step`.

- Хранить `first_tp_price`.
  - Причина отказа: первый TP вычисляется из `start_price ± 3 * step`.

- Использовать `active_side`.
  - Причина отказа: можно ориентироваться по спискам stop-ордеров и цене.

- Использовать платформенный trailing stop.
  - Причина отказа: текущая платформа/API его не поддерживает.

## Формулы и параметры

### Расчёт шага

Точная формула расчёта `step` в доступном тексте чата не зафиксирована полностью. По логам:
- `step = 6.69`
- `start_price ≈ 3345.08`
- BUY entry `3358.46 = start_price + 2 * step`
- BUY первый TP `3365.15 = start_price + 3 * step`

### Entry

Переменные:
- `start_price` -> центр сетки;
- `step` -> абсолютный шаг сетки.

Формулы:
- BUY entry: `start_price + 2 * step`
- SELL entry: `start_price - 2 * step`

Смысл:
- цена открытия первого ордера активной стороны.

Пример:
- `start_price ≈ 3345.08`
- `step = 6.69`
- BUY entry: `3345.08 + 2 * 6.69 = 3358.46`

### Первый Take Profit

Переменные:
- `start_price`
- `step`

Формулы:
- BUY first TP: `start_price + 3 * step`
- SELL first TP: `start_price - 3 * step`

Смысл:
- уровень, после касания которого нужно переносить стопы оставшихся ордеров в BE.

Пример:
- `start_price ≈ 3345.08`
- `step = 6.69`
- BUY TP: `3345.08 + 3 * 6.69 = 3365.15`

### BE с процентным буфером

Переменные:
- `entry_price`
- `be_buffer`
- `start_price`
- `step`

Формулы:
- BUY BE: `(start_price + 2 * step) * (1 + be_buffer)`
- SELL BE: `(start_price - 2 * step) * (1 - be_buffer)`

Смысл:
- уровень нового stop-ордера, который закрывает оставшиеся позиции около безубытка с запасом на комиссии/спред.

Ограничения:
- `be_buffer` задаётся долей, а не числом процентов.
- Для BUY новый Sell Stop не должен оказаться выше текущей рыночной цены при выставлении, иначе может исполниться сразу.
- Для SELL новый Buy Stop не должен оказаться ниже текущей рыночной цены при выставлении, иначе может исполниться сразу.
- Цена должна быть кратна tick size или будет округлена брокером.

Пример рабочего теста:
- BUY entry: `3358.46`
- `be_buffer = 0.0002`
- BUY BE: `3358.46 * 1.0002 = 3359.131692`
- округлено до tick size `0.01`: `3359.13`
- новые stop refs: `19`, `20`
- закрытие: `3359.13`

### Подбор `be_buffer`

Формула оценки:
- `buffer ≈ (spread + commission + запас) / entry_price`

Примерная логика:
- если цена XAUUSD около `3350`;
- если издержки около `0.60`;
- то `0.60 / 3350 ≈ 0.000179`;
- близкое значение: `be_buffer = 0.0002`, то есть `0.02%`.
