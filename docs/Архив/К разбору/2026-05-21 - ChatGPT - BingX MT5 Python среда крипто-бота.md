## Краткое резюме

В чате выбран практический путь разработки крипто-версии сеточного BUY-бота TiTan Atlas под BingX Spot.

Главная цель: быстро портировать существующую MT5-логику в Python, не писать собственный тестер с нуля и не уходить в Linux/Docker/сложные среды.

Финальная рабочая схема:

```text
MT5 BingX → оптимизация параметров
Python → порт логики бота
backtesting.py → подтверждающий тест работоспособности
CCXT → универсальный слой подключения к биржам
BingX Spot → будущая реальная торговля
Windows VPS → среда исполнения
```

- Docker -> отклонено -> пользователь хочет Windows-only без WSL/Linux.
- WSL -> отклонено -> лишний слой, мешал запуску Cursor.
- 3Commas -> отклонено как среда разработки -> это конструктор/инфраструктура, нельзя портировать кастомную MT5-логику ATLAS.
- TradeLocker -> частично применим как полигон -> Python-среда есть, но не финальный путь под BingX Spot.
- Прямой BingX API -> отложено -> больше кода; использовать только если CCXT не справится.
- CCXT -> принято -> быстрее старт, мультибиржевость, меньше кода.
- backtesting.py -> принято -> простой подтверждающий тест без написания движка.

- Глобальный Python не содержит нужных библиотек -> проект работает в `.venv` -> ставить библиотеки внутри `.venv`.

## Идеи

### Идея 1. Использовать BingX как основную биржу и бизнес-платформу

Смысл: не просто запускать бота на бирже, а строить бренд внутри BingX через Master Trader / копитрейдинг / партнерскую структуру.

Польза:
- не нужно строить всю инфраструктуру привлечения с нуля;
- биржа уже имеет механизмы маркетинга и доверия;
- можно использовать копитрейдинг как витрину результата.

### Идея 2. Использовать MT5 BingX как быстрый оптимизатор

Смысл: не строить оптимизатор в Python на первом этапе, а использовать знакомый MT5-оптимизатор.

Польза:
- быстрее старт;
- привычный GUI;
- не нужны Linux, Docker, Freqtrade;
- можно переиспользовать существующий MT5-бот.

### Идея 3. Писать live-бота на Python под Spot

Смысл: MT5 используется как лаборатория, но реальный бот работает на Python через CCXT.

Польза:
- максимальная совместимость с BingX Spot;
- возможность позже добавить другие биржи;
- контроль над исполнением;
- нет зависимости от MT5 в live.

### Идея 4. Использовать `backtesting.py` как подтверждающий тест

Смысл: не заменять MT5-оптимизатор, а проверить, что портированная Python-логика работает и дает понятное поведение.

Польза:
- не писать свой движок;
- не возиться с CSV;
- можно тянуть данные через CCXT;
- есть график и сделки.

### Идея 5. Развивать стратегию через диверсификацию по топовым монетам

Смысл: переносить форекс-логику диверсификации на крипту, но через топовые монеты.

Польза:
- несколько активов могут сгладить equity;
- Master Trader получает более стабильную витрину;
- частые сделки могут создавать регулярное обновление equity.

## Гипотезы

### Гипотеза 1. MT5-фьючерсы BingX достаточно близки к споту для оптимизации BUY-only market-бота

Суть:
Если бот работает только в покупку, без плеча, без стопов, без лимитных ордеров и принимает решения только по цене, то параметры, найденные на фьючерсном графике BingX MT5, могут быть применимы к BingX Spot.

К чему относится:
- TiTan Atlas (BA);
- MT5;
- BingX;
- Python Spot-port.

Как проверить:
- взять один и тот же период;
- протестировать MT5 BingX;
- портировать логику в Python;
- прогнать на спотовых свечах;
- сравнить паттерн сделок, equity, DD, forced events.

Статус:
частично обоснована, не подтверждена финальными крипто-тестами.

### Гипотеза 2. Funding можно игнорировать при оптимизации, если live будет на споте

Суть:
Funding влияет на PnL фьючерсной позиции, но не на саму структуру цены и не на точки входа/выхода. Если реальная торговля будет на споте, funding исчезает.

К чему относится:
- BingX MT5;
- BingX Spot;
- TiTan Atlas (BA).

Как проверить:
- сравнить тесты на фьючерсных данных с расчетом без funding;
- затем прогнать Python-тест на спотовых данных;
- проверить отличие по структуре сделок.

Статус:
логически принято для стартовой оптимизации.

### Гипотеза 3. Profit-share модель подходит ATLAS лучше классических сеток

Суть:
Классическая сетка может зависать и долго не обновлять High Water Mark. ATLAS отличается частыми сделками и защитой через продажу верхних ордеров в убыток, поэтому может чаще обновлять equity и быть лучше для Master Trader.

К чему относится:
- TiTan Atlas (BA);
- BingX Master Trader;
- копитрейдинг.

Как проверить:
- построить крипто-эквити;
- измерить длительность периодов без нового equity high;
- проверить частоту обновления High Water Mark;
- оценить регулярность выплат profit share.

Статус:
не проверена на крипте; на форекс-версии есть сильные предпосылки.

   - Причина: нужен BingX Spot и дальнейшая мультибиржевость.

## Ошибки и решения

### Ошибка 1. Неправильный ранний выбор Docker/Freqtrade

Симптом:
Путь через Docker/Freqtrade оказался некомфортным: WSL, команды, Linux-слой, Docker Desktop.

Причина:
Первоначально акцент был на “готовом крипто-фреймворке”, но пользовательская задача была другой: Windows GUI, MT5-оптимизатор, минимум инфраструктуры.

Решение:
Отказаться от Docker/Freqtrade как основного пути. Оставить MT5 для оптимизации, Python + backtesting.py + CCXT для порта и проверки.

Где проявляется:
среда разработки.

### Ошибка 2. Cursor запускал WSL

Симптом:
При старте Cursor появлялось окно `wsl.exe` / установка WSL / Linux-значок.

Причина:
В системе были включены или частично установлены WSL-компоненты, также ранее ставились Docker/Python. Cursor пытался дергать WSL.

Решение:
Отключить WSL для проекта, убрать WSL-компоненты, работать в Windows PowerShell.

Где проявляется:
Windows / Cursor / терминал.

### Ошибка 3. PowerShell блокировал активацию `.venv`

Симптом:
Команда активации виртуального окружения не запускалась.

Причина:
PowerShell Execution Policy запрещала запуск локального `.ps1`.

Решение:

```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

Где проявляется:
Windows PowerShell / Cursor terminal.

### Ошибка 4. BingX API не отдавал широкий диапазон M1-свечей

Симптом:

```text
bingx {"code":100204, "msg":"date of query is too wide."}
```

Причина:
BingX ограничивает ширину запроса исторических свечей.

Решение:
- уменьшать период;
- либо для первичной проверки использовать Binance/Bybit через CCXT;
- BingX оставить для live и финальной проверки.

Где проявляется:
CCXT / BingX OHLCV / backtesting.py pipeline.

## Правила

1. Не писать собственный тестер, пока достаточно `backtesting.py`.
2. Не использовать ручные CSV и склейку файлов на первом этапе.
3. Данные для проверки загружать через CCXT прямо в коде.
4. MT5 использовать как основной оптимизатор параметров.
5. Python использовать как будущую live-среду.
6. `backtesting.py` использовать только как подтверждение работоспособности порта и возможный простой оптимизатор.
7. CCXT использовать как стартовый API-слой.
8. Прямой API BingX подключать только если CCXT не покрывает нужную функцию.
9. Docker/WSL не использовать в этом проекте без прямой необходимости.
10. Все библиотеки ставить в `.venv`, не в глобальный Python.
11. После валидации зафиксировать версии:
    ```powershell
    pip freeze > requirements.txt
    ```
12. Новые версии библиотек не считать автоматически лучшими.
13. Для трейдинга важнее стабильность и повторяемость, чем самые свежие версии.
14. При переносе MT5 → Python сначала переносить минимальный рабочий цикл:
    - старт сетки;
    - добавление BUY;
    - выход/SELL;
    - затем защиту и усложнения.
15. Если всё работает на демо-логике `backtesting.py`, следующий шаг — перенос реальной логики MT5 в класс `Strategy`.

## Формулы и параметры

### Funding

Формула в чате не задавалась, но смысл зафиксирован:

funding = периодический платеж между лонгами и шортами на фьючерсах

Для проекта:

### High Water Mark

Смысл:

```text
Master Trader получает profit share только от прибыли выше предыдущего максимума equity.
```

Для ATLAS важно:

```text
чем чаще equity обновляет максимум, тем регулярнее потенциальный profit share
```

### Целевая проверка Master Trader модели

Показатель:

```text
максимальная длительность периода без нового equity high
```

Для ATLAS:

- если equity обновляется часто, модель Master Trader подходит лучше;
- если крипто-версия даст долгие периоды без нового максимума, profit share будет рваным.

## Метод и объяснение продукта

### Объяснение для обычного человека

Идея не в том, чтобы пользователь сам торговал криптой. Идея в том, чтобы дать ему доступ к работающей торговой системе через готовую инфраструктуру биржи.

BingX может быть не только биржей, но и площадкой продвижения:
- трейдер показывает результат;
- подписчики подключаются через копитрейдинг;
- система сама копирует сделки;
- бренд строится внутри биржи.

ATLAS отличается от обычной сетки тем, что он не должен зависать в просадке. Его защита продает часть верхних покупок в убыток, чтобы разгружать сетку и продолжать работу.

### Простое объяснение Master Trader логики

Обычная реферальная модель сложна, потому что у многих уже есть аккаунты на крупных биржах. Перенести старый аккаунт под новую реферальную ссылку часто нельзя или неудобно.

Master Trader / копитрейдинг решает это иначе:
- человек может уже быть внутри биржи;
- он подписывается на трейдера;
- трейдер получает долю от результата или выгоду через партнерскую структуру;
- не нужно убеждать всех заново регистрироваться на бирже.

## Офферы и продажные формулировки

### Формулировка 1

> Бот не зависает в просадке, а продолжает работать.

Где использовать:
- пост;
- презентация;
- видео;
- лендинг.

Смысл:
отличие от классических сеточников, которые могут замораживаться надолго.

Риск:
нельзя превращать в обещание отсутствия убытков.

### Формулировка 2

> Защита депозита продает верхние ордера в убыток, чтобы бот мог пережить просадку и продолжить работу.

Где использовать:
- объясняющее видео;
- презентация;
- раздел “как работает защита”.

Смысл:
простое объяснение forced sell / защитного механизма.

Риск:
нужно объяснять, что продажа в убыток — часть риск-менеджмента, а не гарантия прибыли.

### Формулировка 3

> Бренд внутри биржи: не просто бот, а готовая площадка для доверия, копирования и партнерской модели.

Где использовать:
- стратегия продвижения;
- презентация партнерам;
- упаковка проекта.

Смысл:
BingX рассматривается не только как биржа, но как маркетинговая инфраструктура.

Риск:
условия биржи могут меняться.

## Контент, видео и презентации

Идеи для будущего контента:

## Возражения и ответы

### Возражение 1. “У людей уже есть аккаунты на топовых биржах, на реферальной комиссии заработать сложно”

Короткий ответ:
Да, если ставка только на реферальную ссылку, это слабое место.

Подробный ответ:
Лучше строить модель не только на регистрации новых людей, а на Master Trader / копитрейдинге / бренде внутри биржи. Тогда человек может не создавать новый аккаунт, а подключиться к трейдеру внутри платформы.

Что нельзя обещать:
нельзя обещать, что всех существующих пользователей можно перевести под свою партнерку.

### Возражение 2. “Если цикл сетки висит 3–6 месяцев, Master Trader ничего не зарабатывает”

Короткий ответ:
Для классической сетки это риск. Для ATLAS риск ниже, если equity обновляется часто.

Подробный ответ:
Profit share зависит от обновления High Water Mark. Если бот часто фиксирует прибыль и equity обновляет максимум, выплаты будут регулярнее. Если защита часто режет equity и восстановление долгое — выплаты будут редкими.

Что нельзя обещать:
нельзя обещать регулярный profit share без крипто-тестов.

### Возражение 3. “Фьючерсы в MT5 не равны споту”

Короткий ответ:
Да, это не одно и то же, но для BUY-only market-бота без плеча параметры могут быть близки.

Подробный ответ:
Фьючерсная цена обычно близка к спотовой. Funding влияет на результат фьючерсной позиции, но если live будет на споте, funding исчезает. Для первичной оптимизации это приемлемо, но финальную проверку нужно делать на спотовых данных.

Что нельзя обещать:
нельзя обещать 100% совпадение между MT5-фьючерсом и BingX Spot.

5. Profit share зависит от правил биржи и High Water Mark.

7. CCXT может не покрыть все биржевые особенности.
8. Binance/Bybit данные для первичной проверки логики не заменяют финальную проверку на BingX.
9. `backtesting.py` не является полной биржевой симуляцией стакана.
10. На первом этапе комиссии можно отключить для проверки логики, но перед live их нужно учитывать.
11. Нельзя обещать гарантированную доходность.
12. Нельзя обещать отсутствие просадок.
13. Нельзя обещать, что защита депозита исключает убытки.
