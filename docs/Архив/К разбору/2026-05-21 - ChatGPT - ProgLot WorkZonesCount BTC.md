## Краткое резюме

В чате разобран параметр `ProgLot_WorkZonesCount` в прогрессивной зонной системе ATLAS / BA. Главный вопрос: является ли параметр косметическим или реально влияет на результат.

Вывод: `ProgLot_WorkZonesCount` не косметический. Он управляет тем, насколько рано внутри нижней части сетки начинается прогрессивное увеличение лота. При этом `W=1` логически остаётся наиболее чистым базовым стандартом: проглот почти не разгоняется внутри обычной сетки и начинает выполнять компенсационную роль ближе к глубокой зоне / форсу. `W=2–3` — не косметика, а ранний разгон экспозиции. `W=4–5` выглядят слишком агрессивно для универсальной конфигурации.

## Точные факты

- Разбираемый параметр: `ProgLot_WorkZonesCount`.
- Комментарий в коде: `ProgLot: количество рабочих зон (последние зоны), 0 = выкл до форс - выхода`.
- Связанные параметры:
  - `ProgLot_Enable`
  - `ProgLot_ZoneWidthSteps`
  - `ProgLot_WorkZonesCount`
  - `ProgLot_ForceZonesCount`
  - `ProgLot_Multiplier`
  - `lowerOrders`
  - `upperOrders`
  - `forced_exits_counter`
- В текущем коде BA 18.2 BUY параметры заданы так:
  - `input bool ProgLot_Enable = true;`
  - `input int ProgLot_ZoneWidthSteps = 1;`
  - `input int ProgLot_WorkZonesCount = 1;`
  - `int ProgLot_ForceZonesCount = lowerOrders;`
  - `input int ProgLot_Multiplier = 1;`
- В текущем коде `GetProgLot()` вызывается при BUY, то есть влияет на размер новых BUY-ордеров.
- `ProgLot_WorkZonesCount` — это количество рабочих зон внизу сетки, а не количество ордеров.
- `ProgLot_ZoneWidthSteps` задаёт ширину одной зоны в шагах.
- Если `ProgLot_ZoneWidthSteps = 3`, то `WorkZonesCount = 2` означает примерно 6 нижних шагов рабочей области.
- `W=0` отключает прогрессию внутри обычной сетки, но не должен блокировать форс-прогрессию после форс-выхода, если форс-зоны включены.
- `W=1` — прогрессия максимально поздняя, ближе всего к логике компенсации после глубокой зоны / форс-выхода.
- `W=2–3` — более ранняя прогрессия внутри сетки.
- `W=4–5` — сильный ранний разгон, высокий риск узких локальных пиков.

## Справочники и соответствия

### Параметр -> смысл

| Параметр | Смысл |
|---|---|
| `ProgLot_WorkZonesCount` | Количество нижних рабочих зон, где начинает действовать прогрессивный лот. |
| `ProgLot_ZoneWidthSteps` | Ширина одной зоны в шагах сетки. |
| `ProgLot_ForceZonesCount` | Количество форс-зон, продолжающих шкалу зон после обычной сетки. |
| `ProgLot_Multiplier` | Коэффициент усиления прогрессивного лота. |
| `forced_exits_counter` | Компенсирует глубину после форс-закрытий, чтобы проглот не откатывался назад из-за уменьшения числа BUY. |
| `lowerOrders` | Глубина обычной сетки вниз. |

### WorkZonesCount -> практический смысл

| Значение W | Интерпретация |
|---:|---|
| `0` | Внутри обычной сетки прогрессии нет. |
| `1` | Базовый консервативный стандарт; проглот включается максимально поздно. |
| `2` | Умеренно раннее усиление; возможный profit-mode. |
| `3` | Агрессивное раннее усиление; выше прибыль, выше DD. |
| `4–5` | Слишком агрессивно для универсальной базы; много риска и узкий живой коридор. |

### Отличие от shift форс-зон

| Рычаг | Что двигает |
|---|---|
| `ProgLot_WorkZonesCount` | Точку старта увеличения лота внутри сетки. |
| `ProgLot_ForceZonesShift` / аналогичный shift | Архитектурную границу форс-области. |
| `RiskGuardStartShiftSteps` | Границу включения RiskGuard / защитного режима. |
| `ProgLot_ForceZonesCount` | Глубину форс-зон после normal-сетки. |

Короткая формула смысла:

```text
WorkZonesCount = раньше/позже начинаем прогрессивный лот
Force/RiskGuard shift = раньше/позже включаем защитный режим
```

## Идеи

- Использовать `W=1` как логически чистый базовый стандарт для универсальной конфигурации ATLAS.
- Рассматривать `W=2` как отдельный режим повышенной прибыльности, но не как автоматическую замену базе.
- Не смешивать тест `WorkZonesCount` с тестом shift форс-зон: внешне оба двигают эффект раньше/позже, но механически меняют разные вещи.
- Исключить `W=4–5` из базового универсального пула, если цель — робастность, а не локальные пики оптимизатора.

## Выводы

- `ProgLot_WorkZonesCount` — не косметический параметр.
- Параметр напрямую меняет экспозицию через ранность включения прогрессивного лота.
- При одинаковой геометрии `step / lowerOrders / wz` изменение `W` меняет прибыль и просадку на значимые величины.
- `W=1` логически правильный стандарт: прогрессия не разгоняется заранее и ближе к компенсации после глубокой зоны / форса.
- `W=2–3` нельзя называть косметикой: это ранний разгон риска.
- Рост прибыли на `W=2–3` не бесплатный: он сопровождается ростом DD и сужением живого коридора.
- `W=4–5` дают слишком мало живых результатов и не подходят для универсальной базы.
- Оптимизатор может показать локально красивые результаты на `W=3–5`, но это может быть не устойчивость, а узкая статистическая щель.
- Для универсальной конфигурации приоритет должен быть не у лучшего одиночного прохода, а у ширины живой зоны.

## Ошибки и решения

### Ошибка: считать W=2–3 косметикой

- Симптом: предположение, что `W=2–3` почти не меняют логику и компенсируются другими параметрами, например шагом.
- Причина: внешне параметр похож на “сдвиг” и может казаться вторичным.
- Решение: сравнивать одинаковую геометрию и менять только `W`.
- Результат: при фиксированных `step / lowerOrders / wz` `W` заметно меняет прибыль и DD.
- Где проявляется: ATLAS / BA, прогрессивный лот, BTC-тест.

### Ошибка: путать WorkZonesCount и shift форс-зон

- Симптом: считать, что `ProgLot_WorkZonesCount` — то же самое, что shift форс-зон.
- Причина: оба параметра создают эффект “раньше/позже”.
- Решение: разделять механику:
  - `WorkZonesCount` двигает старт увеличения лота;
  - shift двигает защитную границу / архитектурный режим.
- Где проявляется: интерпретация тестов ATLAS.

### Ошибка: выбирать W по максимальной прибыли

- Симптом: `W=3` может выглядеть лучшим по profit при `DD < 50%`.
- Причина: ранний крупный лот хорошо выглядит на откате, но опасен на продолжении тренда.
- Решение: использовать фильтры по DD, количество живых проходов, RF и устойчивость на разных периодах.

## Правила

- Не считать `ProgLot_WorkZonesCount` косметическим параметром.
- Для универсальной конфигурации базово держать `ProgLot_WorkZonesCount = 1`, пока другие активы не докажут обратное.
- `W=2` проверять как отдельный агрессивный / profit-mode, а не как автоматический новый стандарт.
- `W=3` считать агрессивным режимом, требующим отдельной проверки.
- `W=4–5` не включать в базовый универсальный пул без сильных доказательств.
- При анализе `WorkZonesCount` обязательно сравнивать фиксированную геометрию:
  - тот же `gridStepPerc`;
  - тот же `lowerOrders`;
  - тот же `ProgLot_ZoneWidthSteps`.
- Не делать вывод по одному лучшему проходу оптимизатора.
- Сравнивать минимум:
  - лучший profit при `DD < 40%`;
  - лучший profit при `DD < 50%`;
  - количество живых результатов;
  - RF;
  - провалы с DD около/выше 100%.

## Архитектура

### Роль ProgLot_WorkZonesCount в архитектуре

`ProgLot_WorkZonesCount` относится к прогрессивной зонной системе. Он не меняет границы режимов защиты, но меняет поведение новых BUY-ордеров в нижней части сетки.

Упрощённая архитектура:

```text
верхние зоны сетки       -> обычный lot_size
нижние рабочие зоны      -> прогрессивный лот
форс-зоны дальше вниз    -> продолжение прогрессии
```

### Чем WorkZonesCount не является

`ProgLot_WorkZonesCount` не является прямым аналогом:

- `ProgLot_ForceZonesShift`;
- `RiskGuardStartShiftSteps`;
- сдвига forced sell;
- изменения глубины normal/force/super;
- изменения момента входа в защитный режим.

Он меняет только ранность роста лота.

## Формулы и параметры

### Формулы из текущей реализации GetProgLot

Фрагменты логики:

```cpp
int zonesInGrid = lowerOrders / zoneWidth;
int forceZones = ProgLot_ForceZonesCount;
int zonesTotal = zonesInGrid + forceZones;
int workZonesEff = ProgLot_WorkZonesCount + forceZones;
int nonWorkTop = zonesTotal - workZonesEff;
```

```cpp
int current_depth = realBuys - upperOrders + forced_exits_counter;
```

```cpp
int zone = 1 + (shifted / zoneWidth);
```

```cpp
if(workZonesEff > 0 && zone > nonWorkTop)
    workZoneIndex = zone - nonWorkTop;
```

```cpp
return NormalizeLot(baseLot * (double)workZoneIndex * (double)mult);
```

### Смысл переменных

| Переменная | Смысл |
|---|---|
| `zoneWidth` | Эффективная ширина зоны в шагах. |
| `zonesInGrid` | Количество зон внутри обычной сетки. |
| `forceZones` | Количество форс-зон. |
| `zonesTotal` | Общее количество зон: сетка + форс-зоны. |
| `workZonesEff` | Эффективное количество рабочих зон с учётом форс-зон. |
| `nonWorkTop` | Верхние нерабочие зоны, где прогрессия не включается. |
| `current_depth` | Текущая глубина с компенсацией форс-выходов. |
| `workZoneIndex` | Индекс рабочей зоны, влияющий на множитель лота. |
| `mult` | `ProgLot_Multiplier`, но не ниже 1. |

### Пример смысла W при zoneWidth=3

Если `ProgLot_ZoneWidthSteps = 3`:

- `W=1` — примерно последние 3 шага сетки становятся рабочей областью проглота.
- `W=2` — примерно последние 6 шагов.
- `W=3` — примерно последние 9 шагов.

Это не точное правило для всех случаев из-за остатков и округлений, но практический смысл именно такой: чем выше `W`, тем раньше начинается прогрессия.

## Возражения и ответы

### Возражение: W=2–3 — косметика

Короткий ответ: нет, это не косметика.

Подробный ответ:

`W=2–3` меняют ранность включения прогрессивного лота. На фиксированной геометрии это увеличивает profit и DD. Это не косметическое изменение, а изменение экспозиции.

Что нельзя обещать:

- нельзя говорить, что `W=2` или `W=3` универсально лучше;
- нельзя выбирать их только по максимальной прибыли;
- нельзя игнорировать рост DD и падение числа живых проходов.

### Возражение: WorkZonesCount — то же самое, что shift форс-зон

Короткий ответ: нет, это похожий внешний эффект, но другой механизм.

Подробный ответ:

`WorkZonesCount` двигает старт роста лота. Shift форс-зон двигает границы защитной архитектуры. Результаты могут быть похожи по статистике, но причина разная.
