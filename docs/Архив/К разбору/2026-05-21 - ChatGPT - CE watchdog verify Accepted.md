## Краткое резюме

В чате обсуждался перенос защитного механизма из сигнального бота в бот Chandelier Exit. Цель — защитить live-работу от ситуации, когда ордер был отправлен, но по нему не пришёл статус `Accepted`.

Ключевое архитектурное решение: для CE не нужна синхронизация реальной позиции через `sync_with_real_position()`, потому что стратегия работает с одним ордером и не ведёт виртуальный счётчик позиций. Watchdog должен отслеживать только отсутствие подтверждения `Accepted` по `pending_order`.

## Точные факты

- Актуальный файл CE в обсуждении: `CE 7 4 light.py`.
- Источник идеи защиты: `Signal v6 7 BUY.py`.
- Защита переносится в Chandelier Exit / CE Adaptive.
- Платформа реализации: Python / Backtrader.
- Проверенная функция перезапуска потоков уже есть:
  - `_restart_streams(self, *_, **__)`
- Эту функцию менять не нужно.
- `sync_with_real_position()` для CE не нужна.
- Причина отказа от `sync_with_real_position()`:
  - в CE только один ордер;
  - нет виртуального счётчика;
  - нечего синхронизировать как сетку или серию ордеров.
- Статус, который нужен для подтверждения живого канала:
  - `Accepted`.
- `need_verify` не должен ставиться при самой отправке ордера.
- Правильная логика пользователя:
  - ордер отправлен;
  - если статус `Accepted` не пришёл;
  - тогда включается режим проверки `need_verify`;
  - по `need_verify` запускается таймер и reconnect-логика.
- Back-off:
  - `30 * self.verify_attempt`
  - попытки дают 30 / 60 / 90 секунд.
- Максимум попыток:
  - `3`.
- После 3 неудачных попыток:
  - отменить `pending_order`;
  - сбросить `pending_order`;
  - сбросить `need_verify`;
  - сбросить `verify_attempt`;
  - установить `verify_due = datetime.min`.
- `verify_due = None` использовать нельзя.
- Причина:
  - потом возможна ошибка сравнения `datetime.utcnow() < self.verify_due`, если `verify_due` равен `None`.

## Справочники и соответствия

### Переменные watchdog

- `self.pending_order`
  - Смысл: ссылка на ордер, по которому ещё ожидается обработка.
  - В логике CE используется как признак: есть незавершённый ордер или статус ещё не обработан.
  - Если `pending_order is None`, значит статус пришёл или ордер очищен.

- `self.need_verify`
  - Смысл: режим проверки после обнаружения проблемы с подтверждением ордера.
  - Не должен включаться сразу при отправке ордера.
  - Должен включаться, когда обнаружено, что ордер есть, но нужный статус `Accepted` не пришёл.

- `self.verify_attempt`
  - Смысл: номер попытки проверки / reconnect.
  - Используется для back-off.
  - Ограничивается значением `3`.

- `self.verify_due`
  - Смысл: время следующей проверки.
  - Должен всегда быть объектом `datetime`.
  - Для сброса использовать `datetime.min`.
  - Не использовать `None`.

- `self._restart_streams(self, *_, **__)`
  - Смысл: проверенная рабочая функция перезапуска streams.
  - Решение: не менять.

- `self.sync_with_real_position()`
  - Смысл: синхронизация внутреннего состояния с реальной позицией.
  - Решение для CE: не использовать.
  - Причина: в CE один ордер, нет виртуального счётчика.

### Статус ордера

- `Accepted`
  - Смысл: ACK / подтверждение, что брокер принял ордер.
  - Именно этот статус нужен как признак живого канала.
  - Если `Accepted` не пришёл, запускается verify/reconnect-логика.

### Back-off

- Формула:
  - `seconds = 30 * self.verify_attempt`
- Значения:
  - попытка 1 → 30 секунд;
  - попытка 2 → 60 секунд;
  - попытка 3 → 90 секунд.
- После 3 неудач:
  - force clear / reset.

## Идеи

### Идея 1 — перенести защиту из Signal Bot в CE

Суть: использовать watchdog-защиту из сигнального бота для Chandelier Exit, чтобы live-бот не зависал при потере статусов ордеров.

Статус: принято к реализации.

### Идея 2 — отслеживать не позицию, а статус ордера

Суть: для CE достаточно контролировать получение `Accepted`, потому что стратегия не ведёт сетку и виртуальные уровни.

Статус: принято.

### Идея 3 — VERIFY включать не при отправке, а при обнаружении проблемы

Суть: `need_verify` не должен выставляться сразу после `buy()` / `sell()`. Он должен включаться только когда в `next()` обнаружено, что есть `pending_order`, но статус подтверждения не пришёл.

Статус: принято как требование пользователя.

### Идея 4 — полный сброс после 3 неудач

Суть: если после нескольких reconnect-попыток статус всё равно не пришёл, нужно не продолжать бесконечный reconnect, а принудительно очистить состояние.

Статус: принято.

## Выводы

- Для CE watchdog должен быть проще, чем для Signal Grid / сеточного бота.
- В CE не нужен `sync_with_real_position()`.
- Главный контролируемый факт — пришёл ли статус `Accepted`.
- `_restart_streams(self, *_, **__)` уже рабочая, её не надо переписывать или заменять заглушкой.
- `need_verify` должен обозначать именно режим проверки после отсутствия подтверждения, а не сам факт отправки ордера.
- `verify_due` должен всегда оставаться `datetime`.
- `verify_due = None` — ошибка архитектуры.
- После 3 неудачных VERIFY-попыток нужен force clear:
  - отменить ордер;
  - очистить `pending_order`;
  - выключить `need_verify`;
  - сбросить `verify_attempt`;
  - вернуть `verify_due` в безопасное значение `datetime.min`.

## Ошибки и решения

### Ошибка 1 — предлагалось использовать `sync_with_real_position()`

- Симптом: в логике CE появляется синхронизация позиции, хотя стратегия работает с одним ордером.
- Причина: механика была взята из сигнального/сеточного бота, где есть дополнительные состояния.
- Решение:
  - убрать `sync_with_real_position()`;
  - не вызывать `sync_with_real_position(fix=1)`.
- Где проявляется:
  - CE watchdog / VERIFY-блок.

### Ошибка 2 — `verify_due = None`

- Симптом: потенциальный `TypeError` при сравнении:
  - `datetime.utcnow() < self.verify_due`
- Причина:
  - `verify_due` после сброса становится `None`.
- Решение:
  - использовать `self.verify_due = datetime.min`.
- Где проявляется:
  - блок watchdog в `next()` после 3 неудач.

### Ошибка 3 — менять рабочую `_restart_streams`

- Симптом: была предложена новая заглушка `_restart_streams()`.
- Причина:
  - не было учтено, что рабочая функция уже скопирована.
- Решение:
  - оставить существующую `def _restart_streams(self, *_, **__)`;
  - вызывать её как `self._restart_streams()`.
- Где проявляется:
  - архитектура reconnect-защиты.

### Ошибка 4 — ставить `need_verify` сразу при отправке ордера

- Симптом: VERIFY начинается превентивно.
- Причина:
  - неверная трактовка `need_verify` как флага ожидания любого ACK после отправки.
- Решение:
  - не ставить `need_verify = True` в `_enter_long` / `_enter_short`;
  - включать `need_verify` только когда обнаружено, что ордер есть, а `Accepted` не пришёл.
- Где проявляется:
  - методы отправки ордера и начало `next()`.

### Ошибка 5 — бесконечный reconnect без force clear

- Симптом: стратегия может бесконечно перезапускать streams.
- Причина:
  - нет лимита попыток VERIFY.
- Решение:
  - ограничить `verify_attempt` до 3;
  - после 3 попыток отменить и очистить `pending_order`;
  - сбросить VERIFY-состояние.
- Где проявляется:
  - watchdog / reconnect loop.

## Правила

### Правило 1 — не переносить сеточную синхронизацию в одиночную стратегию

Если бот не ведёт виртуальную сетку, счётчик ордеров или внутреннюю модель позиции, не добавлять `sync_with_real_position()` только потому, что она есть в другом боте.

### Правило 2 — `_restart_streams` не менять, если функция уже проверена

Если в стратегии уже есть рабочая функция:
```python
def _restart_streams(self, *_, **__):
```
то при переносе защиты её не заменять заглушками и не переписывать без отдельной причины.

### Правило 3 — `verify_due` всегда должен быть `datetime`

Запрещено присваивать:
```python
self.verify_due = None
```

Для сброса использовать:
```python
self.verify_due = datetime.min
```

### Правило 4 — `need_verify` включать только после обнаружения отсутствия подтверждения

Не ставить:
```python
self.need_verify = True
```
сразу после отправки ордера, если архитектурная логика требует включать VERIFY только при фактическом отсутствии `Accepted`.

### Правило 5 — после 3 неудачных VERIFY-попыток нужен force clear

Минимальный сброс:
```python
self.cancel(self.pending_order)
self.pending_order  = None
self.need_verify    = False
self.verify_attempt = 0
self.verify_due     = datetime.min
```

### Правило 6 — при активном VERIFY торговую логику пропускать

Если VERIFY ещё ждёт таймер или выполняет reconnect, остальная логика `next()` должна быть пропущена через `return`.

## Архитектура

### Выбранная схема watchdog для CE

Компоненты:

1. `pending_order`
   - хранит текущий ордер;
   - если не `None`, значит есть незавершённый ордер.

2. `need_verify`
   - показывает, что бот уже вошёл в режим проверки из-за отсутствия `Accepted`.

3. `verify_due`
   - задаёт время следующей проверки.

4. `verify_attempt`
   - считает попытки reconnect/verify.

5. `_restart_streams()`
   - перезапускает streams;
   - уже проверена и не должна меняться.

6. Force clear после 3 попыток
   - отменяет pending order;
   - очищает состояние;
   - не вызывает `sync_with_real_position()`.

### Логика выполнения в `next()`

1. Проверить:
   ```python
   if self.pending_order is not None or self.need_verify:
   ```

2. Если `need_verify == True` и время ещё не пришло:
   ```python
   if self.need_verify and datetime.utcnow() < self.verify_due:
       return
   ```

3. Если `pending_order is None`:
   - статус пришёл;
   - логировать `VERIFY OK`;
   - сбросить `need_verify`;
   - сбросить `verify_attempt`;
   - не делать `return`;
   - перейти к основной торговой логике.

4. Если `pending_order is not None`:
   - значит канал всё ещё немой или статус не пришёл;
   - логировать `NO-STATUS for order` или `VERIFY FAIL`;
   - вызвать `_restart_streams()`;
   - увеличить `verify_attempt`;
   - если попыток 3 или больше:
     - отменить pending order;
     - очистить watchdog state;
   - иначе:
     - включить `need_verify`;
     - выставить `verify_due`;
   - сделать `return`.

### Обсуждённые альтернативы

#### Альтернатива 1 — включать `need_verify` сразу после отправки ордера

Отвергнута.

Причина:
- пользователь считает это неправильным;
- `need_verify` должен означать не факт отправки, а факт проблемы: ордер отправлен, но `Accepted` не пришёл.

#### Альтернатива 2 — использовать `sync_with_real_position()`

Отвергнута.

Причина:

- нет счётчика;
- нет виртуальной сетки;

Причина:

## Формулы и параметры

### VERIFY back-off

Переменные:
- `self.verify_attempt` — номер попытки;
- `self.verify_due` — время следующей проверки.

Формула:
```python
self.verify_due = datetime.utcnow() + timedelta(seconds=30 * self.verify_attempt)
```

Смысл:
- чем больше попыток, тем дольше пауза перед следующей проверкой.

Ограничения:
- `verify_attempt` ограничивается максимумом 3:
```python
self.verify_attempt = min(self.verify_attempt + 1, 3)
```

Пример:
- attempt 1 → 30 секунд;
- attempt 2 → 60 секунд;
- attempt 3 → 90 секунд.

### Force clear после 3 попыток

Условие:
```python
if self.verify_attempt >= 3:
```

Действия:
```python
self.cancel(self.pending_order)
self.pending_order  = None
self.need_verify    = False
self.verify_attempt = 0
self.verify_due     = datetime.min
```

Смысл:
- не допустить бесконечного reconnect-loop;
- очистить зависший ордер;
- вернуть стратегию в безопасное состояние.

## Цитаты или фрагменты

### Финальный проверяемый фрагмент watchdog-блока

```python
# 1)► Есть ли незавершённый ордер ИЛИ мы уже в режиме VERIFY?
if self.pending_order is not None or self.need_verify:

    # 0.1  Проверяем, пора ли делать VERIFY
    if self.need_verify and datetime.utcnow() < self.verify_due:
        return                      # ещё ждём таймер

    # 0.2  Канал жив: статус пришёл → снимаем флаги и синхронизируемся
    if self.pending_order is None:
        self.log("VERIFY OK - connection restored")
        self.need_verify    = False
        self.verify_attempt = 0
        # НЕ return → идём к торговой логике

    # 0.3  Канал всё ещё «немой»
    else:
        msg = "NO-STATUS for order" if not self.need_verify else "VERIFY FAIL"
        self.log(f"{msg} - restarting streams")
        self._restart_streams()

        # Экспоненциальный back-off 30/60/90 с
        self.verify_attempt = min(self.verify_attempt + 1, 3)
        if self.verify_attempt >= 3:
            self.cancel(self.pending_order)
            self.log("VERIFY FAIL x3 - force clear pending_order & resync")
            self.pending_order  = None
            self.need_verify    = False
            self.verify_attempt = 0
            self.verify_due = datetime.min
        else:
            self.need_verify = True
            self.verify_due  = datetime.utcnow() + timedelta(seconds=30 * self.verify_attempt)
        return                  # пропускаем остальную логику
```

### Комментарий к фрагменту

Фрагмент в целом соответствует выбранной логике:
- `need_verify` не ставится при отправке;
- VERIFY запускается через обнаружение `pending_order`;
- при активном таймере торговая логика пропускается;
- после 3 попыток состояние сбрасывается;
- `verify_due` сбрасывается в `datetime.min`, а не в `None`;
- `sync_with_real_position()` отсутствует.

Остаётся проверить только безопасную отмену:
```python
self.cancel(self.pending_order)
```

Более безопасный вариант при возможных гонках:
```python
o = self.pending_order
self.pending_order = None
if o is not None:
    self.cancel(o)
```
