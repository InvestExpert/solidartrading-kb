## Краткое резюме

В чате разбиралась ошибка Bollinger Buster ADX: бот должен работать с одной сделкой, но иногда появляется две и больше. Пользователь уточнил, что проблема не в связанных стопах и тейках, а именно в основных ордерах.

Главный технический вывод чата: ранний блок

```python
if self.order:
    self.log("SKIP: active order pending")
    return
```

стоит слишком высоко в `next()` и может не дать сработать ветке выхода из позиции. Проверку активного ордера нужно перенести в ветку открытия позиции:

```python
if current_position_size == 0 and not self.order:
```

Так блокировка будет запрещать только новый вход, но не будет глушить выход из уже открытой позиции.

## Идеи

### Идея 1 — блокировать только вход, а не весь `next()`

Суть: проверка активного ордера нужна не как глобальная остановка стратегии, а только как запрет нового входа, когда бот находится вне позиции.

Минимальная форма:

```python
if current_position_size == 0 and not self.order:
```

Смысл:
- если позиции нет и активного ордера нет — можно искать вход;
- если позиция есть — нужно дать отработать логике выхода;
- если позиции нет, но ордер висит — новый вход запрещен.

### Идея 2 — сначала выход, потом вход

Суть: торговый цикл должен иметь приоритет выхода. При открытой позиции сначала проверяется `exit_signal`; только если позиции нет, рассматриваются входы.

Примерная архитектура:

```python
if current_position_size != 0:
    # обработка выхода
    ...
else:
    # обработка входа
    ...
```

или более жестко:

```python
# 1. Сначала выход
if current_position_size != 0:
    if exit_signal:
        self.order = self.close()
    return

# 2. Потом вход
if current_position_size == 0 and not self.order:
    ...
```

Риск: если сделать `return` после выхода, нужно проверить, что это не ломает нужные служебные действия ниже по `next()`.

### Идея 3 — разделить `entry_order` и `exit_order`

Суть: `self.order` как единая переменная может путать входные и выходные ордера. Более надежная архитектура — хранить отдельно:
- `self.entry_order`
- `self.exit_order`

Статус: идея не внедрялась в чате, но отмечена как более надежная альтернатива.

## Гипотезы

### Гипотеза 1 — множественные сделки вызваны зависшими SL/TP-хвостами

Суть гипотезы: после закрытия позиции защитные стоп/тейк-ордера остаются активными и могут создавать новые позиции.

Относится к: Bollinger Buster ADX / Backtrader.

Статус: отвергнута пользователем.

Причина отклонения:
- пользователь прямо указал, что проблемы со связанными ордерами нет;
- пользователь указал, что стопы закрываются вместе с основным ордером;
- проблема только с основными ордерами.

Как проверять, если гипотеза всплывет снова:
- в логах сравнить ref/id основного ордера, SL и TP;
- проверить, остаются ли SL/TP после `self.close()`;
- отдельно проверить, создают ли именно SL/TP новую позицию.

### Гипотеза 2 — повторные основные ордера возникают из-за общего флага `self.order`

Суть гипотезы: `self.order` используется как общий индикатор активности, но стоит в неправильном месте и блокирует не только вход, но и выход.

Относится к: Bollinger Buster ADX / Backtrader.

Статус: частично подтверждена логикой кода и принята как направление правки.

Проверка:
- найти место, где стоит:
  ```python
  if self.order:
      return
  ```
- проверить, находится ли этот блок до обработки выхода;
- в тесте получить ситуацию, где `exit_signal=True`, но `self.order` не пустой;
- убедиться, что после переноса проверки внутрь входной ветки `exit_signal` больше не глушится.

### Гипотеза 3 — лучше сначала обрабатывать выход, затем вход

Суть гипотезы: порядок `exit -> entry` уменьшит риск одновременного закрытия и открытия на одной свече.

Относится к: Bollinger Buster ADX / Backtrader.

Статус: не проверена.

Проверка:
- перестроить `next()` так, чтобы при `current_position_size != 0` сначала обрабатывался выход;
- после отправки `self.close()` делать `return`;
- прогнать тот же лог/тест и сравнить количество основных ордеров.

## Выводы

- Общий ранний блок `if self.order: return` нельзя ставить перед всей торговой логикой, если ниже есть отдельная ветка выхода из позиции.
- Проверка активного ордера должна защищать только от повторного входа, а не блокировать выход.
- Минимальная правка:
  ```python
  if current_position_size == 0 and not self.order:
  ```
- Более надежная архитектура:
  - отдельно учитывать входной ордер;
  - отдельно учитывать выходной ордер;
  - сначала обрабатывать выход;
  - потом обрабатывать вход.
- Диагностика SL/TP как основной причины была ошибочной для этого чата и должна быть помечена как отвергнутая.
- `self.close()` в логике пользователя рассматривается как закрытие всей позиции. Поэтому при дальнейшей диагностике нельзя подменять проблему основного ордера проблемой связанных SL/TP без фактов из логов.

## Ошибки и решения

### Ошибка 1 — глобальный стоппер `self.order`

Симптом:
- бот не выходит по сигналу;
- при этом позже может появляться несколько основных сделок или некорректная последовательность ордеров.

Проблемный код:

```python
# Игнорируем, если висит активный ордер
if self.order:
    self.log("SKIP: active order pending")
    return
```

Причина:
- блок стоит до ветки входа/выхода;
- при любом `self.order` весь `next()` прекращается;
- код не доходит до `else`, где должен обрабатываться выход из позиции.

Решение:
- убрать этот блок из общего начала `next()`;
- перенести проверку ордера в ветку открытия позиции:

```python
if current_position_size == 0 and not self.order:
    # логика открытия
```

Где проявляется:
- Bollinger Buster ADX / Python / Backtrader.

### Ошибка 2 — смешение смысла `self.order`

Симптом:
- сложно понять, какой именно ордер висит: входной, выходной или старый уже завершенный;
- можно случайно заблокировать вход или выход.

Причина:
- одна переменная `self.order` используется как общий индикатор активного ордера.

Решение:
- минимально: аккуратно очищать `self.order` в `notify_order`;
- надежнее: разделить на `entry_order` и `exit_order`.

Где проявляется:
- потенциально во всех Backtrader-стратегиях с отдельными входами и выходами.

### Ошибка 3 — неправильное направление диагностики

Симптом:
- анализ ушел в защитные SL/TP-ордера, хотя пользователь указал, что проблема в основных ордерах.

Причина:
- преждевременное предположение по типичному сценарию с bracket/OCO без подтверждения конкретным логом.

Решение:
- при следующем аудите сначала фиксировать ограничение пользователя:
  - SL/TP не считать главной причиной;
  - анализировать только основные ордера;
  - проверять цепочку `current_position_size`, `self.order`, `self.close()`, `notify_order`.

## Правила

- Если входы выполняются только в блоке `if current_position_size == 0`, то проверку активного входного ордера нужно размещать внутри этой ветки или включать прямо в условие.
- Нельзя ставить общий `return` по `self.order` выше логики выхода, если выход находится в `else`-ветке.
- Выход из позиции должен иметь приоритет над поиском нового входа.
- Проверка `self.order` должна отвечать на вопрос “можно ли открывать новый вход”, а не “можно ли вообще выполнять `next()`”.
- Если используется одна переменная `self.order`, нужно гарантировать ее очистку при финальных статусах ордера:
  - `Completed`
  - `Canceled`
  - `Rejected`
  - `Margin` при наличии такого статуса в используемой логике.
- Не переносить в wiki как факт неподтвержденный вывод о зависших SL/TP-ордерах; только как отвергнутую гипотезу.
- Перед изменением кода проверять всю цепочку:
  - входы;
  - выходы;
  - статус позиции;
  - статус `self.order`;
  - `notify_order`;
  - `self.close()`;
  - повторный вызов `next()`.

## Архитектура

### Текущая схема, обсуждаемая в чате

```python
# где-то выше стоял глобальный блок
if self.order:
    return

if current_position_size == 0:
    # открытие позиции
else:
    # выход из позиции
```

Проблема:
- глобальный блок может остановить выполнение до обработки выхода.

### Минимальная целевая схема

```python
if current_position_size == 0 and not self.order:
    # открытие позиции
else:
    # обработка выхода или отсутствие входа
```

Смысл:
- вход возможен только когда нет позиции и нет активного ордера;
- при открытой позиции логика не блокируется `self.order` и может дойти до выхода.

### Более надежная целевая схема

```python
# 1. Обработка выхода
if current_position_size != 0:
    if exit_signal:
        self.exit_order = self.close()
    return

# 2. Обработка входа
if current_position_size == 0 and not self.entry_order:
    # открытие позиции
```

Плюсы:
- выход имеет приоритет;
- вход и выход не смешиваются;
- меньше риск повторного входа на той же свече.

Минусы:
- больше правок;
- нужно проверить совместимость с остальными действиями в `next()`;
- нужно проверить `notify_order` для двух переменных.

Логика:
- вход по пробою Bollinger Bands при ADX выше порога;
- выход по средней линии Bollinger Bands;
- TP/SL через `strategy.exit`;
- запрет пирамидинга через `pyramiding=1`.

## Формулы и параметры

### Bollinger Bands

```pine
basis = ta.sma(close, bbPeriod)
dev = bbDev * ta.stdev(close, bbPeriod)
upperBB = basis + dev
lowerBB = basis - dev
midBB = basis
```

Переменные:
- `bbPeriod = 24`
- `bbDev = 1.7`

Смысл:
- `basis/midBB` — средняя линия;
- `upperBB` — верхняя Bollinger Band;
- `lowerBB` — нижняя Bollinger Band.

### ADX

```pine
[_, _, adx] = ta.dmi(diLength, adxSmooth)
```

Переменные:
- `diLength = 14`
- `adxSmooth = 14`
- `adxThreshold = 26`

Смысл:
- ADX используется как фильтр силы движения;
- входы разрешены только при `adx > adxThreshold`.

### Long-сигнал

```pine
longSignal = ta.crossunder(close, lowerBB) and close < lowerBB and adx > adxThreshold
```

Смысл:
- цена пересекает нижнюю BB вниз;
- цена закрытия ниже нижней BB;
- ADX выше порога.

### Short-сигнал

```pine
shortSignal = ta.crossover(close, upperBB) and close > upperBB and adx > adxThreshold
```

Смысл:
- цена пересекает верхнюю BB вверх;
- цена закрытия выше верхней BB;
- ADX выше порога.

### Выход из Long

```pine
exitLong = strategy.position_size > 0 and close > midBB and close[1] > midBB
```

Смысл:
- long-позиция есть;
- текущий и предыдущий close выше средней BB.

### Выход из Short

```pine
exitShort = strategy.position_size < 0 and close < midBB and close[1] < midBB
```

Смысл:
- short-позиция есть;
- текущий и предыдущий close ниже средней BB.

### Take Profit

Long:

```pine
limit = close * (1 + takeProfitPct)
```

Short:

```pine
limit = close * (1 - takeProfitPct)
```

Параметр:
- `takeProfitPct = 0.024`

Смысл:
- TP равен 2.4% от цены входа/текущей цены при генерации ордера.

### Stop Loss

Long:

```pine
stop = ta.lowest(low, swingLookback) - slBufferPct
```

Short:

```pine
stop = ta.highest(high, swingLookback) + slBufferPct
```

Параметры:
- `swingLookback = 8`
- `slBufferPct = 0.001`

Смысл:
- Long SL ставится ниже минимального low за lookback с буфером;
- Short SL ставится выше максимального high за lookback с буфером.

Ограничение:
- В коде `slBufferPct` используется как абсолютный вычет/добавка, хотя название содержит `%`. Это нужно проверить отдельно: параметр может быть не процентом, а абсолютным буфером.

## Риски, ограничения и дисклеймеры

- Недостаточно данных для надежного вывода по полной причине множественных основных ордеров без повторного анализа полного лога и текущей версии `Bollinger Buster ADX 1.3 .py`.
- Вывод по `if current_position_size == 0 and not self.order` является минимальной логической правкой, а не полной гарантией отсутствия дублей.
- Если `self.order` не очищается в `notify_order`, бот может залипнуть вне рынка и не открывать новые сделки.
- Если `self.order` хранит exit-ордер, а позиция уже стала 0, вход будет заблокирован до очистки `self.order`. Это может быть как правильной защитой, так и причиной пропуска входа — зависит от ожидаемой логики.
- Если после `self.close()` брокер/Backtrader обновляет `position.size` с задержкой, возможны повторные close-ордера. Это отдельный риск, не закрытый только переносом `if self.order`.
- Если в одной переменной `self.order` смешаны entry и exit, диагностика остается слабой.
- `pyramiding=1` в TradingView не равен полной защите от дублей в Python/Backtrader. Поведение платформ отличается.

## Цитаты или фрагменты

Проблемный фрагмент:

```python
# Игнорируем, если висит активный ордер
if self.order:
    self.log("SKIP: active order pending")
    return
```

Ключевая правка:

```python
if current_position_size == 0 and not self.order:
```

Альтернативная архитектурная идея:

```python
# сначала выход
if current_position_size != 0:
    if exit_signal:
        self.order = self.close()
    return

# потом вход
if current_position_size == 0 and not self.order:

```
