## Краткое резюме

Видео объясняет, когда использовать обычный чат, skills, агентов, субагентов и MCP.

Главная идея: не нужно усложнять рабочую систему ради модных терминов. Нужно выбирать самый простой инструмент, который достаточно хорошо решает задачу.

Для TiTan Bots материал полезен не как техническая информация о торговых роботах, а как методика построения ИИ-инфраструктуры вокруг проекта:

- база знаний;
- Telegram/RAG-ассистент;
- шаблоны повторяющихся задач;
- автоматизация контента;
- обработка багов;
- подготовка задач для Codex или другого кодового агента;
- будущие интеграции с Notion, Google Sheets, CRM, API и аналитикой.

## Точные факты

- Видео посвящено разнице между:
  - обычным чатом;
  - skills;
  - агентами;
  - MCP;
  - CLI;
  - субагентами.
- Главный тезис ролика: не существует «правильного» универсального инструмента. Есть инструмент, подходящий под конкретную задачу.
- Если задачу можно решить обычным чатом, не нужно строить сложную агентную систему.
- Агент отличается от обычного чата доступом к инструментам и циклом выполнения действий.
- Агент может работать с:
  - браузером;
  - файловой системой;
  - настройками;
  - кодом;
  - внешними инструментами.
- Цикл агента в ролике описан как: подумал → сделал → повторил.
- Skills в ролике описаны как сохранённый промпт с большим контекстом.
- Skills нужны, когда задача повторяется второй, третий и последующие разы.
- MCP описан как протокол/прослойка, которая даёт модели доступ к внешним системам.
- Через MCP можно выгружать данные из внешних систем или загружать данные во внешние системы.
- Примеры внешних систем из ролика:
  - Notion;
  - Gmail;
  - CRM;
  - своя база;
  - своё API;
  - аналитика YouTube.
- Субагенты нужны, когда задачу можно разбить на независимые части и выполнять параллельно.
- Пример из ролика: агент каждый день ищет темы для YouTube, анализирует рекомендации, добавляет данные в Notion, а субагенты создают названия, идеи обложек и сценарий.
- Пример из ролика: для повторяющегося стиля обложек создаётся skill, который хранит правила стиля и генерирует промпт для картинки.
- Пример из ролика: баги из Notion можно превратить в единое техническое задание и master-prompt для кодового агента.
- CLI в ролике заявлен в названии, но содержательно почти не раскрыт.

## Справочники и соответствия

### Инструмент -> когда использовать

| Инструмент | Когда использовать | Смысл для TiTan Bots |
|---|---|---|
| Обычный чат | Разовая задача, вопрос, краткий анализ, текст | Быстрые разовые ответы по контенту, логике, текстам, идеям |
| Skill | Повторяющаяся задача с одинаковыми правилами | Шаблоны аудита кода, анализа тестов, промптов для превью, YouTube-описаний |
| Агент | Нужно выполнить цепочку действий автономно | Прочитать файлы, найти проблему, подготовить отчёт, предложить минимальные правки |
| Субагенты | Задача делится на независимые параллельные части | Параллельно анализировать баги, логи, код, тесты, контент |
| MCP | Нужен доступ к внешним системам | Подключить Notion, Google Sheets, Gmail, CRM, аналитику, API |
| CLI | Работа через терминал/командную строку | В видео почти не раскрыт; для TiTan Bots не считать ключевым выводом |

### Задача -> подходящий слой

| Задача | Подходящий слой |
|---|---|
| Разово спросить или проверить идею | Обычный чат |
| Постоянно писать YouTube-описания в одном стиле | Skill |
| Постоянно делать превью/промпты в одном стиле | Skill |
| Автоматически читать список багов и готовить ТЗ | Агент |
| Параллельно проверить несколько багов | Субагенты |
| Получать данные из Notion/таблиц/API | MCP |
| Делать Telegram-бота с ответами по базе проекта | RAG-база, затем агент/MCP при необходимости |

### Уровни усложнения

1. Обычный чат.
2. Skill для повторяющихся задач.
3. RAG-база для поиска по материалам.
4. Агент для автономной цепочки действий.
5. Субагенты для параллельной обработки.
6. MCP для связи с внешними системами.
7. CLI/кодовая среда для глубокой автоматизации.

## Идеи

### Идея 1. Набор skills для TiTan Bots

Создать набор стабильных шаблонов для повторяющихся задач TiTan Bots.

Возможные skills:

- аудит функции торгового бота;
- сравнение двух версий кода;
- анализ MT5 vs Python;
- анализ результатов оптимизатора;
- разбор логов сделок;
- генерация YouTube-описания;
- генерация структуры презентации;
- генерация промпта для превью;
- подготовка материала для Obsidian wiki;
- разбор возражений клиента;
- подготовка поста для Telegram.

Смысл: не объяснять каждый раз правила заново.

### Идея 2. Skill для аудита кода ботов

Отдельный skill должен жёстко задавать порядок проверки:

1. Входы функции.
2. Выходы функции.
3. Вызовы соседних функций.
4. Глобальные флаги.
5. Recovery-логика.
6. Совместимость с соседними блоками.
7. Минимальная правка.
8. Проверка риска поломки.

Смысл: закрепить стиль аудита, который не даёт ИИ перескакивать к исправлениям без проверки цепочки.

### Идея 3. Skill для анализа optimizer-таблиц

Skill должен задавать правила:

- не выбирать одиночный лучший результат;
- искать плато;
- сравнивать DD, прибыль, RF и стабильность;
- учитывать относительное улучшение против базы;
- разделять универсальные и инструментальные параметры;
- не смешивать gridStepPerc/lowerOrders с универсальными настройками.

Смысл: стандартизировать анализ тестов.

### Идея 4. Skill для контента SolidarTrading / TiTan Bots

Skill должен хранить:

- стиль подачи;
- структуру видео;
- ограничения по обещаниям доходности;
- правила эпатажа без гарантии результата;
- формат SEO-описаний;
- формат ключевых слов;
- структуру постов;
- дисклеймеры.

Смысл: ускорить создание контента и сохранить единый стиль.

### Идея 5. RAG-база для Telegram-ассистента

Сначала достаточно сделать базу знаний с поиском по материалам проекта.

Минимальная структура:

- информация о ботах;
- инструкции;
- презентации;
- частые вопросы;
- возражения;
- ограничения;
- отчёты;
- метод использования;
- партнёрские ссылки;
- дисклеймеры.

MCP нужен только позже, если ассистент должен не просто отвечать, а работать с внешними системами.

### Идея 6. Агентный аудит багов

Схема:

1. Баги фиксируются в Notion, Obsidian или таблице.
2. Агент читает список багов.
3. Агент отделяет реальные баги от мусорных или неполных записей.
4. Агент группирует баги по модулю, версии и риску.
5. Агент создаёт единое ТЗ.
6. Агент создаёт master-prompt для Codex/кодового агента.
7. Кодовый агент работает по ТЗ.
8. Человек проверяет результат.

Смысл: убрать ручную первичную сортировку багов, но не отдавать код на полный автопилот.

### Идея 7. Аналитика контента по деньгам, а не просмотрам

Для продвижения TiTan Bots полезно оценивать видео и посты не только по просмотрам.

Нужно отслеживать:

- переходы по ссылкам;
- конверсию в заявки;
- конверсию в регистрацию;
- конверсию в покупку;
- партнёрскую активность;
- доход с конкретного ролика/поста.

Смысл: просмотры могут быть пустыми, если они не дают заявок или продаж.

## Гипотезы

### Гипотеза 1. Для TiTan Bots сейчас полезнее skills и RAG, чем MCP

Суть гипотезы: на текущем этапе TiTan Bots больше выиграет от набора шаблонов и базы знаний, чем от сложной интеграции MCP.

К чему относится: Метод и продвижение, Инкубатор, база знаний TiTan Bots.

Чем проверить:

- сделать 5-10 повторяемых skills;
- сделать простую RAG-базу;
- протестировать Telegram-ассистента на реальных вопросах;
- только после этого оценить, каких внешних действий не хватает.

Статус: не проверена.

### Гипотеза 2. MCP нужен только после появления конкретной внешней системы

Суть гипотезы: MCP не нужно внедрять абстрактно. Он нужен, когда есть точная внешняя система и понятное действие: прочитать, записать, обновить, выгрузить.

К чему относится: архитектура ИИ-инфраструктуры TiTan Bots.

Чем проверить:

- составить список систем: Notion, Obsidian, Google Sheets, CRM, Gmail, Telegram;
- для каждой системы указать действие;
- если действий нет, MCP не нужен.

Статус: частично подтверждается логикой ролика.

### Гипотеза 3. Агентная обработка багов может ускорить разработку, но только с ручным ревью

Суть гипотезы: агент может сократить время на сортировку багов и подготовку ТЗ, но прямое исправление кода без проверки опасно.

К чему относится: Техника, архитектура разработки.

Чем проверить:

- взять 10-20 багов;
- прогнать через агентную схему;
- сравнить качество ТЗ с ручным разбором;
- дать агенту только подготовить ТЗ, а не сразу править код;
- оценить количество неверных выводов.

Статус: не проверена.

### Гипотеза 4. Единые skills снизят разброс качества контента

Суть гипотезы: если зафиксировать структуру постов, описаний, превью и дисклеймеров, качество материалов станет стабильнее.

К чему относится: Метод и продвижение.

Чем проверить:

- создать 3-5 skills для контента;
- сделать серию постов/описаний;
- сравнить скорость подготовки и качество против ручного варианта.

Статус: не проверена.

## Выводы

- Не нужно начинать с MCP, если задача решается обычным чатом, skill или RAG-базой.
- Повторяющиеся задачи нужно упаковывать в skills.
- Агент нужен не для красивого термина, а когда есть цепочка действий и доступ к инструментам.
- Субагенты полезны только там, где части задачи реально независимы.
- MCP нужен для связи ИИ с внешними системами.
- CLI в данном видео не даёт практического вывода.
- Для TiTan Bots самый практичный порядок внедрения:
  1. skills;
  2. RAG-база;
  3. Telegram-ассистент;
  4. агентные сценарии;
  5. MCP для конкретных внешних систем.
- Автономный агент для кода нельзя использовать без контроля.
- Для продвижения TiTan Bots полезна аналитика не только просмотров, но и денег/заявок/конверсий.
- Материал подходит больше для слоя «Метод и продвижение» и «Инкубатор», чем для технического слоя торговых стратегий.

## Ошибки и решения

### Ошибка 1. Усложнение ради модных терминов

Симптом: вместо простой задачи строится сложная система из агентов, MCP и CLI.

Причина: желание использовать модный инструмент без точной необходимости.

Решение: выбирать минимальный инструмент, который закрывает задачу.

Где проявляется: проектирование базы знаний, Telegram-бота, ИИ-ассистента, обработки багов.

### Ошибка 2. Использование MCP без внешней системы

Симптом: обсуждается MCP, но нет конкретного API, базы, CRM или действия.

Причина: преждевременная архитектура.

Решение: сначала определить внешнюю систему и действие:
- что читать;
- что записывать;
- что обновлять;
- с какой частотой;
- кто проверяет результат.

Где проявляется: будущая архитектура TiTan Bots.

### Ошибка 3. Автономный агент сразу правит код

Симптом: агент получает широкую задачу и сам меняет код без строгого ограничения.

Причина: смешение анализа и исполнения.

Решение:
- сначала аудит;
- потом ТЗ;
- потом минимальная правка;
- потом проверка;
- человек утверждает результат.

Где проявляется: разработка торговых ботов, Codex-агенты, исправление багов.

### Ошибка 4. Повторяющийся контент делается вручную

Симптом: каждый раз заново объясняется стиль обложки, описания или поста.

Причина: нет skill с правилами стиля.

Решение: создать skills для повторяющихся форматов.

Где проявляется: YouTube, Telegram, презентации, посты, превью.

## Правила

### Правило 1. Минимальный достаточный инструмент

Если задачу можно решить обычным чатом, не использовать агента, MCP или CLI.

### Правило 2. Повторяющаяся задача -> skill

Если задача будет выполняться второй, третий или последующий раз, нужно рассмотреть создание skill.

### Правило 3. Внешняя система -> MCP

Если ИИ должен работать с Notion, Gmail, CRM, Google Sheets, API или аналитикой, нужен слой интеграции. В терминологии ролика — MCP.

### Правило 4. Открытая цепочка действий -> агент

Если задача требует нескольких шагов, проверки промежуточных результатов и доступа к инструментам, нужен агентный сценарий.

### Правило 5. Независимые части задачи -> субагенты

Если несколько частей задачи можно выполнять параллельно, можно использовать субагентов.

### Правило 6. Кодовый агент не должен работать без ревью
