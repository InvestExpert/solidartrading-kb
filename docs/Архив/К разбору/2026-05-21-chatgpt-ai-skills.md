## Краткое резюме

Видео объясняет концепцию AI-skills: это отдельные текстовые инструкции для AI-агента, которые описывают повторяемый рабочий процесс. Агент не обязан держать все инструкции в контексте постоянно. Он видит короткие описания skills и подгружает нужный skill только тогда, когда задача совпадает с его назначением.

Главная польза для TiTan Bots: не искать готовые маркетинговые skills, а создать собственную библиотеку процедур:
- аудит кода роботов;
- проверка логики MT5/Python;
- анализ результатов оптимизатора;
- подготовка отчетов;
- выжимки чатов по правилам wiki;
- создание базы знаний для AI/RAG-бота;
- подготовка материалов для инкубатора.

Главный вывод: первым нужно делать не маркетинговый skill, а skill аудита кода и логики TiTan Bots.

## Точные факты

### Определение skill

Skill в видео объясняется как текстовая инструкция для AI-агента.

Суть:
- когда пользователь дает определенную задачу, агент должен действовать по заданному процессу;
- skill описывает шаги, критерии, формат результата, примеры и шаблоны;
- это способ передать AI-агенту цифровую экспертизу человека.

Простая аналогия из видео:
- skill похож на рецепт;
- без рецепта результат может быть разным;
- с рецептом результат становится стабильнее.

### Отличие skill от промта и инструкции проекта

Обычный промт:
- вручную копируется в чат;
- работает только в конкретном запросе;
- не является отдельным модулем процесса.

Инструкция проекта:
- загружается в контекст целиком;
- может занимать много места;
- если инструкция большая, агент может терять или игнорировать часть середины.

Skill:
- существует как отдельный файл;
- агент видит короткое описание;
- полный текст skill подгружается только при необходимости;
- разные процессы можно разнести по отдельным skills.

### Структура skill-файла

По видео структура выглядит так:

- папка skill;
- основной файл: `skill.md`;
- в начале файла:
  - название;
  - описание;
  - когда использовать;
- дальше:
  - пошаговая инструкция;
  - критерии проверки;
  - формат результата;
  - примеры;
  - шаблоны;
  - вспомогательные материалы.

Возможные дополнительные файлы рядом с `skill.md`:
- шаблоны;
- примеры;
- справочные документы;
- образцы результата.

### Способы вызова skill

В видео названы три способа:

1. Автоматический выбор агентом.
   - Пользователь дает задачу.
   - Агент сам определяет подходящий skill по описанию.

2. Прямая просьба словами.
   - Пример: "Используй skill для написания отчетов".

3. Специальная команда.
   - В видео упомянуты slash-команды и специальные команды в инструментах.
   - Точный синтаксис зависит от платформы.

### Способы создания skill

В видео названы два подхода.

#### Подход 1. Skill creator

Суть:
- пользователь описывает агенту, какой skill нужен;
- агент создает черновик `skill.md`;
- пользователь дорабатывает критерии, нюансы и примеры.

Плюс:
- быстрый старт.

Минус:
- первая версия часто средняя;
- без опыта пользователя skill будет поверхностным.

#### Подход 2. Реальная задача → skill

Суть:
1. Взять реальную задачу.
2. Решить ее в чате с агентом шаг за шагом.
3. Исправлять результат до нужного качества.
4. После завершения попросить агента:
   - описать процесс;
   - выделить шаги;
   - выделить критерии;
   - зафиксировать, что пользователю не нравилось;
   - превратить процесс в skill.

Плюс:
- skill строится на реальном рабочем процессе, а не на абстрактной инструкции.

Минус:
- дольше по времени;
- итоговый skill все равно нужно проверить вручную.

## Идеи

### Идея 1. Библиотека внутренних skills TiTan Bots

Создать внутреннюю библиотеку skills, которые фиксируют повторяемые рабочие процессы проекта.

Ключевые направления:
- разработка;
- аудит;
- анализ тестов;
- сравнение платформ;
- подготовка wiki;
- подготовка отчетов;
- продвижение;
- клиентские объяснения.

Статус: новая идея, полезна для слоя [TiTan Bots - Инкубатор - Индекс](../../../TiTan%20Bots%20-%20%D0%98%D0%BD%D0%BA%D1%83%D0%B1%D0%B0%D1%82%D0%BE%D1%80%20-%20%D0%98%D0%BD%D0%B4%D0%B5%D0%BA%D1%81.md).

### Идея 2. Skill аудита кода роботов

Создать отдельный skill, который будет применять правила строгого аудита кода:

- сначала ошибки, риски и слабые места;
- потом минимальные правки;
- не удалять существующие комментарии;
- новые комментарии писать на языке исходного кода;
- проверять всю цепочку:
  - входы;
  - выходы;
  - вызовы;
  - глобальные флаги;
  - recovery;
  - совместимость с соседними функциями;
- не предлагать костыли, пока не проверены существующие механизмы;
- прямо указывать риск поломки.

Статус: приоритет 1.

### Идея 3. Skill сравнения MT5 и Python

Создать skill для анализа расхождений между MT5 и Python.

Сравнивать не только строки логов, а поведенческий паттерн:
- число сделок;
- BUY/SELL;
- forced;
- средний profit ordinary SELL;
- средний forced loss;
- winrate;
- глубины;
- live_buy_count;
- повторяющиеся последовательности событий.

Статус: приоритет 2.

### Идея 4. Skill анализа оптимизатора

Создать skill, который принимает результаты MT5-оптимизатора и выделяет:
- плато;
- локальные пики;
- худшие конфигурации;
- DD;
- profit;
- worst-case;
- shortlist;
- универсальные параметры;
- параметры, которые нельзя считать универсальными.

Статус: приоритет 3.

### Идея 5. Skill выжимки чатов для Obsidian

Создать skill, который превращает длинные чаты в структуру для Markdown-wiki TiTan Bots.

Skill должен:
- определять слой wiki;
- отсеивать личное и нерелевантное;
- сохранять больше деталей, а не меньше;
- явно выписывать точные факты, параметры, формулы, гипотезы, выводы;
- готовить файл для Obsidian.

Статус: приоритет 4.

### Идея 6. Security audit сторонних skills

Создать отдельную проверку перед установкой сторонних skills.

Проверять:
- скрытые инструкции;
- попытки доступа к секретам;
- попытки отправить данные наружу;
- команды на чтение файлов;
- команды на поиск ключей, токенов, паролей;
- команды, которые могут изменить проект без разрешения.

Статус: важное правило безопасности.

### Идея 7. Документация → skill

Использовать подход:
- загрузить документацию платформы, API или инструмента;
- попросить агента сделать рабочий skill;
- использовать его для задач по этой платформе.

Применимо к:
- MT5;
- MQL5;
- Python;
- Backtrader;
- CCXT;
- Telegram bot platforms;
- RAG-сервисам;
- Obsidian workflow.

Статус: полезная идея, но требует проверки документации.

### Идея 8. AI/RAG бот по проекту

Использовать skills как часть подготовки AI-бота по базе проекта.

Задача бота:
- отвечать по материалам TiTan Bots;
- объяснять метод;
- находить нужные презентации, инструкции, FAQ;
- поддерживать партнерские ссылки;
- работать без ручного меню, через вопрос обычным языком.

Статус: связано с направлением Telegram-бота и базы знаний.

## Гипотезы

## Ошибки и решения

### Ошибка 1. Использовать skill как длинный промт без структуры

Симптом:
- агент получает большую инструкцию и теряет часть важных требований.

Причина:
- инструкция не разбита на отдельные skills;
- нет четких условий применения;
- слишком много общего текста.

Решение:
- разбить процессы на отдельные skills;
- писать короткое описание, когда skill нужен;
- внутри skill задавать четкие шаги и критерии.

Где проявляется:
- ChatGPT/Codex/AI-агенты;
- разработка и wiki-процессы.

### Ошибка 2. Создавать skill без реального процесса

Симптом:
- skill получается общим и слабым.

Причина:
- пользователь не описал точные действия, критерии и примеры.

Решение:
- сначала решить реальную задачу;
- затем превратить ход работы в skill;
- вручную проверить и доработать.

Где проявляется:
- любые skills.

### Ошибка 3. Ставить сторонние skills без проверки

Симптом:
- агент может получить вредные инструкции.

Причина:
- skill из стороннего источника может содержать команды на сбор файлов, ключей, секретов или передачу данных.

Решение:
- сначала читать skill;
- проверять через отдельный AI-аудит;
- не устанавливать до проверки;
- особенно проверять команды доступа к файлам, секретам, токенам и внешней отправке данных.

Где проявляется:
- установка skills из GitHub, маркетплейсов и сторонних коллекций.

## Правила

1. Skill должен описывать конкретный повторяемый процесс, а не общую роль агента.

2. В начале skill нужно писать:
   - название;
   - назначение;
   - когда использовать;
   - когда не использовать.

3. Для TiTan Bots skill должен содержать:
   - порядок действий;
   - критерии проверки;
   - формат результата;
   - список рисков;
   - запреты;
   - примеры корректного результата.

4. Для разработки ботов:
   - сначала аудит;
   - потом минимальный патч;
   - проверять цепочку входов, выходов, вызовов, флагов, recovery и соседних функций;
   - не удалять существующие комментарии;
   - новые комментарии писать на языке исходного кода;
   - не предлагать костыли до проверки существующих механизмов.

5. Для анализа тестов:
   - не выбирать одиночный лучший пик;
   - искать плато;
   - учитывать DD;
   - учитывать worst-case;
   - отделять универсальные параметры от индивидуальных.

6. Для wiki:
   - сохранять больше деталей, а не меньше;
   - точные данные не заменять общими фразами;
   - если упомянут список, формула или решение — выписать сам список, формулу или решение;
   - личное и нерелевантное пропускать.

7. Для продвижения:
   - не превращать сильную формулировку в обещание гарантированной прибыли;
   - отделять факт от маркетинговой интерпретации;
   - указывать риски и ограничения.

8. Для сторонних skills:
   - не устанавливать до проверки;
   - проверять на вредные команды;
   - не доверять только репутации источника.

Ограничение:

- skill нужно проверять;
- плохой skill закрепит плохой процесс.
