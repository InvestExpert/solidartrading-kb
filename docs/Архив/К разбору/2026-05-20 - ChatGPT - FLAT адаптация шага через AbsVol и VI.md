## Краткое резюме

В чате разобрана попытка создать умный режим адаптации шага сетки для FLAT-бота на MT5.

Главная инженерная идея: разделить адаптацию на две ступени:

1. Первая ступень меняет базовый шаг сетки по абсолютной волатильности рынка.
2. Вторая ступень мягко корректирует уже изменённый шаг по относительной волатильности VI и параметрам буфера/трейлинга.

Ключевой вывод по апрелю: просадка появилась не потому, что вторая ступень плохо реагировала, а потому что первая ступень не поднимала базовый шаг до ожидаемого диапазона около `0.33`. В логах фактический шаг был около `0.24–0.32`, хотя по старым тестам апрель лучше работал около `0.33`.

## Идеи

- Двухступенчатый smart-фильтр:
  - `AbsVol` задаёт глобальный базовый шаг.
  - `VI` делает мягкую коррекцию вокруг этого шага.
- `AbsVol = ATR(30) / ATR(365)` на D1 как показатель месячной абсолютной волатильности.
- `VI = ATR(100) / ATR(1000)` как локальный относительный индекс волатильности на рабочем таймфрейме.
- Вынести коэффициент адаптации шага в параметры, чтобы разные активы могли иметь разные профили.
- Логировать не только итоговый шаг, но и промежуточные значения:
  - `AbsVol`
  - `AbsVolCoef`
  - `step`
  - `step_coef`
  - восстановленный базовый шаг до второй ступени
  - `forcedBufferPerc_smart`
  - `trailingBufferPerc_smart`
  - `eventV`
  - `pnl`
  - `success`

## Гипотезы

- [HYPOTHESIS] Абсолютная волатильность месяца может задавать глобальный оптимальный шаг сетки.
  - Для XAUUSD низкая `AbsVol` около `0.9–1.0` связана с шагом около `0.23`.
  - Высокая `AbsVol` около `1.8–2.0` должна приводить шаг к зоне `0.33`.
  - Проверка: прогнать месяцы с логированием фактического `step_base = step / step_coef`.

- [HYPOTHESIS] Если первая ступень правильно выставляет глобальный шаг, второй фильтр не должен сильно уменьшать шаг.
  - Иначе фильтры конфликтуют: первый расширяет сетку, второй сужает её обратно.
  - Проверка: сравнить базовый шаг после `AbsVol` и итоговый шаг после `VI`.

- [HYPOTHESIS] Старый VI-фильтр может работать лучше как вторая ступень, если первая ступень уже правильно задаёт базовый шаг.
  - Старый фильтр не уменьшал шаг при низкой VI, а только расширял шаг и усиливал буфер/трейлинг при высокой VI.
  - Проверка: прогон января и апреля с одинаковой первой ступенью и старым вторым фильтром.

- [HYPOTHESIS] Формула `step = baseStep * AbsVol^p` с `p=0.52` недостаточно поднимает шаг в апреле.
  - В логах апрельский `AbsVol≈1.835` давал шаг около `0.32`, а не стабильный `0.33`.
  - Проверка: оптимизация по `p`, а также тест альтернативных формул с pivot.

## Выводы

- Нельзя оценивать smart-фильтр только по итоговому PnL. Нужно смотреть фактическую цепочку:
  - какой `AbsVol`;
  - какой базовый шаг после первой ступени;
  - какой `step_coef`;
  - какой итоговый шаг;
  - какой тип завершения `V1–V6`;
  - какой PnL.
- В апреле просадка была связана с тем, что первая ступень не довела шаг до зоны старого успешного режима.
- Если старый режим с базовым `0.33` и старым smart-фильтром давал плюс, а новый режим с `AbsVol` даёт минус, сначала проверять первую ступень, а не подкручивать вторую.
- Оптимизация по шагам с включённым smart-режимом показала, что зона около `0.27–0.30` может быть лучше, чем чрезмерно широкий шаг `0.33–0.35` в конкретной конфигурации. Это значит, что формула адаптации должна проверяться против оптимизационного профиля, а не против одной красивой идеи.
- Формула и квантизация могут искажать результат:
  - округление к `0.01`;
  - возможное двойное квантование;
  - скрытые cap/limit;
  - `floor` вместо округления к ближайшему.

## Ошибки и решения

### Ошибка: попытка лечить апрель второй ступенью

Сначала предполагалось, что апрель просел из-за неправильной реакции VI-фильтра. После проверки логов стало видно, что базовый шаг после первой ступени не доходил до нужного диапазона.

Решение:

- строить график `step_base` по сессиям;
- сравнивать `step_base` и `step_final`;
- проверять, не режет ли шаг квантизация или cap;
- не изменять вторую ступень, пока первая ступень не даёт правильный диапазон.

### Ошибка: формула без якоря

Формула:

```cpp
adjustedBaseStep = baseStep * MathPow(absVol, absVolCoef);
```

Проблема:

- при `AbsVol < 1` шаг уменьшается ниже базового;
- при `AbsVol≈1.8` и `absVolCoef≈0.52` шаг может не дойти до `0.33`;
- результат зависит от того, что считается базовой точкой.

Возможные решения:

- использовать pivot-формулу;
- использовать tanh-демпфер;
- использовать magnet-формулу;
- явно проверять таблицу `AbsVol → step`.

### Ошибка: отсутствие отдельного логирования базового шага

В логах был `step` и `step_coef`, но не было явного `step_base`.

Решение:

- добавить отдельное поле `step_base_before_vi`;
- либо вычислять его при анализе как `step / step_coef`.

### Ошибка: слишком активное изменение второй ступени

Когда первая ступень уже меняет базовый шаг, второй фильтр должен быть мягким.

Решение:

- не использовать жёсткие отрицательные дельты во второй ступени без отдельного теста;
- старый фильтр “только вверх” может быть более стабильной второй ступенью;
- сначала проверить старую вторую ступень поверх новой первой ступени.

## Правила

- Сначала проверять первую ступень адаптации шага, потом вторую.
- Не править буфер и трейлинг, пока не понятно, какой фактический шаг работает.
- Любая формула smart-step должна проверяться таблицей:
  - месяц;
  - `AbsVol`;
  - базовый шаг после первой ступени;
  - итоговый шаг после второй ступени;
  - PnL;
  - V1/V2/V3/V4/V5/V6.
- Если апрель должен работать как старый режим `0.33`, то фактический базовый шаг в апреле должен быть около `0.33`, а не `0.25–0.30`.
- Если оптимизация показывает лучший результат на `0.28–0.30`, не тянуть формулу искусственно к `0.33`.
- Формулы нельзя считать рабочими по одному месяцу. Нужны минимум:
  - январь;
  - февраль;
  - март;
  - апрель;
  - май;
  - июнь;
  - июль;
  - август.
- Проверять не только PnL, но и структуру исходов:
  - V1 как главный источник убытка;
  - V6 как желательный тейк/выход;
  - V5 как форс-мажор;
  - V2–V4 как защитные/частичные сценарии.

## Архитектура

### RegimeSnap

Используется снимок режима на старте сессии:

```cpp
struct RegimeSnap {
   datetime ts;
   double atr100;
   double atr1000;
   double vi;
   double adx50;
   double absVol;
   double absVolCoef;
   double gridStepPerc_smart;
   double gridStepPerc_coef;
   double forcedBufferPerc_smart;
   double trailingBufferPerc_smart;
   double forcedBufferPerc_coef;
   double trailingBufferPerc_coef;
};
```

Назначение:

- зафиксировать рыночный режим на старте сетки;
- не менять параметры внутри сессии;
- анализировать результат после завершения.

### Две ступени smart-фильтра

Первая ступень:

```cpp
adjustedBaseStep = baseStep * MathPow(absVol, absVolCoef);
```

или альтернативная формула.

Вторая ступень:

- использует VI;
- меняет `coef`, `k_coef`, `m_coef`;
- должна быть мягкой, если первая ступень уже активно меняет шаг.

```text

```

Рекомендуемое дополнительное поле:

```text
step_base_before_vi
```

Иначе приходится восстанавливать его как:

```text
step_base = step / step_coef
```

### Квантование шага

Используется квант `0.01`.

Риск:

- двойное квантование;
- преждевременное округление;
- искажение оптимального шага;
- превращение `0.329` в `0.32`, если используется `floor`.

Рекомендация:

- квантовать один раз;
- логировать значение до и после квантования;
- использовать округление к ближайшему, если нет причины делать иначе.

## Что проверить дальше

- Построить таблицу `AbsVol → step_base → step_final` для всех месяцев.
- Отдельно проверить, как формула ведёт себя на:
  - январь;
  - февраль;
  - март;
  - апрель;
  - май;
  - июнь;
  - июль;
  - август.
- Проверить, есть ли cap на шаг выше `0.32`.
- Проверить, не квантуется ли шаг дважды.
- Проверить разные значения `SmartAbsVolCoef`.
- Проверить pivot-формулу:

```cpp
gridStepPerc_smart = gridStepPerc * MathPow(absVol / SmartPivotValue, SmartAbsVolCoef);
```

- Проверить tanh-формулу:

```cpp
gridStepPerc_smart = gridStepPerc * MathExp(absVolCoef * MathTanh(MathLog(absVol / SmartPivotValue)));
```

- Проверить magnet-формулу:

```cpp
gridStepPerc_smart = gridStepPerc + absVolCoef * (MathLog(absVol / SmartPivotValue) / (1.0 + MathAbs(MathLog(absVol / SmartPivotValue))));
```

- Сравнить эти формулы не по одному лучшему месяцу, а по устойчивости плато.
- Проверить, нужен ли ADX как второй рычаг для месяцев, где волатильность не объясняет оптимальный шаг.

### Расчёт AbsVol

```cpp
double CalculateAbsVol()
{
    int available_bars = Bars(_Symbol, PERIOD_D1);
    int short_period = MathMin(30, available_bars / 2);
    int long_period = MathMin(365, available_bars - 10);

    if(short_period < 10 || long_period < 30) return 0.0;

    int atr_short_handle = iATR(_Symbol, PERIOD_D1, short_period);
    int atr_long_handle = iATR(_Symbol, PERIOD_D1, long_period);

    double atr_short_buffer[1];
    double atr_long_buffer[1];

    int short_result = CopyBuffer(atr_short_handle, 0, 1, 1, atr_short_buffer);
    int long_result  = CopyBuffer(atr_long_handle,  0, 1, 1, atr_long_buffer);

    IndicatorRelease(atr_short_handle);
    IndicatorRelease(atr_long_handle);

    if(short_result <= 0 || long_result <= 0) return 0.0;

    return (atr_long_buffer[0] > 0.0) ? atr_short_buffer[0] / atr_long_buffer[0] : 0.0;
}
```
