## Краткое резюме

В чате была выведена и уточнена формула для расчёта рекомендуемого буфера защиты FLAT-сетки в процентах от шага сетки.

Задача: после закрытия стартовых ордеров одной стороны по стопу определить, на сколько должна пройти цена дальше, чтобы закрытие защитных ордеров противоположной стороны вывело цикл в ноль.

Ключевой вывод: если расчёт ведётся в шагах сетки, абсолютный размер шага `S` не нужен. Он сокращается из баланса. Поэтому итоговый `recommendedBufferPercent` можно считать напрямую как процент от одного шага.

## Идеи

- Автоматически рассчитывать рекомендуемый буфер защиты вместо ручного подбора.
- Считать буфер через баланс:
  - убыток стопнутых стартовых ордеров;
  - прибыль защитных ордеров;
  - уже зафиксированную раннюю прибыль.
- Использовать расчётный буфер как диагностическую величину в `OnInit()` и логах тестера.

## Гипотезы

- Ранние фиксации по одному ордеру на каждом шаге уменьшают нужный защитный буфер.
- При равных лотах и линейной сетке буфер в процентах от шага не зависит от абсолютного значения `gridStep`.
- Формула может быть использована как инженерная оценка минимального безубыточного буфера, но требует проверки на реальной механике `partialCloseOrders_Early` и `partialCloseOrders_Stop`.

## Выводы

### Базовая формула без ранних фиксаций

Обозначения:

- `u` — количество стартовых ордеров, закрытых по стопу.
- `N` — количество защитных ордеров, закрываемых после стопа.
- `x` — дистанция от старта сетки до стопа.
- `y` — дополнительная дистанция после стопа до закрытия защиты.

Баланс:

```text
u * x = N * (x + y)
```

Отсюда:

```text
y = (u / N - 1) * x
```

Если `x = d * S`, где:

- `d` — количество шагов до стопа;
- `S` — абсолютный шаг сетки,

то буфер в процентах от шага:

```text
bufferPercent = 100 * (u / N - 1) * d
```

### Формула с ранними фиксациями

Допущение: пока цена шла к границе, фиксировался один защитный ордер на каждом шаге.

Ранняя прибыль в шагах:

```text
1 + 2 + ... + d = d * (d + 1) / 2
```

Итоговая формула:

```text
bufferPercent =
100 * (
    ((u / N) - 1) * d
    - d * (d + 1) / (2 * N)
)
```

Где:

- `u = upperOrders`
- `d = lowerOrders`
- `N = remainingOrders`
- `remainingOrders = totalAvailable + reserveOrders`

При `partialClosePercent = 0`:

```text
totalAvailable = upperOrders - lowerOrders - reserveOrders
remainingOrders = totalAvailable + reserveOrders
remainingOrders = upperOrders - lowerOrders
```

То есть защита равна:

```text
N = upperOrders - lowerOrders
```

## Ошибки и решения

### Ошибка: старая формула буфера была слишком эмпирической

Старая формула:

```cpp
bufferCoeff = (
    2.0 * lowerOrders
    + (double)lowerOrders * (lowerOrders - 1) / 2.0
    + (lowerOrders + 2.0) * 0.1
) / (double)remainingOrders;
```

Проблема:

- содержит магические коэффициенты;
- не выражает напрямую баланс убытка и прибыли;
- неочевидно, как связана с `partialCloseOrders_Stop`;
- плохо объясняется как инженерное правило.

Решение:

- заменить её формулой баланса;
- учитывать ранние фиксации отдельным вычитаемым членом;
- результат считать сразу как процент от шага.

## Правила

- Считать не абсолютные цены, а дистанции от старта сетки.
- Если нужен процент от шага, абсолютный шаг `S` в формуле не участвует.
- `bufferCoeff` должен хранить буфер в долях шага.
- `recommendedBufferPercent = 100.0 * bufferCoeff`.
- Если расчётный буфер меньше нуля, клампить в ноль.
- Если `remainingOrders <= 0`, расчёт невозможен, буфер должен остаться нулевым или требовать отдельной обработки.
- Для `partialClosePercent > 0` формулу нужно отдельно проверить, потому что число ранних и защитных закрытий меняется.

## Архитектура

Расчёт выполняется в `OnInit()` после вычисления:

```cpp
totalAvailable = upperOrders - lowerOrders - reserveOrders;
partialCloseOrders_Early;
partialCloseOrders_Stop;
remainingOrders = totalAvailable + reserveOrders;
```

Рекомендуемый буфер логируется как диагностический параметр:

```cpp
recommendedBufferPercent
```

Он не обязан автоматически заменять `forcedBufferPerc`, но может использоваться:

- для сравнения с ручным параметром;
- для логов оптимизации;
- для инженерной проверки конфигурации;
- как ориентир минимального буфера безубыточности.

## Итоговый кодовый фрагмент

Фрагмент замены расчёта `bufferCoeff`:

```cpp
double bufferCoeff = 0.0;
if(remainingOrders > 0)
{
    bufferCoeff =
        ( ((double)upperOrders / (double)remainingOrders - 1.0) * (double)lowerOrders )
      - ( ((double)lowerOrders * ((double)lowerOrders + 1.0) * 0.5) / (double)remainingOrders );

    if(bufferCoeff < 0.0) bufferCoeff = 0.0;
}

recommendedBufferPercent = 100.0 * bufferCoeff;
```

## Что проверить дальше

- Проверить формулу на реальном `partialClosePercent = 0`.
- Отдельно проверить `partialClosePercent > 0`.
- Сверить, действительно ли `remainingOrders` равно числу защитных ордеров, которые закрываются в `TryPartial`.
- Проверить влияние комиссий, спреда и проскальзывания.
- Проверить, не нужно ли использовать `partialCloseOrders_Stop` напрямую вместо `remainingOrders` в сценариях с разделением Early/Stop.
- Сравнить расчётный `recommendedBufferPercent` с оптимизационными результатами.
