## Краткое резюме

В чате разобрано видео про Codex как AI-агента для разработки. Главная польза для TiTan Bots — не установка Codex сама по себе, а перенос агентного workflow в разработку торговых ботов: правила проекта, планирование перед изменениями, Git-чекпоинты, отдельное ревью, контроль контекста, скилы, субагенты и ограничение опасных разрешений.

Материал лучше сохранить в слой **TiTan Bots - Инкубатор**, потому что это новая рабочая гипотеза по организации AI-разработки, которую нужно протестировать на реальных задачах ATLAS / FLAT / IMPULS перед переносом в основной технический слой.

## Точные факты

- В видео Codex объясняется как продукт OpenAI, где модель работает не только как чат, а как агент, способный выполнять команды, читать файлы, редактировать проект и запускать процессы.
- Работа агента всегда происходит в конкретной папке проекта.
- Для Codex описаны варианты использования:
  - десктопное приложение;
  - расширение для IDE, например VS Code / Cursor;
  - CLI-инструмент в терминале.
- Plan Mode включается через `Shift + Tab`.
- Plan Mode нужен для планирования разработки до внесения изменений.
- В Plan Mode агент работает в режиме чтения и планирования, без прямого редактирования файлов.
- В видео указано, что перед разработкой фичи нужно сначала спланировать:
  - что будет сделано;
  - где будут внесены изменения;
  - какие решения модель не может принять сама;
  - какие вопросы надо уточнить.
- В видео указано, что в Codex на момент записи нет полноценного механизма чекпоинтов или он не гарантирован, поэтому рекомендуется использовать Git.
- Git предлагается как способ вернуться к рабочему состоянию, если агент ушёл не туда или что-то сломал.
- Команда `/review` используется для проверки кода, написанного нейросетью.
- Для `/review` желательно включать высокий уровень размышлений (`high` или `extra high`), чтобы модель тщательнее искала ошибки.
- Команды `/new` и `/clear` очищают контекст, но различаются поведением:
  - `/new` очищает контекст, но оставляет видимой предыдущую переписку;
  - `/clear` очищает терминал визуально и начинает с чистого состояния.
- Команда `/status` показывает информацию о модели, директории, разрешениях, лимитах и состоянии аккаунта.
- Status line можно настроить так, чтобы видеть модель, директорию, лимиты и заполнение контекста.
- Контекст модели нельзя бесконтрольно заполнять до 100%, потому что растёт риск галлюцинаций и потери качества.
- Автоматический compact сжимает старый контекст в summary и вставляет его в новый контекст.
- До автоматического compact лучше не доводить, если задача критичная.
- `AGENTS.md` — Markdown-файл с правилами проекта, который добавляется в контекст агента в начале сессии.
- В `AGENTS.md` можно прописывать:
  - архитектуру проекта;
  - соглашения по именованию;
  - используемые библиотеки;
  - запрещённые действия;
  - команды сборки;
  - команды тестирования;
  - карту проекта;
  - правила коммитов и pull request.
- Для `AGENTS.md` в видео названа рекомендация: держать файл актуальным и не раздувать; если он становится слишком большим, разделять правила на отдельные файлы.
- В видео упомянут лимит-ориентир: примерно до 200 строк для основного файла правил, после чего лучше дробить на отдельные инструкции.
- Возможны два подхода к правилам:
  - централизованный: общий `AGENTS.md` и ссылки/пути на отдельные файлы правил;
  - распределённый: `AGENTS.md` рядом с кодом в отдельных папках.
- Гибридный подход: общие правила держать централизованно, специфичные правила класть рядом с модулем.
- Локальные `AGENTS.md` могут работать только для конкретной папки или части проекта.
- `agents.override.md` может использоваться для временного или приоритетного переопределения правил.
- MCP — протокол подключения внешних инструментов и сервисов к агенту.
- Примеры MCP из видео:
  - браузер;
  - база данных;
  - Figma;
  - Linear;
  - shadcn/ui компоненты;
  - другие внешние сервисы.
- Skills — навыки агента, оформленные как Markdown-файлы.
- Skill может быть глобальным или локальным для проекта.
- Для локальных skills в видео показан путь: `.agents/skills/...`.
- Skill хранится в файле `skill.md`.
- Субагенты — отдельные специализированные агенты под разные роли.
- В видео предложена схема:
  - главный агент-оркестратор принимает задачу, анализирует её, декомпозирует и распределяет;
  - мелкие субагенты выполняют конкретные задачи;
  - code reviewer как субагент должен иметь более сильную модель и высокий reasoning.
- Примеры ролей субагентов:
  - дизайнер;
  - фронтендер;
  - бэкендер;
  - проектировщик базы данных;
  - тестировщик;
  - code reviewer.
- Hooks — механизм запуска скриптов при событиях в Codex/терминале.
- Примеры hooks из видео:
  - уведомление в Telegram, когда агенту нужно подтверждение;
  - уведомление после завершения работы;
  - проигрывание звука после завершения работы агента.
- Plugins — упакованные наборы MCP, skills, субагентов и hooks.
- Пример плагина из видео: Super Powers.
- Super Powers описан как набор скилов для полного цикла разработки фичи, мозгового штурма, планирования и подхода через тесты.

### Минимальный workflow для TiTan Bots

```text
git status
git checkout -b fix/<task>
Codex Plan Mode:
  - что менять
  - где менять
  - что не трогать
  - как проверить
Реализация
/review
Запуск теста / сравнение логов
git diff
commit
```

### Роли субагентов для TiTan Bots

- Оркестратор:
  - принимает задачу;
  - разбивает на подзадачи;
  - следит за ограничениями проекта;
  - не пишет весь код сам, если задача сложная.
- Разработчик:
  - вносит минимальные изменения;
  - не меняет торговую логику без отдельного разрешения;
  - сохраняет существующие комментарии.
- Code reviewer:
  - проверяет цепочку входов, выходов, вызовов, флагов, recovery и совместимость с соседними функциями;
  - должен использовать более высокий уровень размышлений.
- Тестировщик:
  - запускает тесты;
  - проверяет, что изменения не ломают ожидаемый сценарий;
  - фиксирует, какие проверки реально выполнены.
- Аналитик логов:
  - сравнивает MT5/Python поведение;
  - ищет не построчный паритет, а паттерны поведения, если это применимо к задаче.

## Идеи

### Идея 1. Создать `AGENTS.md` для TiTan Bots

Суть: сделать единый файл правил для AI-агента, который работает с кодом TiTan Bots.

Что включить:
- минимальные правки;
- запрет менять торговую логику без явного запроса;
- запрет удалять существующие комментарии;
- новые комментарии писать на языке исходного кода;
- сначала аудит, потом код;
- проверять всю цепочку влияния;
- не предлагать костыли, пока не проверены существующие механизмы;
- перед правкой указывать риск поломки;
- после правки запускать `/review` и тесты;
- для MT5/Python учитывать цель паритета;
- для торговых ботов фиксировать, какие файлы и функции изменены.

Статус: идея для Инкубатора.

### Идея 2. Сделать отдельные локальные правила по папкам

Суть: для крупных проектов TiTan Bots не держать все правила в одном месте.

Возможное разделение:
- `mt5/AGENTS.md` — правила для MQL5 и MT5-логики;
- `python/AGENTS.md` — правила для Python/CCXT/DataProvider;
- `tests/AGENTS.md` — правила для тестов и сравнения результатов;
- `docs/AGENTS.md` — правила для документации;
- `logs/AGENTS.md` — правила анализа логов.

Статус: идея для проверки.

### Идея 3. Создать skills для повторяющихся задач

Возможные skills:
- `mt5-python-parity-audit` — проверка совпадения MT5 и Python логики;
- `grid-bot-risk-audit` — аудит forced sell, RiskGuard, recovery, state rebuild;
- `optimizer-result-analyzer` — анализ результатов MT5 optimizer;
- `trading-logic-reviewer` — проверка торговой логики без вольного рефакторинга;
- `wiki-extract-for-titan-bots` — перенос выводов из чата в Obsidian wiki.

Статус: идея для Инкубатора.

### Идея 4. Использовать субагентов как роли аудита

Суть: не давать одному агенту одновременно писать код, проверять себя и делать выводы.

Схема:
- основной агент готовит план;
- разработчик вносит минимальную правку;
- reviewer проверяет системную совместимость;
- тестировщик проверяет тесты и diff;
- аналитик логов сравнивает поведение.

Статус: идея для будущей архитектуры AI-разработки.

### Идея 5. Hooks для уведомлений и проверок

Суть: использовать хуки не для сложной автоматизации, а для простых событий.

Возможные hooks:
- уведомление после завершения задачи;
- уведомление, если агент ждёт подтверждения;
- автозапуск линтера/теста после изменения файла;
- звуковой сигнал после долгой задачи.

Статус: низкий приоритет.

### Идея 6. MCP подключать только после стабилизации базового workflow

Суть: MCP полезен, но для TiTan Bots сначала важнее базовые правила, Git, план и ревью.

Возможные MCP позже:
- GitHub;
- документация библиотек;
- браузер для актуальных источников;
- локальная база документации;
- Obsidian или файловая wiki, если доступ безопасен.

Статус: отложить.

## Гипотезы

### Гипотеза 1. Максимальная польза будет от базового workflow, а не от MCP

Суть гипотезы: для TiTan Bots главный прирост качества даст связка `AGENTS.md + Plan Mode + Git + /review`, а не продвинутые MCP, плагины или параллельные агенты.

К чему относится: общий процесс разработки TiTan Bots.

Вероятность по оценке из чата: 85%.

Как проверить:
- взять одну небольшую правку в ATLAS или другом боте;
- провести старым способом;
- провести новым workflow;
- сравнить количество лишних изменений, ошибок, время проверки, качество diff.

Статус: не проверена.

### Гипотеза 2. Субагент code reviewer должен быть мощнее исполнителя

Суть гипотезы: агент-исполнитель может работать на среднем уровне размышлений, а reviewer должен использовать более сильную модель / высокий reasoning, потому что он проверяет системные ошибки.

К чему относится: архитектура AI-разработки TiTan Bots.

Как проверить:
- дать одинаковый diff двум reviewer-режимам: medium и high/extra-high;
- сравнить, кто найдёт больше реальных ошибок без ложного шума.

Статус: не проверена.

### Гипотеза 3. Локальные правила снизят риск ошибок в больших проектах

Суть гипотезы: правила рядом с кодом лучше удерживают контекст конкретного модуля, чем один большой общий файл.

К чему относится: ATLAS / FLAT / IMPULS и другие проекты с разными языками, тестами и логикой.

Как проверить:
- создать локальные правила для одной папки;
- дать агенту задачу в этой папке;
- проверить, соблюдает ли он ограничения лучше, чем с одним глобальным файлом.

Статус: не проверена.

## Выводы

- Codex полезен для TiTan Bots не как генератор кода, а как управляемый агент внутри проекта.
- Без правил проекта агент опасен: он может менять больше, чем нужно, и ломать скрытые зависимости.
- Для торговых ботов нельзя переносить подход из учебных лендингов напрямую. Нужны ограничения, ревью и тесты.
- Перед кодом должен быть план. Для TiTan Bots это критично, потому что правка одной функции может повлиять на state, recovery, forced sell, RiskGuard, пересчёт сетки и паритет MT5/Python.
- Git должен быть обязательным чекпоинтом перед каждой AI-сессией.
- `/review` полезен, но не заменяет ручной аудит и тестирование.
- Full Access удобен, но для торговых ботов рискован. Его нельзя включать по умолчанию в рабочем проекте.
- Skills и субагенты полезны, но их стоит внедрять после базового workflow.
- MCP и плагины не являются первым приоритетом для TiTan Bots.
- Основной слой для этой выжимки — Инкубатор, потому что это новая организация разработки, а не уже проверенное техническое правило проекта.

## Ошибки и решения

### Ошибка 1. Давать AI-агенту задачу без правил проекта

Симптом:
- агент действует как новый сотрудник без знания архитектуры;
- меняет не те файлы;
- делает лишний рефакторинг;
- игнорирует ограничения торговой логики.

Причина:
- пустой или плохо заполненный контекст.

Решение:
- создать и поддерживать `AGENTS.md`;
- добавить карту проекта, запреты, команды тестов, правила правок.

Где проявляется:
- любой проект TiTan Bots, особенно MT5/Python стратегии.

### Ошибка 2. Начинать сразу с реализации без Plan Mode

Симптом:
- агент быстро пишет код, но не фиксирует границы изменений;
- появляются побочные правки;
- сложнее проверить diff.

Причина:
- нет предварительной декомпозиции задачи.

Решение:
- сначала Plan Mode;
- реализация только после утверждения плана или после проверки плана пользователем.

Где проявляется:
- правки торговой логики, recovery, RiskGuard, forced sell, state rebuild.

### Ошибка 3. Использовать Full Access в рабочем проекте

Симптом:
- агент может выполнять команды и менять файлы без подтверждений.

Причина:
- режим полного доступа удобен, но снимает защитные барьеры.

Решение:
- использовать default permissions для рабочих проектов;
- Full Access только в изолированной копии или песочнице.

Где проявляется:
- код ботов, state-файлы, логи, репозитории, конфиги.

### Ошибка 4. Считать `/review` достаточной проверкой

Симптом:
- ревью может не заметить системный баг;
- агент проверяет не всю цепочку влияния.

Причина:
- LLM-review не является формальной верификацией.

Решение:
- `/review` использовать как дополнительный слой;
- затем проверять тестами, diff и логами.

Где проявляется:
- любые изменения в торговой логике.

### Ошибка 5. Доводить контекст до перегруза

Симптом:
- агент начинает забывать условия;
- даёт уверенные, но неточные ответы;
- теряет ограничения задачи.

Причина:
- слишком длинная сессия и переполненный контекст.

Решение:
- дробить задачи;
- делать summary/checkpoint;
- не полагаться на автоматический compact в критичных задачах.

Где проявляется:
- длинные аудиты больших файлов и перенос MT5 → Python.

## Правила

- Без плана агент код не меняет.
- Перед каждой AI-правкой нужен Git checkpoint или отдельная ветка.
- Агент должен менять только минимально необходимое.
- Агент не должен удалять существующие комментарии.
- Новые комментарии должны быть на языке исходного кода.
- Агент не должен менять торговую логику без явного запроса.
- Агент должен сначала указать, какие функции и файлы будут затронуты.
- Агент должен явно указать, что не будет трогать.
- После правки обязателен `git diff`.
- После правки обязателен `/review` или отдельный reviewer.
- После review обязателен тест или проверка логов, если изменение влияет на поведение стратегии.
- Full Access не включать по умолчанию для торговых ботов.
- MCP не подключать без явной необходимости.
- Skills должны быть короткими, проверяемыми и не должны закреплять спорные гипотезы как факты.
- `AGENTS.md` должен быть актуальным; устаревшие правила опаснее отсутствующих.
- Если `AGENTS.md` становится слишком большим, его нужно дробить на отдельные файлы правил.
- Глобальные правила — для всего проекта; локальные правила — только для конкретной папки или модуля.
- Временные запреты лучше выносить в override-файл и потом удалять.

## Контент, видео и презентации

Идеи контента:

### Видео / пост 1. «ИИ не пишет нам роботов вслепую»

4. Почему это важно для торговых ботов: маленькая ошибка может изменить поведение стратегии.
5. Ограничение: ИИ ускоряет работу, но не гарантирует прибыль и не заменяет проверку.

### Видео / пост 2. «Как мы не даём ИИ сломать торгового бота»

Структура:
1. Боль: AI может быстро написать код, но так же быстро сломать рабочую стратегию.
2. Решение: агент работает по правилам.
3. Процесс: `AGENTS.md → Plan Mode → правка → review → тест → git diff`.
4. Доказательство: показать пример diff до/после на безопасной задаче.
5. Ограничение: даже такой процесс не исключает ошибок, поэтому нужна финальная проверка человеком.

### Видео / пост 3. «Почему Full Access опасен для торговых ботов»

Структура:
1. Интрига: удобно дать ИИ полный доступ, но это не всегда разумно.
2. Риск: агент может менять файлы и выполнять команды без подтверждения.
3. Безопасный подход: песочница, Git, ограниченные разрешения.
4. Вывод: скорость не должна быть выше контроля.

## Возражения и ответы

### Возражение 1. «Если код пишет ИИ, значит это ненадёжно»

Короткий ответ:
- Ненадёжно, если ИИ пишет без контроля. Надёжнее, если он работает внутри строгого процесса.

Подробный ответ:
- ИИ используется как помощник, а не как автономный автор стратегии.
- Перед правкой фиксируется план.
- После правки проверяется diff.
- Отдельно запускается review.
- Поведение проверяется тестами и логами.

Что нельзя обещать:
- нельзя обещать отсутствие ошибок;
- нельзя обещать прибыль из-за использования ИИ.

### Возражение 2. «AI-агент может сломать стратегию»

Короткий ответ:
- Может. Поэтому ему нельзя давать бесконтрольный доступ к рабочему проекту.

Подробный ответ:
- Нужно использовать Git-чекпоинты, правила проекта, ограниченные разрешения и отдельную проверку.
- Full Access допустим только в изолированной копии.

Что нельзя обещать:
- нельзя обещать, что агент никогда не ошибётся.

### Возражение 3. «Зачем вообще AI, если всё равно нужна проверка?»

Короткий ответ:
- AI ускоряет рутинную работу, но финальное решение остаётся за человеком.

Подробный ответ:
- AI быстрее находит связанные места, готовит план, предлагает diff, проверяет код вторым проходом.
- Но торговая логика требует отдельной валидации.

Что нельзя обещать:
- нельзя обещать, что AI заменяет разработчика или аудитора.
