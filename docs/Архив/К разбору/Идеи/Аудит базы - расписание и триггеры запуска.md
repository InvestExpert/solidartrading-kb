# Аудит базы - расписание и триггеры запуска

## Lifecycle

Эта карточка живет в `К разбору/Идеи`.

Это не active/current знание и не ответ для RAG. Идея становится active wiki-карточкой только после проверки, решения владельца и отдельного переноса в нужный слой.

## Оценка ценности

7/10

## Идея

Определить, когда и как запускать audit-stack TiTan Bots:

- вручную перед важной задачей;
- после крупных wiki/import изменений;
- перед публикацией;
- по расписанию;
- через dry-run;
- через отдельный отчет/уведомление.

## Почему может быть ценно

- снизит риск забыть аудит после изменений;
- отделит обязательные проверки от редких глубоких pass;
- не даст запускать тяжелые слои без причины;
- поможет понять, нужен ли scheduler вообще.

## К чему относится

- FAST;
- DEEP;
- SOURCE-MARKER;
- RESEARCH;
- METHOD;
- META;
- публикация базы;
- ручные входы `Ошибки` / `Идеи`.

## Что проверить

- какие слои должны запускаться всегда;
- какие слои запускать только при изменениях в их зоне;
- нужен ли pre-publish check;
- нужен ли lightweight daily/weekly check;
- должен ли scheduler только сообщать о проблемах или что-то auto-fix;
- как не превратить расписание в шум.

## Как понять, что идея сработала

- есть простое правило запуска аудита;
- понятно, какие pass запускаются вручную, а какие автоматически;
- нет лишних очередей, панелей и шумных reports;
- audit-stack остается `PASS/0` после обычного рабочего цикла.

## Риски или сомнения

- scheduler может создать лишний шум;
- current/freshness проверки могут начать подменять RESEARCH;
- автоматический запуск не должен обрабатывать `Ошибки` / `Идеи` как active input;
- нельзя запускать тяжелые pass без practical residue.
