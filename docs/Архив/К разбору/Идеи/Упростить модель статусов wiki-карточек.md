# Упростить модель статусов wiki-карточек

## Оценка ценности

`8/10`

## Идея

Упростить модель статусов и route-полей wiki-карточек.

Текущая проблема: для простой карточки может появляться слишком много служебных YAML-полей: `status`, `usage_mode`, `public_faq_route`, `proof_role`, `public_proof`, `current_proof_status`, `current_proof_owner` и другие. Это усложняет работу человека, импорт, METHOD и RAG.

Цель: оставить минимальный набор полей, который реально управляет публикацией и маршрутом.

## Почему может быть ценно

- Меньше YAML-шума в карточках.
- Меньше ошибок при импорте и ручной правке.
- METHOD проще объяснять и проверять.
- RAG получает более понятную модель: публиковать / не публиковать / использовать как контекст.
- Новые карточки проще создавать по шаблону.

## К чему относится

- база TiTan Bots;
- METHOD audit;
- FAST/DEEP/RESEARCH взаимодействие;
- публикация GitHub / Telegram AI;
- шаблоны карточек;
- skill `titan-bots-wiki-import`.

## Что проверить

- Какие YAML-поля реально используются скриптами.
- Какие поля дублируют друг друга.
- Можно ли свести непубличные historical/context карточки к модели:

```yaml
status: draft
usage_mode: CONTEXT_ONLY
public_faq_route: false
```

- Какие поля должны остаться только как optional text/link, а не обязательный статус.
- Не сломает ли упрощение публикацию на GitHub и Telegram AI.

## Возможное решение

Ввести будущий отдельный audit-pass:

```text
STATUS-MODEL-OPTIMIZATION
```

Задача прохода:

1. собрать все используемые YAML-поля статусов и route-роли;
2. найти дублирующие поля;
3. предложить минимальную модель;
4. обновить шаблоны;
5. обновить METHOD/FAST правила;
6. мигрировать существующие карточки только после утверждения.

## Как понять, что идея сработала

- Для обычной карточки хватает 1-3 служебных полей.
- Исторические/context карточки не публикуются и не попадают в ошибки без длинного набора статусов.
- METHOD объясняет причину finding простыми словами.
- Пользователь может понять YAML без отдельной карты всех статусов.

## Риски или сомнения

- Можно слишком сильно упростить и потерять важные различия между proof, material, draft и internal.
- Нужно сначала проверить реальные зависимости скриптов и публикации.
- Нельзя механически удалять поля, пока не утверждена новая модель.

## Источник

- Обсуждение METHOD-route-quality и historical proof/material карточек.

## Связанные материалы

- `Оптимизатор базы TiTan Bots` - обработанная и архивированная идея; отдельный оптимизатор поглощен audit-stack.
- `Система/Правила обработки/YAML-модель active wiki.md` - активное правило целевой YAML-модели для постоянных карточек `Вики`.

## Итог обработки

Решение принято и внедрено как правило active wiki:

- обычная active wiki карточка не должна получать `status: active` и `use: public` автоматически;
- для новых ограниченных карточек используется `use: context` или `use: internal`;
- legacy `usage_mode`, `public_faq_route`, `proof_role`, `public_proof`, `current_terms_status`, `public_terms` не удаляются механически и остаются читаемыми;
- METHOD/DEEP получили совместимость с `use`;
- массовая миграция существующих карточек не выполнялась.
