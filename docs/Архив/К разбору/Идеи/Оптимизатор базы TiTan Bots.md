# Оптимизатор базы TiTan Bots

## Оценка ценности

`10/10`

## Идея

Сделать отдельный процесс или скрипт `Оптимизатор базы`, который проверяет не технические ссылки, а качество и порядок wiki TiTan Bots.

Он не должен автоматически переписывать базу без одобрения. Его задача - найти проблемы, сгруппировать их и предложить план улучшения.

## Почему может быть ценно

- база быстро растет, и старые заметки не всегда соответствуют новым шаблонам;
- нельзя тратить токены на массовую нормализацию прямо во время важных импортов;
- нужен отдельный способ находить технический долг базы;
- будущему Telegram-боту и AI-помощнику нужна более чистая база знаний;
- шаблоны уже появились, но старые заметки пока не приведены к ним.
- при переходе в новый чат перед импортом нужно быстро понять, находится ли важная логика в базе, а не только в старом диалоге.

## К чему относится

- Obsidian wiki;
- Codex workflow;
- импорт сырья;
- шаблоны;
- архив;
- подготовка базы для будущего AI/Telegram-бота.

## Что проверить

- Какие проверки можно сделать простым скриптом без LLM.
- Какие проверки требуют анализа Codex/LLM.
- Как формировать отчет без автоматической правки wiki.
- Где хранить отчет аудита.
- Как не превращать аудит в лишнюю бюрократию.
- Добавить ли в аудит отдельный блок "готовность нового чата к импорту": проверка правил импорта, структуры `Материалы`, состояния ссылок, синхронизации skill и наличия ключевых правил в `Система`.

## Как понять, что идея сработала

- После запуска видно список проблем базы.
- Проблемы сгруппированы по важности.
- Можно выбрать 1-2 действия и быстро улучшить базу.
- `check-links.ps1` остается быстрым техническим скриптом.
- Оптимизатор не мешает текущему импорту материалов.
- Новый чат перед импортом может быстро проверить базу и показать недостающие правила без немедленных правок.

## Риски или сомнения

- можно сделать слишком сложный аудит;
- можно начать чинить старые заметки вместо текущей работы;
- можно получить слишком большой отчет, который никто не будет разбирать.
- можно превратить разовую самопроверку нового чата в обязательную бюрократию перед каждым мелким импортом.

## Источник

- Обсуждение шаблонов и будущей нормализации базы в Codex 2026-05-27.

## Итог закрытия

Статус: `absorbed / processed`

Отдельный `Оптимизатор базы` не нужен. Смысл карточки поглощен текущей вертикалью audit-stack:

- FAST закрывает техническую и машинно-читаемую гигиену: ссылки, YAML, индексы, forbidden paths, template structure;
- DEEP закрывает безопасные semantic auto-fix и grouped residue через существующие `К разбору/Ошибки` / `К разбору/Идеи`;
- METHOD закрывает FAQ/RAG route usefulness и качество цепочек для Telegram/RAG;
- META закрывает audit/system integrity: skills, scripts, templates, reports, prompts, `.codex` skill sync;
- SYNTHESIS создает новые идеи/ошибки из Active Wiki после anti-dup, не меняя Active Wiki.

Остаток `готовность нового чата к импорту` оформлен не как новый процесс, а как optional readiness-паттерн в `Audit Stack Map` и `META - Правила работы`:

- для нового чата, крупного импорта или изменения правил использовать FAST + META-SYSTEM;
- для полной проверки базы использовать weekly audit-stack;
- не запускать полный stack перед каждой мелкой карточкой.

Старое упоминание будущего оптимизатора в `Процесс обработки YouTube-видео` заменено на текущий audit-stack.

Карточка больше не является активной идеей и переносится в архив очереди как обработанная.
