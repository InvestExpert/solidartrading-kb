# S003 — AI token cost optimization / Claude Code / Ponytail / Headroom / Memento / Mem0

## Назначение

Сырой входящий материал для последующего разбора Codex.

Материал не является текущим знанием базы, правилом разработки, рекомендацией к внедрению или проверенным техническим выводом.

## Статус источника

- Источник: YouTube-видео по ссылке пользователя.
- Транскрибация: передана пользователем отдельным текстовым файлом.
- Статус транскрибации: **частичная**.
- Ограничение: в конце файла указано, что транскрипт обрезан до первых 30 минут в рамках бесплатного лимита.
- Проверка внешних фактов: **не выполнена**.

## Короткое содержание

Видео разбирает проблему расходов на токены при работе с Claude Code / Codex-подобными агентами и предлагает стек из нескольких инструментов для снижения затрат:

1. **Ponytail** — уменьшение избыточного кода и выходных токенов модели.
2. **Headroom** — сжатие tool outputs / логов / JSON / HTML / больших ответов инструментов перед попаданием в контекст модели.
3. **Memento / Memanta** — локальная память между сессиями, типизация памяти, conflict detection, provenance.
4. **Mem0 / Memo** — более зрелая система памяти с сильными бенчмарками, но с другими компромиссами.

Основная идея: экономия токенов не решается одним инструментом, потому что утечки находятся на разных слоях:

- output модели;
- input от инструментов;
- повторное объяснение контекста между сессиями.

## Сырые claims автора

> Все пункты ниже — claims из видео. Не считать проверенными фактами без отдельной проверки.

### 1. Где утекают токены

- Модель пишет лишний код, переусложняет решения и повторяет уже существующую логику.
- Tool outputs могут возвращать слишком большие логи, JSON, RAG-куски, историю диалогов.
- Новые сессии требуют повторного объяснения проекта, решений и предпочтений.
- Выходные токены дороже входных, поэтому сокращение кода может быть важнее сокращения контекста.

### 2. Общая экономика

- Автор утверждает, что при тяжелых workflow можно снизить счет в 1.5–3.5 раза.
- Автор утверждает, что меньше $50/мес экономия может не окупить время настройки.
- Автор утверждает, что больше $200/мес инструменты могут окупиться быстро.
- Автор отдельно предупреждает, что проценты экономии разных инструментов не перемножаются.

### 3. Prompt caching

- Автор утверждает, что Anthropic prompt caching работает как встроенный слой экономии.
- Рекомендуемый первый шаг — проверить `/cost` и долю `cache reads`.
- Если `Claude.md` / `CLAUDE.md` раздут до 1000 строк, автор предлагает сократить его примерно до 200 строк.

### 4. Ponytail

- Слой: output styling / slash command / инструкция для модели.
- Цель: модель пишет меньше лишнего кода.
- Ключевая логика:
  - не писать код, если он не нужен;
  - сначала искать решение в стандартной библиотеке;
  - затем в нативных возможностях платформы;
  - затем в уже подключенных библиотеках проекта;
  - использовать минимальную реализацию;
  - не резать валидацию, обработку ошибок, безопасность и accessibility.
- Автор утверждает, что Ponytail особенно полезен на Opus и UI/backend-задачах с большим количеством boilerplate.
- Автор предупреждает, что на reasoning-задачах или простых задачах эффект может быть нулевым или отрицательным.
- Автор сравнивает Ponytail и Caveman:
  - Ponytail режет код;
  - Caveman режет разговорный стиль;
  - ставить оба вместе может быть бессмысленно.

### 5. Headroom

- Слой: proxy / wrapper между инструментами и моделью.
- Цель: сжатие tool outputs до попадания в контекст.
- Автор описывает режимы:
  - wrap CLI;
  - local proxy;
  - library import.
- Типы данных:
  - JSON;
  - исходный код;
  - разговорный текст;
  - логи;
  - shell output;
  - HTML.
- Автор утверждает, что Headroom полезен при больших логах, JSON, MCP outputs, RAG-корпусах.
- Автор предупреждает о рисках:
  - возможные проблемы AST-сжатия Python;
  - баги с Codex Desktop;
  - проблемы Mac Intel;
  - CCR retrieval bugs;
  - memory overhead;
  - overhead на коротких сообщениях.

### 6. Memento / Memanta

- Слой: память между сессиями.
- Автор описывает инструмент как локальную систему памяти без классической vector DB.
- Claims:
  - локальная установка через Docker / Ollama;
  - 13 типизированных категорий памяти;
  - conflict detection;
  - provenance tracking;
  - отсутствие API-ключей;
  - подходит early adopters.
- Автор считает преимуществом типизацию памяти:
  - `instruction`;
  - `fact`;
  - `decision`;
  - `goal`;
  - `commitment`;
  - `preference`;
  - `relationship`;
  - `context`;
  - `event`;
  - `learning`;
  - `observation`;
  - `artifact`;
  - `error`.

### 7. Mem0 / Memo

- Автор позиционирует Mem0/Memo как более зрелый и безопасный выбор для большинства.
- Claims:
  - больше комьюнити;
  - выше бенчмарки памяти;
  - больше интеграций;
  - production usage;
  - managed cloud.
- Минусы по автору:
  - silent override при конфликтующих фактах;
  - нет строгой типизации памяти;
  - LLM call на write;
  - векторная база требует индексации;
  - возможный vendor lock-in при облачной версии.

### 8. Порядок внедрения по автору

1. Проверить prompt caching.
2. Почистить `Claude.md` / `CLAUDE.md`.
3. Поставить Ponytail.
4. Поставить Headroom только при тяжелых tool outputs.
5. Поставить Memento / Mem0 только для долгих проектов с большим количеством сессий.

## Возможная применимость к базе TiTan / SolidarTrading

### A. Для Cursor / Codex / Claude Code workflow

- Сделать отдельный технический checklist по снижению токенов:
  - audit prompt size;
  - минимизация постоянного контекста;
  - разделение active context / raw / archive;
  - запрет тащить длинные логи целиком;
  - краткие summary вместо полных файлов, если не нужна точность;
  - routing к нужным owner-карточкам вместо общего дампа базы.

### B. Для текущей Obsidian-базы

- Идея совпадает с уже принятой архитектурой:
  - `Вики` = active knowledge;
  - `К разбору` = сырьё / входы;
  - `Система` = правила / скрипты / аудит;
  - `Архив` = неактивное.
- Можно проверить, нет ли избыточного контекста в:
  - системных промптах;
  - Codex prompts;
  - `CLAUDE.md`;
  - индексах;
  - audit reports;
  - master research files.

### C. Для обработки логов и тестов ATLAS

- Headroom-подобный слой потенциально применим к:
  - MT5 optimizer outputs;
  - большим XML/CSV;
  - Python logs;
  - trade logs;
  - длинным comparison reports.
- Риск: сжатие логов может скрыть важные крайние случаи, если алгоритм потеряет структуру или контекст.

### D. Для памяти между сессиями

- Идея Memento/Mem0 пересекается с потребностью:
  - помнить проектные решения;
  - хранить rejected decisions;
  - фиксировать reason why;
  - не перетирать старые решения новыми без истории.
- Но для текущей базы уже есть Obsidian + Git/архивная логика, поэтому memory-tool не должен заменить канонические карточки.

## Возможные идеи для Codex

### IDEA-01 — Контекстный бюджет для Codex / Cursor

Создать правило или checklist:

```text
Перед крупным Codex-заданием:
1. оценить, какие файлы реально нужны;
2. не грузить весь проект без причины;
3. не передавать большие логи целиком;
4. использовать extracted summary + точечные файлы;
5. сохранять raw отдельно;
6. результат переносить в Вики только после проверки.
```

### IDEA-02 — Audit prompt diet

Проверить, не раздуты ли:
- ChatGPT guide;
- Codex prompts;
- `CLAUDE.md`;
- audit-stack prompts;
- project context docs.

Цель: убрать повторение, оставить routing и owner rules.

### IDEA-03 — Tool-output policy для логов и тестов

Сформировать правило:

```text
Большие логи не отправлять модели целиком, если задача решается агрегатами.
Сначала строить deterministic summary:
- total rows;
- key groups;
- worst cases;
- deltas;
- outliers;
- sample rows;
- ссылки на raw.
```

### IDEA-04 — Memory conflict policy

Вынести принцип:

```text
Новая память не должна молча затирать старое решение.
Если есть конфликт:
- сохранить старую версию;
- указать дату / источник;
- указать причину изменения;
- пометить актуальный owner.
```

### IDEA-05 — Проверка применимости Ponytail-логики к промптам пользователя

Для кода у пользователя уже есть правило минимальных правок. Можно усилить промпты:

```text
Не изобретать новую архитектуру, пока не проверены существующие механизмы.
Не писать новый код, если достаточно минимальной правки.
Не добавлять слой, если существующий слой уже владелец.
```

## Риски и слабые места материала

### RISK-01 — Много claims требуют внешней проверки

В видео много чисел:
- цены токенов;
- количество звезд GitHub;
- даты релизов;
- лицензии;
- результаты бенчмарков;
- баги из issues;
- утверждения про компании и авторов;
- реальные проценты экономии.

Без web/research эти claims нельзя переносить в `Вики` как факты.

### RISK-02 — Возможная путаница названий

В транскрипции встречаются искажения:
- `Memanto`, `Memento`, `Memanta`;
- `Memo`, `Mem0`, `МЭМ`;
- `Moorsh-Chek`;
- `rack`, `рак`, `RAG`;
- `код-код`, `Claude Code`;
- `clod.markdown`, вероятно `CLAUDE.md`.

Перед созданием карточек нужно нормализовать названия по официальным источникам.

### RISK-03 — Транскрипт обрезан

Файл содержит только первые 30 минут. Последняя часть видео может содержать:
- финальные выводы;
- команды установки;
- ссылки;
- уточнения;
- дисклеймеры;
- сравнение с другими инструментами.

Для полной обработки нужен полный transcript.

### RISK-04 — Технические инструменты быстро устаревают

Все сведения о GitHub stars, версиях, багфикcах, лицензиях и поддержке могут устареть за дни/недели.

### RISK-05 — Compression может ломать аудит

Если сжимать логи, XML, CSV или код перед анализом, можно потерять:
- редкий edge case;
- строку с ошибкой;
- порядок событий;
- точную сделку;
- паттерн forced/recovery;
- compatibility issue.

Для аудита кода и торговой логики raw должен оставаться доступен.
