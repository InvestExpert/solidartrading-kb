## Короткое содержание

Видео показывает концепцию персональной агентной AI-системы на базе текстовых файлов, Obsidian, Claude Code/другой оболочки, skills, Supabase, Wiki LLM, VPS-синхронизации и n8n-интеграций.

Основная логика автора: агентная система должна быть не просто чатом, а управляемой файловой операционной средой, где модель получает нужный контекст через маршрутизацию, skills, raw/wiki-структуры, внешние коннекторы и регулярные рутины.

## Главные claims из видео

- Современная агентная система описывается как слои: AI-модель → контекст/память → harness/оболочка → рабочие процессы → коннекторы.
- Хорошая система должна быть model-agnostic и harness-agnostic: не зависеть жёстко от конкретной модели или конкретной оболочки.
- Практическая база агентной системы может быть построена как набор правильно расположенных текстовых файлов, которые попадают в контекст модели в нужный момент.
- Obsidian используется как удобный редактор и навигатор по Markdown-файлам, а не как обязательная вычислительная часть.
- Корневой routing-файл вроде CLAUDE.md/Cloud.md описывает структуру каталога и помогает агенту понимать, где искать нужные данные.
- В структуре предлагаются отдельные зоны: контекст, daily, intelligence, projects, resources, skills.
- Скиллы используются как способ автоматически создавать структуру файлов, настраивать рутины и подключать дополнительные workflows.
- Агент может проактивно запускаться по расписанию, читать подключённые источники и собирать ежедневный todo/list по важным темам.
- Supabase используется как внешний источник операционных данных; в демонстрации акцент сделан на read-only доступе.
- Wiki LLM по методу Карпатова используется как отдельная структура для ingest/query: raw-файлы обрабатываются в сеть связанных wiki-заметок.
- Локальную Obsidian-базу можно синхронизировать с VPS через Syncthing и использовать агентную оболочку на сервере.
- Через n8n можно вызывать Claude Code/headless-режим на сервере и подключать Telegram, Slack, email и другие внешние интерфейсы.

## Возможная применимость к базе TiTan / SolidarTrading

- Сделать для базы TiTan/SolidarTrading отдельную карту маршрутизации для AI-агентов: где лежат Вики, К разбору, Система, Архив, правила, промпты, индексы.
- Создать импортный workflow для AI-видео: видео/транскрипт → сырьевой MD → candidates → К разбору/Идеи или Ошибки → только после проверки в Вики.
- Сделать skill/промпт для Codex: обрабатывать `К разбору/Сырье` без изменения Active Wiki, создавать только идеи с source_basis, required_test и next_step.
- Использовать модель `model/harness agnostic`: база должна работать через ChatGPT, Codex, Telegram-бот, Coze и будущие агенты без привязки к одному интерфейсу.
- Проверить, нужен ли отдельный `AI Operator` для ежедневной выжимки новых сырых материалов, встреч, чатов и задач по проектам.
- Рассмотреть read-only connector policy: внешние источники по умолчанию доступны агентам только на чтение; запись разрешается только после явного owner-решения.
- Добавить отдельный raw-to-ideas процесс для материалов про AI, чтобы они не смешивались с Golden Connect и торговыми ботами.
- Рассмотреть синхронизацию Obsidian-базы с VPS для внешнего Telegram/RAG-интерфейса, но только после проверки безопасности и доступа.
- Использовать структуру `raw → extracted claims → risks → checks → candidate ideas`, чтобы Codex не тратил токены на первичную расшифровку.
- Сделать антидубль по идеям перед созданием новых карточек: AI-видео часто повторяют одни и те же концепты разными словами.

## Риски и слабые места

- Нельзя переносить claims из видео напрямую в Вики как verified knowledge. Это источник/мнение автора, а не подтверждённый факт.
- Автоматическая обработка raw-файлов может создать шум, дубли и псевдоидеи, если нет quality gate.
- Привязка к Claude Code skills может быть не полностью model/harness agnostic: формат skills и доступные коннекторы зависят от оболочки.
- Коннекторы к Supabase, n8n и серверу несут риск утечки данных или несанкционированной записи, если не ограничить права.
- n8n execute-command на сервере — потенциально опасная поверхность: ошибка в команде или prompt injection может вызвать нежелательные действия.
- Синхронизация Obsidian с VPS через Syncthing требует отдельной модели доступа, backup и контроля конфликтов.
- Автоматическое дозаписывание контекста агентом может загрязнять базу неточными выводами.
- Если корневой routing-файл станет источником знаний, возникнет конфликт с принципом: Вики — knowledge, Система — правила, К разбору — входы.
- Транскрибация содержит ошибки распознавания: `CLДкод`, `Superbase`, `N8 CNN`, `Гермес/Герме`, искажения имён и терминов.
