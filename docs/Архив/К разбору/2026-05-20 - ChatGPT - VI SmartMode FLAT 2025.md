## Краткое резюме

В чате разбиралась настройка SmartMode для FLAT-бота на MT5. Основная задача — сделать адаптивный фильтр по VI, который не просто увеличивает прибыль, а стабилизирует месячные результаты за 2025 год.

Ключевая работа шла вокруг:
- адаптивного шага сетки;
- буфера защиты;
- трейлинга;
- низкой, средней и высокой VI;
- статистики исходов V1, V1.5, V2, V5, V6, V7;
- месячных тестов: апрель, май, январь, февраль;
- годового теста 2025.

Основной практический итог: новый двусторонний VI-мэппинг стал работать заметнее после масштабирования параметров. По тестам 2025 сформировался рабочий центр параметров около:

- `SmartCoefDelta = 0.4`
- `SmartBufDelta = 0.3`
- `SmartTrailDelta = 0.2`

Для мая это подтвердилось как сильная зона. Для января/февраля дополнительные агрессивные варианты low-vol мэппинга существенного улучшения не дали, поэтому было решено оставить оригинальный протестированный мэппинг.

## Идеи

- Делать VI-мэппинг не только вверх, но и вниз:
  - при низкой VI уменьшать шаг сетки;
  - при высокой VI усиливать buffer/trailing;
  - при средней VI делать плавный переход без резких скачков.

- Разбить низкую VI на реальные корзины, а не оставлять одну грубую зону `VI <= 1.2`.

- Использовать отдельные корзины:
  - `VI <= 0.50`
  - `0.50 < VI <= 0.55`
  - `0.55 < VI <= 0.65`
  - `0.65 < VI <= 0.75`
  - `0.75 < VI <= 0.90`
  - `0.90 < VI <= 1.00`
  - `1.00 < VI <= 1.10`
  - `1.10 < VI <= 1.20`

- Для середины `1.20–1.60` не оставлять большую ступень. Делать плавный переход:
  - `1.20–1.30`
  - `1.30–1.40`
  - `1.40–1.50`
  - `1.50–1.60`

- Для высокой VI усиливать `k_coef` и `m_coef` сильнее, чем `coef`, потому что при высокой волатильности важнее защита и управление выходом, а не только расширение сетки.

## Гипотезы

- В низкой волатильности главная проблема — слишком мало движений до тейка. Поэтому уменьшение шага может увеличить количество завершённых тейков.

- В низкой VI чрезмерное сжатие buffer/trailing может дать обратный эффект: больше ранних выбросов и ухудшение ExitPNL.

- В высокой VI шаг не должен расти слишком агрессивно. Лучше сильнее адаптировать buffer/trailing, чтобы сглаживать выходы.

- Симметричный мэппинг вверх/вниз не обязан быть лучше. Рынок и механика сетки асимметричны:
  - low VI → задача зацепить движение;
  - high VI → задача пережить амплитуду и не получить дорогой выход.

- Если фильтр влияет только на один месяц, это слабый фильтр. Он может быть полезен, но для полноценной универсальности должен давать устойчивый эффект хотя бы на группе режимов.

## Выводы

- Старый фильтр почти не влиял на другие месяцы и давал заметный эффект в основном в апреле/феврале. Это недостаточно для полноценного универсального SmartMode.

- Масштабирование параметров было нужно. Старые значения вроде `0.001`, `0.01` были неудобны для оптимизации. Новый масштаб с параметрами вроде `0.4 / 0.3 / 0.2` удобнее.

- На годовом тесте 2025 с новым двусторонним мэппингом появилась рабочая зона:
  - `SmartCoefDelta ≈ 0.4–0.6`
  - `SmartBufDelta ≈ 0.2–0.3`
  - `SmartTrailDelta ≈ 0.2–0.3`

- После дополнительных проверок:
  - `0.4` в buffer/trailing хуже;
  - `0.25` заметного эффекта не даёт;
  - рабочий базовый вариант оставлен около `0.4 / 0.3 / 0.2`.

- Май подтвердил пользу зоны `Coef≈0.4–0.5`, `Buf≈0.2–0.3`, `Trail≈0.2–0.3`.

- Для января/февраля дополнительные варианты low-vol мэппинга L1/L2 почти не повлияли. Вывод: оставлять оригинальный мэппинг, потому что он уже протестирован и не ломает год.

## Ошибки и решения

### Ошибка: анализ без проверки реальных VI запусков

В феврале было несколько запусков с `VI <= 1.2`, включая очень низкие значения около `0.5–0.75`. Изначально мэппинг не учитывал отдельные зоны ниже `0.75`.

Решение:
- построить список запусков по VI;
- добавить нижние корзины под реальные данные;
- не строить мэппинг только на абстрактных порогах.

### Ошибка: слишком грубая зона `1.2–1.6`

В старом варианте была крупная зона `1.2–1.6`. Это скрывало поведение внутри середины.

Решение:
- разбить `1.2–1.6` на ступени по 0.1;
- сохранить старое значение на `VI=1.6`, чтобы не сломать уже протестированный уровень.

### Ошибка: неправильное масштабирование

При переходе на удобные параметры нельзя просто умножать без понимания масштаба. Нужно, чтобы новые параметры давали тот же физический эффект.

Решение:
- использовать коэффициенты прямо в формуле;
- например, вместо `3.0*(0.01*SmartCoefDelta)` писать `0.03*SmartCoefDelta`;
- не плодить лишние переменные, если код должен оставаться простым для оптимизации.

### Ошибка: квантование процента вместо шага

В коде было:

```cpp
double stepQuant = 0.01; // квант 0.01%
g_snap.gridStepPerc_smart = MathFloor(g_snap.gridStepPerc_smart * coef / stepQuant + 0.5) * stepQuant;
```

Проблема: квантуется процент, а не фактический шаг. Это может искажать геометрию сетки.

Решение:
- применять коэффициент к шагу;
- квантовать уже рассчитанный шаг;
- не делать двойное квантование, если выше уже есть квантование базового smart-step.

### Ошибка: усложнение вместо минимального изменения

Были предложения с лишними переменными и длинными блоками. Пользователь выбрал минимальный стиль:
- одна строка на условие;
- без лишних промежуточных переменных;
- сохранять формат существующего кода.

## Правила

- Сначала смотреть реальные значения VI в логах, потом предлагать пороги.

- Для low-vol режима сначала проверять, сколько запусков реально попадает в `VI <= 1.2`.

- В low VI не надо автоматически усиливать всё подряд. Основной рычаг — шаг. Buffer/trailing трогать осторожно.

- В high VI не делать ставку только на шаг. Основной рычаг — buffer/trailing.

- Не стремиться к симметрии мэппинга. Стратегическая цель — стабильность, а не красивая линейная лестница.

- Если новый вариант почти не влияет на результаты, откатывать его и оставлять протестированный оригинал.

- Для wiki и дальнейшей разработки сохранять только:
  - проверенные выводы;
  - финальные мэппинги;
  - причины откатов;
  - ошибки и решения.

## Архитектура

### SmartMode в Regime_BeginSession

Адаптивные параметры фиксируются при старте сессии:

- `g_snap.vi`
- `g_snap.adx50`
- `g_snap.absVol`
- `g_snap.gridStepPerc_smart`
- `g_snap.gridStepPerc_coef`
- `g_snap.forcedBufferPerc_smart`
- `g_snap.trailingBufferPerc_smart`
- `g_snap.forcedBufferPerc_coef`
- `g_snap.trailingBufferPerc_coef`

Это правильная архитектура: параметры фиксируются на старте сетки и не меняются хаотично внутри сессии.

### Слои адаптации

В текущем коде есть два слоя:

1. `AbsVol`-формула для базового smart-step:
   - текущий вариант Magnet:
   - `S = S0 + coef * (ln(VI/pivot) / (1 + |ln(VI/pivot)|))`

2. VI-мэппинг поверх:
   - меняет `coef`;
   - меняет `k_coef`;
   - меняет `m_coef`.

### Логирование

Полезные поля в `STAT_ROW`:

```text
ts_start, VI, TI, AbsVol, AbsVolCoef, step, step_coef,
forcedBufferPerc_smart, trailingBufferPerc_smart,
forcedBufferPerc_coef, trailingBufferPerc_coef,
ATR_ratio, eventV, pnl, success, reason
```

Это хороший формат для последующего анализа в Python/Excel/Obsidian.
