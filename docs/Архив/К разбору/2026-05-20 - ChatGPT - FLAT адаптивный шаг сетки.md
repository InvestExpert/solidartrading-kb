## Краткое резюме

В чате анализировалась задача адаптации шага сетки FLAT-бота под разные режимы волатильности.

Главный вопрос: можно ли подобрать одну формулу и один базовый шаг, чтобы бот стабильно работал на разных месяцах, не ловил глубокие провалы и сохранял прибыльные плато.

Ключевой вывод: простая адаптация шага только через месячный индекс волатильности помогает частично, но не решает всю задачу. Февраль и апрель показали разные рабочие зоны. Если формула даёт хороший результат в апреле, она может ухудшать февраль. Это говорит, что одной оси `AbsVol` может быть недостаточно.

## Идеи

- Разделить SmartMode на две независимые фазы:
  - первая фаза: адаптация базового шага сетки от глобальной волатильности;
  - вторая фаза: мягкая локальная подстройка, но без разрушения плато.
- Считать стабильность важнее пика прибыли:
  - широкий рабочий диапазон лучше одиночной вершины;
  - формула, которая даёт резкие провалы при малом изменении base/coef, считается плохой.
- Проверять формулу не по одному месяцу, а по конфликтным режимам:
  - высокая волатильность;
  - низкая волатильность;
  - слабые месяцы;
  - прибыльные месяцы.
- Рассматривать `0.33` не как догму, а как возможный устойчивый узел текущей механики FLAT.
- Не лечить плохую формулу клампами по результату. Если без ограничителей формула нестабильна, она инженерно слабая.

## Гипотезы

- `0.33` часто всплывает как хороший шаг из-за связки:
  - `gridStepPerc`;
  - `forcedBufferPerc`;
  - `trailingBufferPerc`;
  - геометрии сетки;
  - структуры выходов V1–V6.
- Старый вариант формулы мог работать стабильнее, потому что фактически использовал низковолатильный режим как базис и в основном увеличивал шаг.
- Pivot-формула с `SmartPivotValue = 1.50` даёт локальную стабильность, но может слабо смещать слабые месяцы.
- Если февраль и апрель не имеют пересечения хороших зон по одной формуле, то проблема не в оптимизации коэффициента, а в недостаточности самой модели.
- Для месяцев вроде марта/июля может потребоваться второй рычаг, например трендовость через ADX, а не только волатильность.

## Выводы

- Главный управляющий параметр в текущих тестах — `gridStepPerc_smart`.
- Изменение шага может сильно менять PnL, даже если остальная торговая логика не меняется.
- Первая фаза SmartMode должна менять только шаг и не трогать торговую логику.
- Вторая фаза, если она меняет сразу шаг, буфер и трейлинг, легко вносит хаос.
- Формула должна оцениваться по стабильности плато, а не по лучшему одиночному проходу.
- Для универсального FLAT-режима недостаточно смотреть только на прибыль:
  - нужно учитывать худший месяц;
  - размах PnL;
  - устойчивость соседних параметров;
  - поведение на слабых и сильных месяцах.
- Если формула даёт глубокий провал при небольшом изменении коэффициента, она опасна для live-режима, потому что волатильность меняется внутри месяца и дня.

## Ошибки и решения

### Ошибка: оценка по пику прибыли

Проблема: одиночный лучший результат может быть случайным резонансом.  
Решение: оценивать плато и соседние значения.

### Ошибка: постоянная смена формулы во время тестов

Проблема: если формула постоянно меняется, нельзя понять, что реально улучшило результат.  
Решение: фиксировать одну формулу на серию тестов и менять только один параметр или узкую пару параметров.

### Ошибка: попытка исправить нестабильность клампами

Проблема: кламп может скрыть дефект формулы и зарезать потенциально рабочие режимы, например шаг около `0.41`.  
Решение: сначала найти устойчивую формулу без искусственных границ. Ограничители использовать только как риск-контроль, а не как способ сделать плохую формулу хорошей.

### Ошибка: считать, что pivot решает конфликт месяцев

Проблема: pivot смещает запись формулы, но сам по себе не решает конфликт рабочих зон февраля и апреля.  
Решение: проверять реальные эффективные шаги и итоговые плато по месяцам.

### Ошибка: трогать торговую логику при поиске формулы шага

Проблема: изменение количества ордеров, уровней или механики выхода смешивает эффекты.  
Решение: на этапе поиска формулы менять только `gridStepPerc_smart`.

## Правила

- Не менять торговую логику при тесте формулы адаптивного шага.
- Сначала тестировать первую фазу SmartMode отдельно.
- Вторую фазу подключать только после стабилизации первой.
- Оценивать формулу по стабильности:
  - широкий плюс;
  - отсутствие глубоких провалов;
  - приемлемый худший месяц;
  - улучшение слабых месяцев без разрушения сильных.
- Не считать апрель единственным эталоном.
- Не считать февраль единственным эталоном.
- Не делать вывод по одному PnL без анализа соседних параметров.
- Если хорошие зоны февраля и апреля не пересекаются, не продолжать бесконечно крутить coefficient. Нужно пересмотреть модель.
- Большие сырые таблицы не переносить в wiki полностью. Сохранять выводы и ключевые диапазоны.

## Формулы, которые обсуждались

### v1 — простая степенная формула

```mql5
g_snap.gridStepPerc_smart = gridStepPerc * MathPow(g_snap.absVol, g_snap.absVolCoef);
```

Наблюдение:
- давала смещение по месяцам;
- могла работать стабильнее, потому что низкая волатильность фактически была ближе к базису;
- в ряде тестов старый вариант давал общий результат лучше pivot-варианта.

### v2 — pivot-степенная формула

```mql5
g_snap.gridStepPerc_smart = gridStepPerc * MathPow(g_snap.absVol / SmartPivotValue, g_snap.absVolCoef);
```

Наблюдение:
- при `SmartPivotValue = 1.50` дала хорошую локальную стабильность в апреле;
- но февраль и апрель не пересеклись по хорошим зонам;
- при слишком большом коэффициенте размах PnL резко увеличивался;
- при малом коэффициенте апрель был плавнее, но февраль мог ухудшаться.

### v3 — log-damper

```mql5
g_snap.gridStepPerc_smart =
    gridStepPerc * MathExp(
        (g_snap.absVolCoef / (1.0 + MathAbs(MathLog(g_snap.absVol / SmartPivotValue))))
        * MathLog(g_snap.absVol / SmartPivotValue)
    );
```

Идея:
- приглушить показатель при больших отклонениях от pivot;
- сохранить локальную реакцию.

Ограничение:
- эффект смещения оказался слабым.

### v4 — tanh

```mql5
g_snap.gridStepPerc_smart =
    gridStepPerc * MathExp(
        g_snap.absVolCoef * MathTanh(MathLog(g_snap.absVol / SmartPivotValue))
    );
```

Идея:
- плавно ограничить влияние крайних значений волатильности;
- не рвать прибыльное плато.

Наблюдение:
- потенциально лучше для стабильности;
- требует полного теста по месяцам.

### v5 — magnet-формула

```mql5
g_snap.gridStepPerc_smart =
    gridStepPerc
    + g_snap.absVolCoef
    * (MathLog(g_snap.absVol / SmartPivotValue)
       / (1.0 + MathAbs(MathLog(g_snap.absVol / SmartPivotValue))));
```

Идея:
- магнитить шаг к базовому `gridStepPerc`;
- коэффициент задаёт максимальную дельту к шагу;
- насыщение защищает от крайних скачков.

Риск:
- это уже другая логика: coefficient становится не степенью, а абсолютной дельтой шага.

## Технические наблюдения по параметрам

Из справки по FLAT:

- `GridStep S`:
  - меньше шаг → больше сделок, чаще тейки, меньше тейк на одну сделку;
  - больше шаг → меньше сделок, дольше удержание, больше тейк, но ниже частота.
- `Buffer k·S`:
  - больше буфер → защита дальше, реже срабатывает;
  - меньше буфер → защита ближе, но слабее.
- `Trailing m·S`:
  - влияет на скорость подтягивания BE;
  - слишком резкая подстройка может менять структуру исходов V1–V6.

## Что проверить дальше

- Повторить тесты v1 и v2 на одном наборе месяцев с одинаковыми параметрами.
- Для каждой формулы считать:
  - сумму PnL;
  - худший месяц;
  - размах PnL;
  - количество месяцев в плюсе;
  - устойчивость соседних значений base/coef.
- Проверить v4 `tanh` как кандидат на компромисс между смещением и стабильностью.
- Проверить v5 `magnet` отдельно, потому что коэффициент меняет смысл.
- Разделить тесты:
  - только первая фаза;
  - первая + вторая фаза;
  - без изменения буфера/трейлинга;
  - с изменением буфера/трейлинга.
- Для месяцев, где волатильность не объясняет оптимальный шаг, проверить ADX/трендовость как второй рычаг.
- Не переносить в live формулу, которая даёт глубокие провалы при небольшом изменении входного VI.
