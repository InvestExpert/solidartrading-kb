# Codex - проверить стек экономии токенов Ponytail Headroom Memento Mem0

## Оценка ценности

`8/10`

Идея может быть ценной, потому что TiTan Bots уже много работает через Codex, Obsidian, skills, audit-stack и большие raw/import задачи. Если дополнительные инструменты реально снижают расход токенов без потери качества, это может ускорить импорт, аудит и работу с кодом. Но источник является частичным YouTube-разбором, а claims по инструментам быстро устаревают, поэтому это не активное правило и не инструкция к установке.

## Lifecycle

Эта карточка живет в `К разбору/Идеи`.

Это не active/current знание и не ответ для RAG. Идея становится active wiki-карточкой только после проверки, решения владельца и отдельного переноса в нужный слой. Если реальная идея отклонена, перенести в `Архив/К разбору/Отклонено` с причиной и условием возврата.

Если карточка оказалась не настоящей идеей, а старым пробелом, остатком импорта, дублем задачи формирования базы или закрытым owner-gap, после закрытия перенести ее в обычный `Архив/К разбору`, не в `Отклонено`.

`К разбору/Идеи` и `К разбору/Ошибки` являются отдельными ручными входами после audit-stack. Status/queue/warning/guard не является финальным закрытием без terminal owner или lifecycle destination.

## Идея

Проверить, нужен ли TiTan Bots дополнительный стек экономии токенов для Codex/Cursor/Claude Code-подобных workflow:

- Ponytail - ограничение избыточного кода и лишнего output;
- Headroom - сжатие больших tool outputs, логов, JSON, HTML и похожих портянок перед попаданием в контекст;
- Memento / Memanta - локальная память между сессиями с типизацией, provenance и конфликтами;
- Mem0 / Memo - более зрелый вариант агентной памяти с другими компромиссами.

Цель проверки - не поставить все подряд, а понять, есть ли практическая польза сверх уже принятых Caveman и Context Mode.

## Почему может быть ценно

- Может снизить расход токенов на больших импортных задачах, аудитах, поиске дублей и работе с длинными логами.
- Может уменьшить повторное объяснение контекста между сессиями.
- Может помочь отделить три разных источника затрат: лишний output модели, большие ответы инструментов и повторная загрузка контекста.
- Может дать более строгую политику для tool-output: меньше читать сырые портянки в чат, больше извлекать только нужные выводы.
- Может дополнить текущую системную карточку `Правила оптимизации токенов Codex`, если проверка покажет реальную пользу.

## К чему относится

- Codex workflow в TiTan Bots.
- Большие импорты Obsidian raw materials.
- Audit-stack и generated reports.
- Работа с логами, тестами, backtest/output таблицами и большими `rg`.
- Память между сессиями и конфликтующие решения проекта.

## Что уже есть

В базе уже приняты как контролируемый эксперимент:

- Caveman;
- Context Mode.

Эта идея не заменяет их и не требует менять действующие правила до проверки.

## Что проверить

- Актуальные официальные страницы и репозитории Ponytail, Headroom, Memento/Memanta и Mem0/Memo.
- Работают ли эти инструменты с Codex в текущей среде, а не только с Claude Code.
- Не ломают ли они качество импорта, audit-fast/audit-deep, wiki-линковку, торговые выводы и работу с кодом.
- Есть ли риск скрытой потери смысла при сжатии логов, отчетов или raw materials.
- Можно ли использовать только отдельные идеи из видео без установки внешних инструментов.
- Какой минимальный тест даст честное сравнение с текущей связкой Caveman + Context Mode.

## Как понять, что идея сработала

- На одинаковых типовых задачах видно снижение input/output/tool tokens по `/status`.
- Проверки базы и качество итоговых карточек не ухудшаются.
- Codex не пропускает важные факты из raw materials и логов.
- Время на большие задачи сокращается или хотя бы не растет.
- Появляется понятное правило: какой инструмент использовать, в каких задачах и когда не использовать.

## Риски или сомнения

- Транскрипт источника обрезан до первых 30 минут.
- Claims автора по ценам, процентам экономии, бенчмаркам и версиям инструментов требуют внешней проверки.
- Инструменты и их интеграции быстро устаревают.
- Агрессивное сжатие может повредить аудит, импорт материалов, финансовые расчеты и разбор торговой логики.
- Установка всех инструментов сразу может усложнить workflow больше, чем сэкономить токены.

## Предварительный порядок проверки

1. Сначала проверить текущую экономию через уже принятые Caveman и Context Mode.
2. Отдельно оценить простую дисциплину: короткие инструкции, узкий `rg`, `ctx_execute_file` для извлечения, не читать большие outputs в чат.
3. Проверить Ponytail только на задачах, где Codex реально пишет слишком много кода.
4. Проверить Headroom только на задачах с большими tool outputs.
5. Проверить Memento/Mem0 только для долгих проектов с повторяющимся контекстом и конфликтами решений.
