# RLM для рекурсивного аудита и защиты от context rot

## Оценка ценности

`9/10`

Идея ценная, потому что может усилить умный слой аудита, onboarding по большой базе и разбор сложных участков кода/вики без перегруза одного чата. Оставить в инкубаторе, потому что RLM-подход еще не проверен на TiTan Bots и не должен сразу становиться правилом аудита.

## Lifecycle

Эта карточка живет в `К разбору/Идеи`.

Это не active/current знание и не ответ для RAG. Идея становится active wiki-карточкой только после проверки, решения владельца и отдельного переноса в нужный слой. Если идея устарела или отклонена, перенести в `Архив/К разбору/Отклонено` или другой закрытый архив входов.

`К разбору/Идеи` и `К разбору/Ошибки` являются отдельными ручными входами после audit-stack. Status/queue/warning/guard не является финальным закрытием без terminal owner или lifecycle destination.

## Идея

Проверить RLM-подход для TiTan Bots: главный агент не пытается держать всю базу или весь код в одном контексте, а рекурсивно делит задачу на узкие изолированные разборы.

Возможная схема:

- главный агент находит проблемные зоны базы, аудита или кодовой папки;
- для каждого участка запускается отдельный изолированный sub-RLM / subagent с чистым контекстом;
- sub-RLM видит только конкретный файл, раздел или историю изменений и возвращает короткий вердикт;
- агрегатор собирает результаты в один отчет: что важно, где риск, что проверять первым.

## Почему может быть ценно

- снижает риск `context rot`, когда длинный чат начинает хуже помнить правила, границы и уже найденные выводы;
- может помочь анализировать большие wiki/codebase без бесконечных ручных `grep`;
- подходит для задач, где нужно не просто найти строки, а понять роль файла, связность, активность изменений и риск;
- может стать основой будущего умного слоя аудита: не хранить знания в файлах аудита, а проверять структуру базы через управляемые рекурсивные проходы;
- может ускорить onboarding по проекту: какие папки важны, где активная разработка, какие файлы похожи на источники ошибок.

## К чему относится

- TiTan Bots wiki;
- FAST / DEEP / METHOD / META как возможные потребители отчетов, но не как место хранения самой идеи;
- Codex workflow;
- будущие правила работы с большими контекстами;
- возможная связка с context-mode, subagents, graph/search tooling.

## Что проверить

- Есть ли доступная реализация RLM или проще начать с обычных Codex subagents/context-mode.
- Дает ли рекурсивный разбор лучший отчет, чем текущий `rg` + ручной анализ.
- Можно ли применять подход к wiki без записи проектных знаний в аудит.
- Какие задачи подходят первыми:
  - поиск перегруженных wiki-карточек;
  - анализ больших source/transcript файлов;
  - обзор папок кода перед рефакторингом;
  - поиск проблемных мест по git history;
  - проверка качества связей и ролей страниц.
- Как ограничить стоимость по токенам и не запускать sub-RLM там, где достаточно обычного поиска.

## Как понять, что идея сработала

- отчет по большой зоне базы стал короче и точнее, чем ручной разбор;
- меньше пропусков из-за перегруженного контекста;
- sub-RLM возвращают проверяемые короткие вердикты, а не пересказ всего файла;
- итоговый агрегатор выдает приоритеты и next owner, а не размытый список наблюдений;
- подход не нарушает правило: файлы аудита хранят только логику проверки, а не проектные знания.

## Риски или сомнения

- Видео содержит сильные claims про качество RLM; их нужно проверять, не переносить как факт.
- Рекурсивный подход может резко поднять расход токенов.
- Слишком много subagents может дать шум вместо качества.
- Для простых задач обычный `rg`, FAST-аудит или context-mode может быть достаточно.
- Нельзя автоматически внедрять это в active audit-stack без отдельного решения и теста.
